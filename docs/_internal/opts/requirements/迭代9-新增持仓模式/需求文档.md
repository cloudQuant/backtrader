# 迭代9：新增双向持仓模式 — 需求文档

> 版本：v1.1
> 状态：技术经理审阅版
> 关联文件：`初始需求.md`、`设计文档.md`、`任务.md`、`测试文档.md`

---

## 1. 审阅结论

本次审阅基于当前代码实现，对原始需求稿做了四个关键修正：

1. **不能把 `cerebro.broker.set(position_mode=...)` 作为正式契约**  
   当前代码没有通用的 `broker.set(...)` 接口，只有构造参数、`set_param(...)` 和少量 `set_xxx(...)` 方法。文档必须以**构造时配置**为主，运行期修改只允许作为受限能力讨论。

2. **不能直接把 `getposition(data, side=None)` 当成统一改签方案**  
   `BtApiBroker` 当前签名为 `getposition(data, clone=True)`，如果把第 2 个参数直接改成 `side` 会破坏兼容。新增方向参数必须是**追加的关键字参数**，不能侵占现有位置参数。

3. **不能把 `Trade / notify_trade / Analyzer` 适配写成“顺手完成”**  
   当前 `Strategy._addnotification()` 以 `tradeid` 为核心分组键，双向持仓下如果不把 `position_side` 纳入 trade 分组，长短腿会被错误合并。该问题要么进入实现范围，要么显式降级，不能模糊表述。

4. **不能在文档里重新发明一套与 `CommInfoBase` 无关的价值/PnL公式**  
   现有 broker 的现金、保证金、杠杆、`shortcash`、`fundmode`、`cashadjust` 都建立在 `CommInfoBase` 语义上。双向持仓必须在**复用现有结算语义**的前提下扩展，而不是另起炉灶。

## 2. 背景与问题定义

### 2.1 当前实现边界

当前项目的持仓模型本质是**净头寸模式（Netting）**：

| 维度 | 当前实现 | 影响 |
|------|---------|------|
| 持仓对象 | 单个 `Position(size, price)` | 同一标的只能存在一个净仓 |
| 持仓存储 | `BackBroker.positions` / `BtApiBroker.positions` / `TickBroker._positions` | 结构天然按净仓建模 |
| 下单解释 | `buy` 增加净多，`sell` 增加净空或平多 | 无法表达“开空但不动已有多头” |
| `close()` | 按净仓正负决定 `buy/sell` | 双边都有仓位时语义不充分 |
| `Trade` 分组 | `data + tradeid` | 双向模式下长短腿可能被合并 |

### 2.2 本轮要解决的问题

本轮要解决的是**同一 `data/symbol` 下长短两条仓位腿并存**的问题，而不是：

1. 多策略之间的持仓隔离
2. 各交易所特有的 `close_today` / `close_yesterday`
3. 锁仓减免保证金、组合保证金、交易所风控细节
4. 所有 Observer / Analyzer 的完整双向语义升级

### 2.3 本轮成功标准

本轮成功的最低标准是：

1. 在 broker 级新增 `position_mode`
2. `dual_side` 模式下支持开多、平多、开空、平空四类基础动作
3. 长短腿状态、均价、已实现盈亏彼此独立
4. `net` 模式零退化
5. 四类 broker 的策略接口语义保持一致

## 3. 需求定义

### 3.1 P0 — 必须实现

#### R-P0-01: `position_mode` 配置与冻结规则

**描述**：统一新增 broker 级参数 `position_mode`。

**功能需求**：

1. 四类 broker 统一支持参数 `position_mode`
2. 取值限定为 `"net"` 或 `"dual_side"`
3. 默认值为 `"net"`
4. 正式支持的配置方式为：
   - 构造 broker 时传入：`BackBroker(position_mode="dual_side")`
   - 或通过 store/broker 工厂在实例创建时传入
5. 若保留 `set_param("position_mode", ...)`，只能在 **broker.start() 之前且首笔订单提交之前** 调用
6. broker 一旦 `start()`，或已经提交过订单，`position_mode` 必须冻结

**明确排除**：

1. 不把 `cerebro.broker.set(position_mode=...)` 作为正式 API 契约
2. 不支持运行中动态切换净仓/双向模式

**验收标准**：

- 无参数时默认进入 `net`
- 非法值报 `ValueError`
- 冻结后修改 `position_mode` 报错，错误信息包含“已启动/已有订单”

#### R-P0-02: 双向持仓内部模型

**描述**：在 `dual_side` 模式下，同一标的维护独立的多头腿和空头腿。

**功能需求**：

1. broker 内部维护两套持仓映射：
   - `long_positions[data] -> Position`
   - `short_positions[data] -> Position`
2. 两个 `Position` 实例都使用 `size >= 0` 表示该腿的数量
3. 长短腿分别维护自己的均价、数量、开平更新状态
4. 对于**已经公开暴露** `positions` 的 broker：
   - `positions` 必须继续表示**净头寸兼容视图**
   - 不能把 `positions` 作为双向模式下的唯一真实来源
5. `Strategy.position`、`Strategy.positions`、`Strategy.positionsbyname` 默认仍表示净头寸视图

**验收标准**：

- 可同时出现 `long=100` 且 `short=50`
- 平多不影响空头腿，平空不影响多头腿
- 兼容视图中的净仓数量等于 `long - short`

#### R-P0-03: `buy` / `sell` 接口扩展与订单元信息

**描述**：在现有下单入口上扩展双向语义，不新增主函数族。

**功能需求**：

1. `buy()` / `sell()` 新增语义参数：
   - `position_side`: `"long"` | `"short"` | `None`
   - `offset`: `"open"` | `"close"` | `None`
2. `net` 模式下：
   - 两个参数默认都为 `None`
   - 任一参数显式传值时应报错
3. `dual_side` 模式下：
   - 两个参数都必须显式传入
   - 合法组合仅限：

| 订单 | `position_side` | `offset` | 业务含义 |
|------|-----------------|----------|----------|
| `buy` | `long` | `open` | 开多 |
| `sell` | `long` | `close` | 平多 |
| `sell` | `short` | `open` | 开空 |
| `buy` | `short` | `close` | 平空 |

4. broker 必须统一校验非法组合
5. 订单元信息的 **P0 真实来源** 定义为：
   - `order.info.position_side`
   - `order.info.offset`
6. 若后续给 `Order` 增加直达属性 `order.position_side` / `order.offset`，应作为对 `order.info` 的只读镜像，不改变语义来源
7. 四类 broker 必须统一其订单创建路径，不能依赖“某个 broker 可以把未知 kwargs 直接塞进 `Order(...)`”这样的偶然行为

**验收标准**：

- 合法组合可下单
- 非法组合被拒绝
- `notify_order` 中可稳定读取 `order.info.position_side` 和 `order.info.offset`

#### R-P0-04: `close()` 语义扩展

**描述**：在双向持仓下补齐平仓方向语义。

**功能需求**：

1. `Strategy.close()` 新增可选参数 `position_side=None`
2. `net` 模式下行为完全不变
3. `dual_side` 模式下：
   - `close(data, position_side="long")` 表示平多
   - `close(data, position_side="short")` 表示平空
   - 仅单边有仓时，`close(data)` 允许自动推断
   - 双边都有仓时，`close(data)` 必须报错
4. `size` 仍支持部分平仓
5. 超量平仓必须被明确拒绝，不能静默截断

**验收标准**：

- 仅单边持仓时 `close(data)` 可正常工作
- 双边持仓时 `close(data)` 报明确错误
- 部分平仓只影响目标腿

#### R-P0-05: `getposition()` 与策略查询兼容

**描述**：在不破坏现有位置参数的前提下扩展明细查询。

**功能需求**：

1. `side` 必须作为**新增关键字参数**追加，不能破坏现有参数顺序
2. 建议形式：
   - `BackBroker / TickBroker / MixBroker`: `getposition(data, side=None)`
   - `BtApiBroker`: `getposition(data, clone=True, side=None)`
3. `side` 取值：
   - `None`：返回净头寸视图
   - `"long"`：返回多头腿
   - `"short"`：返回空头腿
4. `Strategy.getposition()` 需要同步扩展，能够透传 `side`
5. `Strategy.position` 属性继续等价于净头寸视图，不暴露双向细节
6. `BtApiBroker` 新增 `side` 后，不得改变原有 `clone=True/False` 行为

**验收标准**：

- `broker.getposition(data)` 在 `net` 模式与现有一致
- `broker.getposition(data, side="long")` / `side="short"` 能读到独立持仓腿
- `broker.getposition(data, clone=False, side="long")` 在 `BtApiBroker` 下仍保持现有 clone 语义

#### R-P0-06: 执行、现金、PnL 与估值语义

**描述**：双向持仓必须复用现有结算模型，而不是另起一套公式。

**功能需求**：

1. `dual_side` 模式下的现金、保证金、已实现盈亏、未实现盈亏计算，必须继续基于：
   - `CommInfoBase.getoperationcost`
   - `CommInfoBase.getvaluesize`
   - `CommInfoBase.getvalue`
   - `CommInfoBase.profitandloss`
   - `CommInfoBase.cashadjust`
2. 长短腿的已实现盈亏分别结算，再汇总到账户层
3. 必须保留以下现有语义：
   - `shortcash`
   - `stocklike`
   - `leverage`
   - `fundmode`
   - futures 类产品的逐日盯市/现金调整
4. 文档中不再写死“统一价值公式”，而是要求**每条持仓腿按现有 comminfo 语义独立求值**

**验收标准**：

- `net` 模式估值零退化
- `dual_side` 下账户现金与总价值与 comminfo 语义一致
- `shortcash=True/False` 的行为差异被保留

#### R-P0-07: Trade/通知最小正确性

**描述**：双向模式下不能把长短腿 trade 混在一起。

**功能需求**：

1. `notify_order` 在 P0 必须正确输出 `position_side` / `offset`
2. 对 `Trade` 的最小要求二选一，必须在设计阶段冻结：
   - 方案 A：双向模式下 trade 分组键改为 `(tradeid, position_side)` 或等价复合键
   - 方案 B：P0 不承诺 `notify_trade` / Analyzer / Observer 在 dual_side 下正确
3. 本审阅建议采用**方案 A 作为实现方向，方案 B 作为降级预案**

**验收标准**：

- 不能出现同一 `tradeid` 下长短腿 trade 被错误合并的情况
- 如果选择降级预案，则文档、测试和错误提示中必须显式声明

#### R-P0-08: 四类 broker 的一致性要求

**描述**：四类 broker 在策略接口层保持统一语义。

**功能需求**：

1. `BackBroker`：改造 `check_submitted()`、`_execute()`、`_get_value()`、`getposition()`
2. `TickBroker`：改造订单创建路径、`_execute()`、`getvalue()`、`state_values()`、`order_history`
3. `MixBroker`：沿用 `TickBroker` 的双向执行语义，并确保 `MidFreqContext` 查询结果不失真
4. `BtApiBroker`：
   - 策略层接口与回测 broker 对齐
   - 引入 provider 能力契约，启动前完成模式支持性校验
5. 四类 broker 必须共享相同的参数命名和错误口径

**验收标准**：

- 相同策略在四类 broker 下的开平语义一致
- 不支持双向模式的 live provider 必须在启动期或装配期明确失败

### 3.2 P1 — 应当实现

#### R-P1-01: `positions` / `positionsbyname` 的双向查询补充

**描述**：补上批量查询能力，减少策略层绕 broker 内部结构。

**功能需求**：

1. 对已经暴露 `positions` 的 broker，继续保持净头寸兼容视图
2. 可新增：
   - `getpositions(side=None)`
   - `getpositionsbyname(side=None)`
3. 未明确指定 `side` 时，仍返回净头寸视图

#### R-P1-02: `notify_trade` 与 Analyzer/Observer 适配

**描述**：在 trade 分组键问题解决后，再适配上层消费链路。

**功能需求**：

1. `notify_trade` 能区分长腿与短腿
2. `TradeAnalyzer`、`Transactions`、`PositionsValue` 等不再错误合并长短腿
3. 如无法在本迭代落地，必须拆分成后续迭代，不得写入 P0 验收

#### R-P1-03: Live provider 能力声明

**描述**：定义 live provider 对双向持仓的能力契约。

**功能需求**：

1. store 或 contract metadata 至少提供一种可机读能力来源，例如：
   - `store.supports_position_mode(mode)`
   - `store.get_broker_capabilities()`
   - `contract_metadata["position_modes"]`
2. 缺失能力声明时，默认仅支持 `net`

### 3.3 P2 — 可选实现

#### R-P2-01: 语法糖方法

1. `close_long(data, size=None)`
2. `close_short(data, size=None)`

#### R-P2-02: 批量撤单

1. `cancel_all(data=None, position_side=None)`
2. 明确只作用于挂单，不作用于持仓

## 4. 约束条件

### 4.1 兼容性约束

| ID | 约束 | 说明 |
|----|------|------|
| C-01 | `net` 模式零退化 | 默认行为不变 |
| C-02 | 不破坏现有位置参数 | 例如 `BtApiBroker.getposition(data, clone=True)` 的 `clone` 位置不能被占用 |
| C-03 | `Strategy.position` 继续表示净头寸 | 不在属性层暴露双向细节 |
| C-04 | 已公开的 `positions` 兼容视图必须保留 | 不能让现有依赖突然失效 |
| C-05 | 不解决策略级持仓隔离 | 双向持仓不等于“每个策略独立仓位簿” |

### 4.2 技术约束

| ID | 约束 | 说明 |
|----|------|------|
| T-01 | 复用 `Position` | 不新增 `PositionBook` 类 |
| T-02 | 运行中不可切换模式 | 模式冻结是硬约束 |
| T-03 | 复用 `CommInfoBase` 结算语义 | 不单独发明价值/PnL公式 |
| T-04 | 统一订单元信息通道 | P0 以 `order.info` 为准 |
| T-05 | 不以 bracket 封装作为 P0 交付前提 | 当前 `buy_bracket/sell_bracket` 会把 kwargs 广播到子单，语义需单独设计 |

### 4.3 质量约束

| ID | 约束 | 说明 |
|----|------|------|
| Q-01 | 性能可控 | `net` 模式性能退化不超过 2% |
| Q-02 | 错误信息可操作 | 必须指出模式、参数组合和修复建议 |
| Q-03 | 核心路径高覆盖 | 双向持仓核心执行路径测试覆盖率 >= 90% |

## 5. 本轮非目标

1. 不做策略级仓位隔离或“多策略各自一套多空腿”
2. 不做 `close_today` / `close_yesterday`
3. 不做锁仓减免保证金、组合保证金
4. 不把 `buy_bracket()` / `sell_bracket()` 的双向语义纳入 P0 验收
5. 不承诺所有 Analyzer / Observer 在本轮内全部双向化
6. 不承诺所有 live provider 原生支持 `dual_side`

## 6. 风险评估

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|---------|
| 只扩参数、不改持仓真源 | 高 | 接口支持双向，内部仍按净仓运行 | `long_positions` / `short_positions` 作为真实来源，`positions` 仅做兼容视图 |
| 直接修改 `getposition` 第 2 参数 | 高 | 破坏 `BtApiBroker` 兼容 | `side` 只能追加为关键字参数 |
| TickBroker 直接把未知 kwargs 传给 `Order(...)` | 高 | 双向参数无法跨 broker 统一 | 先统一订单创建和元信息注入路径 |
| Trade 分组不改 | 高 | `notify_trade` 和 analyzer 结果错误 | 将 `position_side` 纳入 trade 分组键或显式降级 |
| 文档写死估值公式 | 中 | 与 `shortcash` / futures 逻辑冲突 | 强制复用 `CommInfoBase` 语义 |
| bracket/OCO 语义未冻结 | 中 | 子单继承错误的 `offset`/`position_side` | 从 P0 排除，单独设计 |
| live provider 能力无法声明 | 中 | 回测支持、实盘失败 | 增加 broker-store 能力契约 |

## 8. 术语表

| 术语 | 定义 |
|------|------|
| 净头寸模式（Net） | 同一标的只有一个净仓位，长短轧差 |
| 双向持仓模式（Dual-Side） | 同一标的同时维护独立的多头腿和空头腿 |
| 持仓腿（Leg） | 双向模式下的 `long` 或 `short` 独立仓位 |
| 净头寸兼容视图 | 对外保持旧接口兼容的净仓表示 |
| `position_side` | 订单作用的持仓腿，`long` 或 `short` |
| `offset` | 开平动作，`open` 或 `close` |
| provider 能力契约 | live store/provider 声明自身是否支持某种持仓模式的接口或元数据 |
