### 背景

现在的持仓模式只有净头寸模式，需要新增持仓模式。希望可以在 `bbroker.py`、`tickbroker.py`、`mixbroker.py` 和 `btapibroker.py` 中支持新的持仓模式。

在一个 `symbol` 或者 `data` 上可以同时持有多单和空单。

### 原始诉求

1. 希望能够新增一个参数，默认情况下保持现在这种净持仓模式，但也可以切换到“同时持有多单和空单”的模式。
2. 希望调研并给出建议：是否需要给 `buy`、`sell`、`close`、`cancel` 等方法增加参数来支持双向持仓，还是应该增加新的函数来实现。

### 需求梳理与迭代9实施计划

#### 1. 业务定义

1. 当前系统的持仓模型本质上是**净头寸模式**：同一 `data/symbol` 只有一个持仓对象，`size > 0` 表示净多，`size < 0` 表示净空，`size == 0` 表示空仓。
2. 本轮希望新增的模式，本质上是**双向持仓模式**（建议命名为 `hedged` 或 `dual_side`，本文统一使用 `hedged`）：同一 `data/symbol` 下，多头和空头是两条彼此独立的持仓腿，可以同时存在。
3. 双向持仓模式下，需要支持以下典型场景：
   - 同一合约先开多，再独立开空，两边同时持有
   - 只平多头，不影响同一合约下的空头持仓
   - 只平空头，不影响同一合约下的多头持仓
   - 多头和空头各自维护开仓均价、持仓数量、已实现盈亏等状态
4. 本轮的核心不是单纯增加一个配置项，而是要明确以下语义：
   - 订单在两种持仓模式下如何解释
   - 持仓对象如何表达
   - `close`、`getposition`、`getvalue`、PnL、保证金、通知等接口如何定义
   - 四类 broker 在回测/实盘路径中如何保持一致语义

#### 2. 对当前问题的分析结论

1. 当前代码中的持仓核心抽象是单个 `Position(size, price)`，它天然是净头寸模型，不足以直接表达“同一标的同时存在 long leg 和 short leg”。
2. 当前 `Strategy.close()` 的实现也是围绕净头寸设计：
   - 若当前仓位为正，则通过 `sell` 平仓
   - 若当前仓位为负，则通过 `buy` 平仓
   因此在双向持仓模式下，`close(data)` 会出现语义不充分的问题，因为它无法知道“要平多头还是平空头”。
3. 当前各 broker 的 `buy` / `sell` 方法都已经支持 `**kwargs`，因此从接口演进角度看，**优先在现有下单接口上增加扩展参数**，会比新造一批平行函数更自然，也更利于兼容现有策略代码风格。
4. `cancel(order)` 当前是基于订单对象本身取消，订单已具备足够身份信息，因此从业务定义上看，`cancel` 通常**不需要因为新增持仓模式而强制增加新参数**。
5. `btapibroker.py` 的支持不能只看本地接口，还要考虑真实 provider / 交易所是否支持双向持仓。如果底层不支持，应在启动阶段或下单阶段给出明确错误，而不是静默退化。

#### 3. 架构判断与建议

##### 3.1 总体建议

1. 建议新增统一参数：`position_mode`。
2. 建议取值：
   - `net`：默认值，保持当前净头寸行为
   - `dual_side`：允许同一 `data/symbol` 同时持有多头和空头
3. 建议该参数在四类 broker 上都采用相同命名和默认值，避免不同 broker 出现不同模式名或不同默认语义。

##### 3.2 关于 `buy/sell/close/cancel` 的接口建议

**结论：优先扩展现有函数参数，不建议把新增语义全部做成新函数。**

建议如下：

1. `buy` / `sell`
   - 建议增加可选参数：`position_side`、`offset`
   - 推荐语义：
     - `position_side`: `"long" | "short" | None`
     - `offset`: `"open" | "close" | None`
   - 在 `net` 模式下：
     - 默认可忽略这两个参数
     - 若显式传入，建议限制或报错，避免用户误以为已进入双向模式
   - 在 `dual_side` 模式下：
     - `position_side` 用于说明该订单作用于哪一条持仓腿
     - `offset` 用于说明这是开仓单还是平仓单

2. `close`
   - 建议增加可选参数：`position_side`
   - 在 `dual_side` 模式下，如果同一标的同时持有多头和空头，则 `close(data)` 不再天然明确。
   - 建议规则：
     - 若双边都有仓位，调用 `close(data)` 但未传 `position_side`，应报错或拒绝
     - 若只存在单边仓位，则允许推断并执行
     - 若传入 `position_side="long"`，表示平多头
     - 若传入 `position_side="short"`，表示平空头

3. `cancel`
   - 建议保持现有签名不变
   - 原因：撤单针对的是具体订单，不是抽象持仓腿；订单一旦创建，就应已经携带足够的模式/方向信息
   - 如果后续需要批量撤销某一侧订单，可另行增加批量接口，但不建议作为本轮核心变更

4. 是否增加新函数
   - 本轮**不建议**把核心语义建立在 `buy_long`、`sell_short`、`close_long`、`close_short` 这类新主接口上
   - 原因：
     - 会显著增加策略 API 面
     - 会和现有 `buy/sell/close` 形成两套并行风格
     - 现有 broker / strategy / order 流程本来就围绕 `buy/sell/close` 展开
   - 可选建议：后续如果希望提升策略层可读性，可以把 `close_long` / `close_short` 作为 `close(..., position_side=...)` 的**语法糖包装函数**，但不作为底层基础契约

##### 3.3 持仓对象建议

don't use new PositionBook, use long_position and short_position in the broker, it's the position instance. 


##### 3.4 查询接口建议

1. `getposition(data)` 在净头寸模式下保持现状。
2. 在双向持仓模式下，建议明确以下两类能力：
   - 兼容查询：返回净头寸视图
   - 明细查询：返回双向持仓明细
3. 建议候选方案：
   - 方案A：扩展 `getposition(data, side=None)`
4. 从兼容性和可读性出发，建议优先评估：
   - `getposition(data)` 保留净头寸返回
   - 新增 `getposition(data, side="long"|"short")` 或单独的明细查询接口
5. 该问题需要在正式需求文档中明确，否则策略在双向模式下很难安全地读取持仓状态。

##### 3.5 四类 broker 的一致性建议

1. `bbroker.py`
   - 负责低频回测的订单执行与持仓回写
   - 需要支持 `position_mode=dual_side` 下的持仓拆分、平仓腿选择、PnL 计算
2. `tickbroker.py`
   - 负责高频撮合与账户回写
   - 需要保证双向持仓下，撮合结果正确回写到对应持仓腿，而不是仍按净仓逻辑对冲
3. `mixbroker.py`
   - 应沿用其所依赖执行核心的同一持仓模式契约
   - 不应单独发明与 `bbroker` / `tickbroker` 不一致的双向持仓语义
4. `btapibroker.py`
   - 对策略接口层要尽量表现出与回测 broker 一致的模式切换能力
   - 对 provider 能力层要增加校验：底层不支持双向持仓时，应显式报错或拒单

### 技术建议补充（基于当前代码结构）

#### 建议1：先扩展参数，不先扩展主函数族

当前 broker 和 strategy 的主交易入口已经稳定围绕 `buy/sell/close/cancel` 构建，建议优先在这套接口上增加最少必要参数，而不是再开一套新的主入口。

推荐方向：

```python
self.buy(data=data, size=1, position_side="long", offset="open")
self.sell(data=data, size=1, position_side="short", offset="open")
self.close(data=data, position_side="long")
```

这样做的好处是：

1. 与现有策略书写风格一致
2. 便于 broker 通过 `order.info` 或等价字段携带扩展语义
3. 对 `net` 模式保持较好的向后兼容空间

#### 建议2：`cancel` 不作为本轮重点扩展对象

`cancel(order)` 操作的是一个已经创建好的订单。只要订单对象内部已经记录了 `position_mode`、`position_side`、`offset` 等信息，撤单本身通常不需要新增入参。

因此建议结论是：

1. `buy/sell`：建议扩展
2. `close`：建议扩展
3. `cancel`：建议保持不变

#### 建议3：本轮必须把“持仓对象重构”写入正式范围

如果正式需求只写“给 `buy/sell` 增加参数”，而不写“重构持仓内部模型”，后续实现极有可能在表面支持双向模式，但实际成交回写、PnL 和仓位查询都仍然按净头寸运行。

因此在后续 `需求文档.md` 中，建议把以下能力列为显式需求：

1. 同一标的下多头腿和空头腿独立维护
2. 平仓必须显式作用于指定持仓腿或遵循明确推断规则
3. 账户回写、手续费、盈亏统计不能再仅依赖单一净仓位对象

#### 建议4：实盘路径要增加“能力检查”需求

对于 `btapibroker.py`，不能只定义接口长什么样，还要定义：

1. provider 支持双向持仓时如何运行
2. provider 不支持双向持仓时如何拒绝
3. 错误信息如何让用户清楚知道是“本地配置问题”还是“远端接口不支持”
