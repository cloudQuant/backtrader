# 迭代9：新增双向持仓模式 — 任务分解与实施计划

> 版本：v1.1
> 状态：技术经理审阅版
> 关联文件：`需求文档.md`、`设计文档.md`、`测试文档.md`

---

## 1. 实施总策略

本迭代不是“给 `buy/sell` 多加两个参数”这么简单，而是一次**持仓真源、执行路径、兼容视图、通知链路**的联合改造。排期必须围绕下面四个高风险点组织：

1. `position_mode` 必须是可冻结的 broker 级配置
2. 订单扩展参数必须跨四类 broker 统一落到 `order.info`
3. 双向模式下真实持仓必须从净仓切换为“长腿 + 短腿”
4. `Trade` 分组键如果不改，长短腿通知会被错误合并

## 2. 范围与优先级

### 2.1 P0 范围

| 模块 | 文件 | 目标 |
|------|------|------|
| Strategy | `backtrader/strategy.py` | `close()` / `getposition()` / trade key |
| Order | `backtrader/order.py` | 可选只读别名或辅助属性 |
| BackBroker | `backtrader/brokers/bbroker.py` | 双向核心实现 |
| TickBroker | `backtrader/brokers/tickbroker.py` | 订单元信息统一、双向执行 |
| MixBroker | `backtrader/brokers/mixbroker.py` | 继承适配与查询一致性 |
| BtApiBroker | `backtrader/brokers/btapibroker.py` | side 关键字扩展、能力校验、同步 |

### 2.2 P0 明确排除

1. `buy_bracket()` / `sell_bracket()` 的 dual-side 正式支持
2. `close_today` / `close_yesterday`
3. 组合保证金、锁仓减免
4. 所有 Analyzer / Observer 的完整双向化

### 2.3 优先级划分

| 优先级 | 内容 |
|--------|------|
| P0 | 配置冻结、订单元信息、双向持仓真源、`close/getposition`、BackBroker/TickBroker/MixBroker/BtApiBroker 基础适配 |
| P1 | `positions` 批量查询增强、Analyzer/Observer 进一步适配、provider 能力声明标准化 |
| P2 | 语法糖方法、批量撤单 |

## 3. 分阶段计划

## Phase 0：需求与实现边界冻结

### 目标

把所有“会破坏兼容”或“会误导排期”的问题先冻结。

### 任务

1. 冻结 `position_mode` 的正式配置方式
   - [ ] 明确只承诺构造时配置
   - [ ] 冻结后修改的报错口径

2. 冻结 `getposition` 扩展方案
   - [ ] 确认 `side` 只能追加为关键字参数
   - [ ] 确认 `BtApiBroker.getposition(data, clone=True, side=None)` 的兼容方案
   - [ ] 确认 `Strategy.getposition(..., side=None)` 是否进入本迭代

3. 冻结订单元信息落点
   - [ ] P0 统一采用 `order.info.position_side` / `order.info.offset`
   - [ ] 评估是否补充 `Order` 只读别名

4. 冻结 `Trade` 路径决策
   - [ ] 选择方案 A：trade key 引入 `position_side`
   - [ ] 或方案 B：dual_side 下不承诺 `notify_trade`

5. 冻结非目标
   - [ ] 明确 `buy_bracket` / `sell_bracket` 不纳入 P0
   - [ ] 明确 Analyzer/Observer 完整适配不纳入 P0

### 退出标准

1. `需求文档.md` 和 `设计文档.md` 标记为技术经理审阅版
2. 研发排期基于冻结后的边界重估

## 5. 风险与依赖

| 风险 | 等级 | 应对 |
|------|------|------|
| Trade key 改造晚于 broker 落地 | 高 | 提前在 Phase 1/2 冻结决策 |
| TickBroker 继续把未知 kwargs 传给 `Order(...)` | 高 | Phase 1 就修正下单路径 |
| `positions` 兼容视图设计不清 | 高 | Phase 2 必须明确“真源 vs 兼容视图” |
| BtApi provider 无法给出双向能力声明 | 中 | 提供 metadata 兜底协议，缺省仅支持 `net` |
| bracket/OCO 语义扩散到 P0 | 中 | 文档和测试中明确排除 |
