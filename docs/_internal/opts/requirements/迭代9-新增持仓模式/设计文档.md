# 迭代9：新增双向持仓模式 — 设计文档

> 版本：v1.1
> 状态：技术经理审阅版
> 关联文件：`需求文档.md`、`任务.md`、`测试文档.md`

---

## 1. 关键设计修订

相对原始设计稿，本版设计做了以下关键修订：

1. **参数声明方式跟随现有 broker 风格，而不是强行统一成 `params` 元组**  
   `BackBroker` / `TickBroker` / `MixBroker` 目前主要使用 `ParameterDescriptor`，`BtApiBroker` 仍保留 legacy `params`。本轮统一的是**外部参数名和语义**，不是类内声明形式。

2. **`side` 必须是追加的关键字参数**  
   不能破坏 `BtApiBroker.getposition(data, clone=True)` 的现有位置参数。

3. **P0 以 `order.info` 作为双向元信息真实来源**  
   这是跨四类 broker 最稳妥的落点。若后续增加 `Order` 直达属性，只能做只读别名。

4. **内部持仓与外部兼容视图分离**  
   双向模式下，`long_positions` / `short_positions` 是真实来源；对外暴露的 `positions` 只是净头寸兼容视图。

5. **Trade 分组必须考虑 `position_side`**  
   否则长短腿会在 `Strategy._addnotification()` 中被错误合并。

6. **P0 不把 bracket 包装函数作为强制交付项**  
   当前 `buy_bracket()` / `sell_bracket()` 会把 `**kwargs` 广播给主单与子单，和 `offset` 语义天然冲突，必须单独设计。

---

## 2. 设计原则

1. **净仓零退化**：`position_mode="net"` 下尽量不进入双向分支
2. **兼容优先**：保住 `Strategy.position`、`broker.positions`、`BtApiBroker.getposition(..., clone=True)` 等已有入口
3. **显式优于猜测**：双向模式下只在单边持仓场景允许 `close()` 自动推断
4. **结算复用**：现金、价值、PnL 必须建立在现有 `CommInfoBase` 能力之上
5. **真实来源单一**：双向模式下所有执行、预检查、估值都以长短腿存储为准，净头寸视图不能反向驱动真实逻辑

---

## 3. 架构总览

```text
Strategy Layer
  buy/sell(position_side, offset)
  close(position_side)
  getposition(side=...)
          |
          v
Broker Layer
  position_mode = net | dual_side
  - net:     positions[data] -> Position
  - dual:    long_positions[data]  -> Position
             short_positions[data] -> Position
             positions[data]       -> net compatibility view
          |
          v
Execution / Accounting Layer
  CommInfoBase.getoperationcost / getvalue / getvaluesize
  CommInfoBase.profitandloss / cashadjust
          |
          v
Notification Layer
  order.info.position_side / order.info.offset
  trade key = (tradeid, position_side) or documented downgrade
```

## 4. 详细设计

### 4.1 `position_mode` 参数与冻结策略

#### 4.1.1 参数声明

按各 broker 现状声明参数：

```python
# BackBroker / TickBroker / MixBroker
position_mode = ParameterDescriptor(default="net", doc="net | dual_side")

# BtApiBroker（保持现有 legacy 风格）
params = (
    ...
    ("position_mode", "net"),
)
```

#### 4.1.2 冻结规则

建议新增内部状态：

```python
self._position_mode_frozen = False
self._position_mode_frozen_reason = None
```

冻结时机：

1. `start()` 调用完成后
2. 首笔订单进入 `submit()` 或 `transmit()` 之后

辅助方法建议：

```python
def _freeze_position_mode(self, reason):
    self._position_mode_frozen = True
    self._position_mode_frozen_reason = reason

def _ensure_position_mode_mutable(self):
    if self._position_mode_frozen:
        raise ValueError(
            f"position_mode is frozen after {self._position_mode_frozen_reason} "
            "and cannot be changed at runtime"
        )
```

说明：

1. 文档不再定义通用 `broker.set(...)`
2. 若暴露 `set_param("position_mode", ...)`，内部必须先调用 `_ensure_position_mode_mutable()`

### 4.2 订单元信息设计

#### 4.2.1 P0 元信息真实来源

P0 统一采用：

```python
order.info.position_side  # "long" | "short" | None
order.info.offset         # "open" | "close" | None
```

原因：

1. `BackBroker` / `BtApiBroker` 已经通过 `order.addinfo(**kwargs)` 承载扩展参数
2. `TickBroker` 当前直接把 `**kwargs` 传给 `Order(...)`，而 `OrderParams` 只接受固定字段，未知字段会报错
3. 用 `order.info` 可先完成四类 broker 统一，再决定是否向 `Order` 主体下沉属性

#### 4.2.2 broker 统一的订单创建辅助函数

建议新增统一辅助逻辑：

```python
def _attach_position_meta(self, order, position_side=None, offset=None, **kwargs):
    if position_side is not None:
        order.addinfo(position_side=position_side)
    if offset is not None:
        order.addinfo(offset=offset)
    if kwargs:
        order.addinfo(**kwargs)
    return order
```

设计要求：

1. `BackBroker` / `BtApiBroker`：保留现有 `BuyOrder(...)` + `order.addinfo(...)` 风格
2. `TickBroker` / `MixBroker`：必须先改成“先创建 order，再写 info”，不能再把未知 kwargs 直接传给 `Order(...)`

#### 4.2.3 可选的只读别名

若后续需要兼容 `order.position_side` 写法，推荐通过 `OrderBase` 只读属性映射：

```python
@property
def position_side(self):
    return getattr(self.info, "position_side", None)

@property
def offset(self):
    return getattr(self.info, "offset", None)
```

### 4.3 持仓存储与兼容视图

#### 4.3.1 推荐数据结构

```python
# net 模式
self.positions = defaultdict(Position)

# dual_side 模式
self.long_positions = defaultdict(Position)
self.short_positions = defaultdict(Position)
self.positions = defaultdict(Position)   # 净头寸兼容视图
```

说明：

1. `long_positions` / `short_positions` 是真实来源
2. `positions` 只是兼容层，供旧接口和属性读取
3. 不引入新的 `PositionBook` 类

#### 4.3.2 净头寸兼容视图同步

建议使用单点同步函数：

```python
def _sync_net_position(self, data):
    long_pos = self.long_positions[data]
    short_pos = self.short_positions[data]

    net_pos = self.positions[data]
    net_size = long_pos.size - short_pos.size

    if net_size > 0:
        net_price = long_pos.price
    elif net_size < 0:
        net_price = short_pos.price
    else:
        net_price = 0.0

    net_pos.fix(net_size, net_price)
    net_pos.datetime = max(
        long_pos.datetime or short_pos.datetime,
        short_pos.datetime or long_pos.datetime,
    )
    return net_pos
```

注意：

1. `positions[data].price` 在 dual-side 下只是**兼容字段**
2. 当长短腿同时存在时，单个 `Position(price)` 无法完整表达两侧均价，因此兼容视图的 `price` 不能作为双向结算依据

### 4.4 `getposition()` 设计

#### 4.4.1 broker 签名

```python
# BackBroker / TickBroker / MixBroker
def getposition(self, data, side=None):
    ...

# BtApiBroker
def getposition(self, data, clone=True, side=None):
    ...
```

约束：

1. `side` 必须通过关键字传递
2. `BtApiBroker` 继续保留 `clone=True/False`

#### 4.4.2 Strategy 层透传

建议把 `Strategy.getposition` 改成：

```python
def getposition(self, data=None, broker=None, side=None, **kwargs):
    data = data if data is not None else self.datas[0]
    broker = broker or self.broker
    return broker.getposition(data, side=side, **kwargs)
```

兼容性说明：

1. `Strategy.position` 仍调用 `getposition(side=None)`
2. `Strategy.position` 只返回净头寸兼容视图

### 4.5 双重符号体系

双向模式下需要同时保留两套符号语义：

| 语义层 | 规则 |
|--------|------|
| 订单/Trade 层 | 继续沿用 backtrader 传统：买为正、卖为负 |
| 持仓腿内部层 | `long_positions` / `short_positions` 都使用正数表示该腿数量；开仓 `+qty`，平仓 `-qty` |

映射关系如下：

| 动作 | 订单方向 | 目标腿 | 持仓腿更新量 | 对外执行位符号 |
|------|----------|--------|-------------|---------------|
| 开多 | buy | long | `+Q` | `opened=+Q` |
| 平多 | sell | long | `-Q` | `closed=-Q` |
| 开空 | sell | short | `+Q` | `opened=-Q` |
| 平空 | buy | short | `-Q` | `closed=+Q` |

这意味着：

1. `Position.update()` 在目标腿内部始终接收“腿内增减量”
2. `order.execute(...)` / `Trade.update(...)` 仍看到传统 backtrader 符号
3. broker 必须在“腿内数量”和“订单执行位数量”之间做一次显式转换

### 4.6 参数校验与动作路由

建议统一校验矩阵：

```python
VALID_COMBOS = {
    ("buy", "long", "open"),
    ("sell", "long", "close"),
    ("sell", "short", "open"),
    ("buy", "short", "close"),
}
```

建议抽出统一路由：

```python
def _resolve_dual_side_action(self, order):
    position_side = getattr(order.info, "position_side", None)
    offset = getattr(order.info, "offset", None)

    if position_side == "long":
        leg_positions = self.long_positions
    elif position_side == "short":
        leg_positions = self.short_positions
    else:
        raise ValueError("Missing position_side for dual_side order")

    if offset == "open":
        leg_delta = abs(self._get_remaining_size(order))
    elif offset == "close":
        leg_delta = -abs(self._get_remaining_size(order))
    else:
        raise ValueError("Missing offset for dual_side order")

    return position_side, offset, leg_positions, leg_delta
```

### 4.7 `close()` 设计

建议逻辑：

```python
def close(self, data=None, size=None, position_side=None, **kwargs):
    if self.broker.get_param("position_mode") == "net":
        ...

    long_pos = self.broker.getposition(data, side="long")
    short_pos = self.broker.getposition(data, side="short")

    if position_side is None:
        if long_pos and short_pos:
            raise ValueError("Both long and short positions exist; specify position_side")
        if long_pos:
            position_side = "long"
        elif short_pos:
            position_side = "short"
        else:
            return None

    if position_side == "long":
        close_size = abs(size) if size is not None else long_pos.size
        return self.sell(data=data, size=close_size, position_side="long", offset="close", **kwargs)

    if position_side == "short":
        close_size = abs(size) if size is not None else short_pos.size
        return self.buy(data=data, size=close_size, position_side="short", offset="close", **kwargs)
```

### 4.8 BackBroker 改造点

`BackBroker` 是本轮主实现载体。

#### 4.8.1 需要改造的方法

1. `__init__` / `init`
2. `buy()` / `sell()`
3. `check_submitted()`
4. `_execute()`
5. `_get_value()`
6. `getposition()`

#### 4.8.2 `check_submitted()` 预执行逻辑

当前 `check_submitted()` 依赖：

```python
position = positions.setdefault(order.data, self.positions[order.data].clone())
cash = self._execute(order, cash=cash, position=position)
```

双向模式下不能再只克隆净头寸，否则会把“平多”和“开空”都错误投射到同一个净仓上。

建议做法：

1. 预执行阶段同时维护两套临时腿仓位
2. `_execute(..., cash=..., position=...)` 在 dual-side 下改为接收“目标腿的 position clone”
3. 预执行结束后，仅使用 cash 结果做保证金判定，不更新真实持仓

#### 4.8.3 `_execute()` 的 dual-side 分支

建议拆为公共小函数，减少 bar/tick/live 三套实现重复：

```python
def _execute_dual_side_leg(self, order, leg_position, exec_price, exec_qty, dt):
    ...
```

核心要求：

1. 先在目标腿上做 `pseudoupdate()` 或 `update()`
2. 再按腿方向换算已实现盈亏符号
3. 继续使用 `CommInfoBase` 计算现金、保证金、盯市调整
4. 成功执行后同步 `positions[data]` 兼容视图

### 4.9 TickBroker / MixBroker 改造点

#### 4.9.1 TickBroker

重点受影响方法：

1. `buy()` / `sell()`：订单元信息注入方式需要统一
2. `_execute()`：从单净仓更新改成目标腿更新
3. `getvalue()`：需要按两条腿分别求值再汇总
4. `state_values()` / `order_history`：建议追加 `position_side` / `offset`

#### 4.9.2 MixBroker

`MixBroker` 继承 `TickBroker`，但还有 `MidFreqContext`。

必须同步考虑：

1. `MidFreqContext.get_position()` 当前读取 `_positions`
2. dual-side 下若 `_positions` 仍表示净仓，则上下文查询默认返回净头寸兼容视图
3. 若后续需要侧向明细查询，可增补：
   - `get_position(symbol, side=None)`
   - `snapshot()` 中增加 `long_position` / `short_position`

### 4.10 BtApiBroker 与 live 能力契约

#### 4.10.1 兼容签名

必须保留：

```python
def getposition(self, data, clone=True, side=None):
    ...
```

#### 4.10.2 provider 能力判断

当前代码没有统一的 `supports_dual_side()` 接口，因此不应在文档里硬绑定这个方法名。

建议能力判断顺序：

```python
def _supports_position_mode(self, mode, data_name=None):
    if hasattr(self.store, "supports_position_mode"):
        return bool(self.store.supports_position_mode(mode))

    if hasattr(self.store, "get_broker_capabilities"):
        caps = self.store.get_broker_capabilities() or {}
        modes = caps.get("position_modes")
        if modes is not None:
            return mode in modes

    rules = self._contract_rules_for(data_name or "__account__")
    modes = rules.get("position_modes")
    if modes is not None:
        return mode in modes

    return mode == "net"
```

#### 4.10.3 live 持仓同步

dual-side live 同步必须满足其一：

1. provider 返回分方向持仓，例如 `direction=long/short`
2. provider 明确返回两条独立腿

若 provider 只返回净仓，且用户要求 `dual_side`，则应在启动前拒绝，而不是装作支持。

### 4.11 Trade 与通知设计

#### 4.11.1 最小实现方向

建议在 `Strategy._addnotification()` 中引入有效 trade key：

```python
def _trade_key(order):
    if getattr(order.info, "position_side", None) is None:
        return order.tradeid
    return (order.tradeid, getattr(order.info, "position_side"))
```

然后把：

```python
datatrades = self._trades[tradedata][order.tradeid]
```

改成：

```python
tradekey = _trade_key(order)
datatrades = self._trades[tradedata][tradekey]
```

这样可以避免同一 `tradeid` 下长短腿合并。

#### 4.11.2 降级预案

若本迭代无法完成 trade key 改造，则必须在 dual-side 模式下显式声明：

1. `notify_trade` 不作为正确性验收项
2. 依赖 `TradeAnalyzer`、`Transactions`、`PositionsValue` 的结果不保证正确

### 4.12 P0 非目标：bracket / OCO 包装函数

当前 `Strategy.buy_bracket()` / `sell_bracket()` 会把 `**kwargs` 广播到主单、止损单、止盈单：

```python
kargs.update(kwargs)
```

在 dual-side 下这会带来两个问题：

1. 主单与子单不应共享同一个 `offset`
2. 某些子单的 `position_side` 可以继承，但 `offset` 必须重写为 `close`

因此本轮设计结论是：

1. **显式 parent/child 订单** 可以在后续单独设计支持
2. `buy_bracket()` / `sell_bracket()` 不纳入 P0 正式支持范围

## 6. 实施建议

1. 先在 `BackBroker` 做完整闭环，验证“订单参数 -> 持仓腿 -> 现金/PnL -> 净视图 -> 通知”链路
2. 然后把公共逻辑抽成 broker 级 helper，再落到 `TickBroker` / `MixBroker`
3. 最后处理 `BtApiBroker` 的 provider 能力契约和 live 同步
4. trade key 改造如果进入 P0，必须与 broker 落地并行推进，不能拖到最后
