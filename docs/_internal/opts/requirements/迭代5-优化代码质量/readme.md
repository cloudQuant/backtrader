# 迭代 5 - 优化代码质量需求文档

## 0. 使用入口

本目录下的文档分工如下：

1. `readme.md`
   - 用于说明迭代 5 的背景、目标、范围和验收要求。
2. `代码质量改进优化.md`
   - 用于作为后续反复复用的主 Prompt 说明书。
   - 后续如果要发起具体的代码质量治理任务，**优先引用该文件**。

本目录文档默认共享以下执行硬约束：

1. 每次代码修改完成、准备提交前，必须运行 `pytest tests -n 8`。
2. 只有当 `pytest tests -n 8` 全部通过后，才可以视为本轮改动满足提交前要求。

## 1. 背景

当前 `backtrader` 仓库已经完成了前几轮偏功能稳定性、仓库治理和运行链路治理相关的工作，但代码质量优化仍然需要一个可持续、可重复执行的方法，而不是一次性的零散修修补补。

本轮的核心诉求不是直接指定某一个文件做重构，而是先沉淀出一套适合后续**反复调用**的高质量提示词与执行约束，让后续每一轮“代码质量改进”都能遵循统一标准推进。

用户希望采用 **BMAD-method** 风格来执行这类任务，因此需要先产出一份明确的需求文档，用于指导后续反复使用的 Prompt 设计、输入方式、输出格式、执行边界和验收标准。

## 2. 本轮目标

本轮目标如下：

1. 形成一份可长期复用的 **代码质量优化 Prompt 模板**。
2. 让该 Prompt 能适配不同规模的代码质量任务，包括单文件、小批次、多文件治理。
3. 约束执行过程必须遵循“先分析、再计划、后修改、再验证”的 BMAD 风格闭环。
4. 明确代码质量优化的关注点，避免优化过程偏离主题。
5. 让每次调用 Prompt 时都能输出结构化结果，便于追踪、复盘和重复迭代。

## 3. 问题定义

如果没有统一的 Prompt 和执行规范，后续代码质量优化任务容易出现以下问题：

1. 优化目标不稳定：一次关注命名，一次关注性能，一次又改动架构，缺少主线。
2. 改动粒度失控：容易出现“大面积改写”但收益不明确。
3. 验证不充分：只做表面重构，没有验证行为是否保持一致。
4. 输出不可复用：每次都重新描述要求，成本高，且质量波动大。
5. 缺少优先级：P0/P1/P2 问题混在一起，导致低价值改动干扰高价值问题处理。

因此，本轮的真正产物不是某一次具体代码提交，而是一套**高复用、强约束、面向执行**的代码质量优化提示体系。

## 4. 适用范围

### 4.1 适用场景

该 Prompt 主要用于以下场景：

1. 清理明显代码气味：死代码、重复逻辑、命名混乱、过长函数、过深嵌套。
2. 提升可维护性：职责拆分、边界清晰化、失败路径显式化、减少隐式行为。
3. 增强回归安全性：补测试、补断言、增加异常上下文、减少静默失败。
4. 收敛技术债：修正临时调试逻辑、删除不可达代码、统一实现风格。
5. 批量治理某一主题问题：例如日志、错误处理、参数校验、抽象契约一致性。

### 4.2 不适用场景

以下情况不应直接使用本 Prompt 作为主执行模板：

1. 需求本质是新增业务功能，而不是质量优化。
2. 任务需要大规模架构重写，且尚未先形成单独设计文档。
3. 任务依赖外部接口、协议、数据库结构的大改动。
4. 任务核心是性能压测、容量规划或基础设施部署，而不是代码质量。

## 5. 本轮交付物

本轮应产出以下内容：

1. 一份正式的 `迭代5` 需求文档。
2. 一份可直接复用的 **标准版 Prompt 模板**。
3. 一份适合快速调用的 **精简版 Prompt 模板**。
4. 一套固定的执行输出格式。
5. 一套验收标准，确保后续每轮优化任务都能判断是否达标。

## 6. Prompt 设计原则

后续复用 Prompt 时，必须遵守以下原则：

1. **先审查，再改动**
   - 先定位问题、划分优先级、确认影响范围。
   - 不能在未理解上下文前直接重写代码。

2. **小步快跑，可验证，可回退**
   - 每次只处理一个明确的问题域。
   - 每一轮改动都应能独立验证。

3. **优先解决高价值问题**
   - 先处理正确性、稳定性、异常可见性、契约一致性。
   - 再处理命名、结构、可读性等中低风险问题。

4. **禁止无关扩张**
   - 不借“优化代码质量”之名引入新功能或大面积重写。
   - 不为追求“更优雅”而破坏现有稳定行为。

5. **证据驱动**
   - 每次都要说明为什么改、改了什么、如何验证。
   - 不能只给结论，不给依据。

6. **测试优先于主观判断**
   - 如果已有测试，必须优先利用。
   - 如果缺少关键覆盖，应补最小必要测试。

## 7. 代码质量优化关注点

后续使用 Prompt 时，优先从以下维度审查：

1. 正确性
   - 是否存在明显 bug、边界条件缺失、契约违背、状态不一致。

2. 异常可见性
   - 是否存在静默失败、宽泛捕获异常、错误上下文不足。

3. 可维护性
   - 是否存在过长函数、职责混杂、重复逻辑、命名模糊、魔法值。

4. 可测试性
   - 是否难以隔离测试，是否缺少必要的回归保护。

5. 一致性
   - 是否与项目当前风格、抽象层级、接口约定不一致。

6. 仓库卫生
   - 是否引入调试残留、临时脚本、不可达代码、无用依赖。

## 8. 标准版 Prompt 模板

以下模板应作为后续反复调用时的主模板：

```text
你现在以 BMAD-method 的方式执行一次“代码质量优化”任务。

目标仓库：backtrader
目标范围：{填写文件、目录、模块或问题域}
本轮主题：{例如：异常可见性、死代码清理、参数校验、主循环可维护性、Store 契约一致性}

请严格按以下流程执行：

第一步：先审查现状
1. 阅读目标范围内的代码与相关测试。
2. 识别代码质量问题，并按 P0 / P1 / P2 分类：
   - P0：正确性、稳定性、契约错误、回归风险高
   - P1：可维护性差、重复逻辑、异常处理不清晰
   - P2：命名、格式、轻量重构、可读性优化
3. 明确哪些问题在本轮应处理，哪些问题应暂缓。

第二步：给出改造计划
1. 只提出小批次、可验证、可回退的改造方案。
2. 说明每一项改动的目标、影响文件、潜在风险、验证方式。
3. 如果任务过大，先缩小范围，不要一次性大改。

第三步：实施改动
1. 优先处理 P0，再处理 P1，最后才考虑 P2。
2. 保持对外行为稳定，不新增无关功能。
3. 删除死代码、调试残留和明显重复逻辑。
4. 对关键失败路径补充明确异常或日志上下文。
5. 如有必要，补最小必要测试来锁定行为。

第四步：验证结果
1. 先运行与改动直接相关的最小测试集。
2. 再说明还需要哪些进一步验证。
3. 如果发现失败，优先修复源码，不要跳过测试制造假通过。

第五步：按固定格式输出结果
请按以下结构输出：

## 本轮范围
## 发现的问题（按 P0/P1/P2）
## 实施计划
## 实际改动
## 风险与未处理项
## 验证结果
## 下一轮建议

额外约束：
1. 不要做无关重构。
2. 不要省略验证说明。
3. 不要宣称“已优化完成”却没有证据。
4. 如果某项问题不适合本轮处理，请明确写出原因。
```

## 9. 精简版 Prompt 模板

当任务范围较小时，可使用以下精简版：

```text
请用 BMAD-method 的方式，对 {目标文件或模块} 做一轮代码质量优化。

要求：
1. 先审查，再给出小步计划，再实施。
2. 只处理高价值问题，不做无关重构。
3. 优先修复正确性、异常处理、契约一致性和明显代码气味。
4. 如有必要补最小测试，并说明验证结果。
5. 最终按“问题列表 / 改动内容 / 风险项 / 验证结果 / 下一步建议”输出。
```

## 10. 推荐输入参数

为了让 Prompt 更稳定，后续每次调用时建议补充以下输入：

1. 目标范围
   - 文件路径、模块名、目录名或问题域。

2. 本轮主题
   - 例如：
   - `异常可见性治理`
   - `主循环整理`
   - `Store 契约统一`
   - `死代码和调试残留清理`
   - `参数校验补强`

3. 边界限制
   - 是否允许新增测试。
   - 是否允许跨文件改动。
   - 是否允许做轻量重构。

4. 验收口径
   - 是否要求相关测试通过。
   - 是否要求保留现有对外 API。
   - 是否要求输出下一轮建议。

## 11. 标准输出要求

每次调用该 Prompt，输出至少应包含以下内容：

1. 本轮审查范围。
2. 问题清单与优先级。
3. 本轮确定处理的问题。
4. 实际修改内容。
5. 未处理项及原因。
6. 风险说明。
7. 验证结果。
8. 下一轮建议。

如果是纯分析轮次，也必须明确说明：

1. 为什么暂不改代码。
2. 当前阻塞点是什么。
3. 下一步最小可执行动作是什么。

## 12. 验收标准

本轮文档与后续 Prompt 复用效果，应满足以下验收要求：

1. Prompt 可重复使用，不依赖一次性上下文描述。
2. Prompt 能明确约束“先审查、再计划、后改动、再验证”的节奏。
3. Prompt 能引导输出结构化结果，而不是散乱回答。
4. Prompt 能限制无关改动，避免任务范围失控。
5. Prompt 能支持后续多轮代码质量优化任务持续复用。

## 13. 后续使用建议

建议后续把“代码质量优化”任务拆成多轮执行，而不是一次性要求全面重构。推荐节奏如下：

1. 先按主题分轮次：
   - 异常处理
   - 死代码清理
   - 契约一致性
   - 结构化重构
   - 测试补强

2. 每轮只处理一个主要问题域。

3. 每轮结束后记录：
   - 已处理项
   - 遗留项
   - 下轮建议

4. 如果某一轮已经开始涉及较大设计变更，应先单独补设计文档，再进入实现。

## 14. 建议的直接调用方式

后续你可以直接基于本文件中的模板，用类似下面的方式发起任务：

```text
请基于 backtrader/docs/_internal/opts/requirements/迭代5-优化代码质量/readme.md 中定义的方法，
对 backtrader/backtrader/cerebro.py 做一轮以“异常可见性治理”为主题的代码质量优化。

要求：
1. 先审查并按 P0/P1/P2 列问题。
2. 只做小批次可验证改动。
3. 如有必要补最小测试。
4. 最后输出问题、改动、风险和验证结果。
```

## 15. 结论

迭代 5 的核心不是立即完成某一次具体代码重构，而是建立一套可持续复用的“代码质量优化作业方法”。

本文件定义的内容，应该被视为后续所有代码质量治理任务的统一入口：

1. 先用统一 Prompt 审查问题。
2. 再按优先级做小批次治理。
3. 每轮都给出验证证据。
4. 持续滚动推进，而不是一次性粗暴重构。
