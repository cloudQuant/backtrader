# 代码质量改进优化

## 1. 文档定位

本文件用于作为 `backtrader` 仓库后续代码质量治理任务的统一指导文档。

与 `readme.md` 的关系如下：

1. `readme.md` 用于说明迭代 5 的背景、目标和整体要求。
2. `代码质量改进优化.md` 用于提供可以直接复用的 Prompt、执行步骤、输出格式和使用方法。

后续如果需要发起代码质量改进任务，优先引用本文件。

## 2. 使用目标

本文件希望解决的问题不是“某一次重构怎么做”，而是“后续如何持续、稳定、可验证地推进代码质量改进”。

核心目标如下：

1. 让代码质量优化任务可以反复复用同一套 Prompt。
2. 让每次改进都遵循统一方法，而不是临时发挥。
3. 让每轮优化都聚焦高价值问题，避免无关重构。
4. 让每轮输出都结构化，便于复盘、追踪和继续迭代。
5. 让代码质量治理形成长期机制，而不是一次性动作。

## 3. 适用范围

本文件适用于以下任务：

1. 单文件代码质量优化。
2. 某一模块的小批次质量治理。
3. 某一类问题的专题治理。
4. 为已有代码补充回归保护和异常可见性。
5. 在不扩展业务功能的前提下提升可维护性。

本文件不适用于以下任务：

1. 明确是新功能开发，而非质量改进。
2. 明确需要大规模架构重写，但还没有设计文档。
3. 明确依赖数据库、协议、外部系统大改的任务。
4. 纯部署、纯压测、纯运维操作类任务。

## 4. 代码质量优化总原则

后续所有代码质量改进任务，都应遵守以下原则：

1. **先审查，再改动**
   - 先搞清楚问题是什么、影响范围在哪里、是否已有测试保护。
   - 未理解上下文前，不直接重写代码。

2. **小步快跑，可验证，可回退**
   - 每次只处理一个明确的问题域。
   - 每轮改动都要能单独说明价值和验证方式。

3. **优先处理高价值问题**
   - 优先级顺序：正确性 > 稳定性 > 异常可见性 > 契约一致性 > 可维护性 > 可读性。

4. **禁止无关扩张**
   - 不要借“代码优化”之名扩展功能。
   - 不要把小范围治理任务做成大面积架构改写。

5. **测试优先，证据优先**
   - 优化后必须给出验证依据。
   - 如果没有测试，也要明确指出缺口和建议补法。

6. **保持对外行为稳定**
   - 若非明确要求，不随意改变对外 API、调用方式和语义。

7. **提交前测试硬门禁**
   - 每次代码修改完成、准备提交前，必须运行 `pytest tests -n 8`。
   - 只有当该测试命令通过后，本轮代码改进才可以视为完成，并可以提交与本轮任务对应的代码改动。
   - 如果测试失败，必须优先修复源码或回退不合适的改动，不能带着失败结果提交。

## 5. 代码质量问题分级标准

后续审查问题时，统一按以下方式分类：

### P0

必须优先处理的问题：

1. 明显 bug。
2. 契约违背或状态不一致。
3. 静默失败、异常被吞掉。
4. 高概率回归风险。
5. 会导致错误结果或不可预测行为的问题。

### P1

重要但可以在小范围内分批处理的问题：

1. 复杂度过高。
2. 重复逻辑明显。
3. 模块职责混乱。
4. 错误处理不清晰。
5. 难以测试或维护成本明显偏高。

### P2

优化价值存在，但优先级较低的问题：

1. 命名不清晰。
2. 局部可读性差。
3. 轻量格式问题。
4. 不影响行为的小型结构整理。

## 6. 统一执行流程

每次使用本文件发起任务时，都应遵循以下流程：

### Step 1：审查现状

1. 阅读目标代码。
2. 阅读相关测试和调用方。
3. 列出问题清单，并按 P0/P1/P2 分类。
4. 判断本轮应该处理哪些问题，不处理哪些问题。

### Step 2：制定计划

1. 只给出小批次、可验证、可回退的计划。
2. 明确每项改动的目标、影响文件、风险和验证方式。
3. 如果范围过大，优先缩小任务，不要直接全改。

### Step 3：实施优化

1. 先做 P0，再做 P1，最后视情况做 P2。
2. 删除死代码、调试残留、明显重复逻辑。
3. 对关键失败路径补充更明确的异常或上下文。
4. 必要时补最小必要测试来锁定行为。

### Step 4：验证结果

1. 优先运行与改动直接相关的最小测试集。
2. 明确说明哪些验证已完成。
3. 明确说明哪些验证尚未完成。
4. 在每次修改完成、准备提交代码前，必须运行 `pytest tests -n 8`。
5. 只有当 `pytest tests -n 8` 全部通过后，才可以将本轮任务视为可提交状态，并提交本轮对应的代码改动。
6. 如果失败，优先修复源码，不用跳过测试来制造假通过。
7. 提交时应只提交本轮任务实际涉及且已验证通过的相关代码，不夹带无关改动。

### Step 5：输出结果

必须输出结构化结果，不能只给一句“已经优化完成”。

## 7. 标准输出格式

每次执行代码质量改进任务时，至少应输出以下内容：

```text
## 本轮范围
- 本次处理的文件、模块或问题域

## 发现的问题
- P0：
- P1：
- P2：

## 本轮实施计划
- 本轮准备处理哪些项
- 哪些项暂不处理，以及原因

## 实际改动
- 改了哪些文件
- 每项改动解决了什么问题

## 风险与未处理项
- 当前仍有哪些风险
- 哪些问题被明确延期

## 验证结果
- 已执行的验证
- 未执行但建议补充的验证
- 是否已运行 `pytest tests -n 8`
- `pytest tests -n 8` 是否全部通过
- 当前是否已满足提交本轮对应代码的条件

## 下一轮建议
- 后续最值得继续推进的方向
```

## 8. 主 Prompt 模板

这是后续最推荐直接使用的主 Prompt。

```text
请基于 `backtrader/docs/_internal/opts/requirements/迭代5-优化代码质量/代码质量改进优化.md` 中定义的方法，
对以下目标范围执行一轮代码质量改进优化：

目标范围：{填写文件/目录/模块/问题域}
本轮主题：{例如：异常可见性治理、死代码清理、参数校验补强、主循环整理、契约一致性治理}
边界限制：{例如：仅允许小批次改动 / 允许补最小测试 / 不允许新增功能 / 尽量保持外部 API 不变}

请严格遵守以下要求：

1. 先审查目标范围内的代码、相关测试和主要调用方。
2. 识别问题并按 P0 / P1 / P2 分类。
3. 只提出并执行小批次、可验证、可回退的改动。
4. 优先处理正确性、稳定性、异常可见性、契约一致性问题。
5. 不做无关重构，不扩展新功能，不随意改变外部行为。
6. 如有必要，补最小必要测试来锁定行为。
7. 每次修改完成、准备提交前，必须运行 `pytest tests -n 8`，并要求测试全部通过。
8. 对已经完成的验证与尚未完成的验证，都要明确说明。
9. 最终必须按固定结构输出：
   - 本轮范围
   - 发现的问题
   - 本轮实施计划
   - 实际改动
   - 风险与未处理项
   - 验证结果
   - 下一轮建议

如果你判断这次任务范围过大，不要直接大改，而是先缩小范围并说明推荐的拆分方式。
```

## 13. 推荐的长期推进节奏

建议把代码质量优化拆成多轮推进：

1. 第一轮：异常可见性与正确性。
2. 第二轮：死代码、调试残留和明显重复逻辑。
3. 第三轮：抽象契约一致性与边界清晰化。
4. 第四轮：结构整理与可测试性补强。
5. 第五轮：低风险可读性提升和命名优化。

这样可以避免一次性大改，同时更容易验证收益和控制风险。

## 14. 使用时的硬约束

后续如果使用本文件指导代码优化，默认应遵守以下硬约束：

1. 不在单轮任务中同时处理过多主题。
2. 不在没有上下文的情况下直接重写核心文件。
3. 不用“更优雅”作为唯一理由发起大改。
4. 每次修改完成、准备提交代码前，必须运行 `pytest tests -n 8`。
5. 只有当 `pytest tests -n 8` 全部通过后，才可以视为本轮改动满足提交前要求，并提交本轮对应代码。
6. 提交时只提交本轮实际处理且已验证通过的相关改动，不夹带无关文件。
7. 不在缺乏验证证据时宣称优化完成。
8. 不把未解决问题伪装成已完成事项。
