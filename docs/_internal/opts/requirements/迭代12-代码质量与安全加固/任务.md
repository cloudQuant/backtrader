## 0. 背景与定位

迭代 4/5/11 已经把"仓库卫生、策略实例化、主循环可见性、Store 契约、注释完整性、
mypy 趋势"等问题推进到健康区间。本迭代换一个切入角度：**以安全（Security）为
主线、以质量（Quality）为辅线**，对 `dev` 分支做一轮系统性的加固。

之所以单独立项，是因为此前的迭代主要围绕"可维护性 / 正确性 / 注释"，而
`backtrader` 已经具备**实盘交易能力**（CTP/SimNow 穿透式认证、`BtApiStore` 实盘
下单、`TradeLogger` 的 MySQL 持久化、行情/数据源的网络请求）。这类涉及**资金、
凭证、网络、持久化**的路径，安全要求高于普通回测代码，需要专门治理。

### 与既有迭代的边界

| 已有迭代 | 已覆盖 | 本迭代不重复 |
|----------|--------|--------------|
| 迭代11 P2 | mypy 收敛 <120 | 本迭代只做**安全敏感模块**的类型补强，不抢通用收敛任务 |
| 迭代11 P3 | 低风险高复杂度收尾 | 本迭代只在安全敏感函数上顺带降复杂度 |
| 迭代11 P4 | 源码 TODO 清查 | 本迭代复用其结论，不重复清查 |
| 迭代5 | 代码质量方法论与 Prompt | 本迭代直接沿用其 P0/P1/P2 分级与提交门禁 |

> 任一时刻每个文件只允许一个执行者领取（沿用迭代11 台账机制）。

### 贯穿硬约束（与迭代5/11 一致，不可违反）

### P1 — 安全加固主线 🔴

#### S-1 异常可见性治理（静默吞异常）

- **现状**：`bandit` 报告 24 处 `try/except/pass`(B110) + 4 处 `try/except/continue`(B112)，
  另有 261 处 `except Exception` 宽捕获。分布上多数集中在 line 系统热路径
  （`linebuffer.py`、`lineiterator.py`、`functions.py`、`metabase.py`），少数在
  **安全敏感路径**（`observers/trade_logger.py:337`、`strategy.py:615/621`、
  `bokeh/app.py:344`、`utils/log_message.py:91`）。
- **风险**：热路径静默有其性能/兼容理由（已在 ruff `SIM105` ignore 中记录），
  但**交易/持久化/日志路径**的静默会导致"下单或落库失败却无声"，属于资金可见性隐患。
- **目标**：把静默异常**分类**，而非一刀切：
  1. **安全敏感路径**（trade_logger / strategy 交易钩子 / store）：静默改为
     `logger.debug/warning` 记录上下文（已部分如此，需补齐），保留 `pass/continue`
     的控制流不变。
  2. **line 系统热路径**：保持现状，但为每处补一行"为什么必须静默"的注释，
     并在该处加 `# nosec B110`（带理由），使 bandit Low 清单**可解释、可审计**。
- **边界**：只加日志/注释/nosec，**不改控制流、不改 except 捕获范围**。
- **验收**：`bandit -r backtrader -c pyproject.toml` 的 B110/B112 要么有 `# nosec`
  理由标注、要么已补日志；`pytest tests -n 8` 通过。

#### S-2 网络请求鲁棒性

- **现状**：`feeds/quandl.py:236` 的 `urlopen(url)` **未设置 timeout**；
  `utils/py3.py` 的 `urlopen` shim 直接透传，无默认超时。
- **风险**：远端无响应时连接无限挂起，拖死数据加载线程（DoS/hang）。
- **目标**：
  1. 为 `urlopen` 调用增加合理默认 `timeout`（如 30s），允许通过参数覆盖。
  2. 复核所有网络出口（quandl、influxfeed、其它 feeds/stores）是否都有超时与
     明确的异常处理；TLS 默认走系统校验（不得出现 `verify=False`，当前已无）。
- **边界**：保持默认行为兼容（仅新增超时，不改返回结构）。
- **验收**：网络调用均带超时；新增/补强对应单元测试（mock urlopen 验证 timeout 透传）。

#### S-3 实盘凭证暴露面收敛

- **现状**：`stores/btapistore.py` 把 CTP `password`/`auth_code` 以明文存于实例属性
  （`self.password`），已有 `_masked_account_id()` 对账号脱敏；但需确认：
  1. `__repr__`/`__str__`/异常信息/日志中**不会泄露 password/auth_code**。
  2. 凭证不会被 `emit_runtime_event` 等事件/通知通道带出。
  3. 配置文件/示例中不出现真实凭证（示例应使用占位符）。
- **风险**：实盘密码出现在日志、异常栈、事件流中会造成凭证泄露。
- **目标**：
  1. 审计 `btapistore.py` 全部日志/事件/异常路径，确保凭证字段被脱敏或排除。
  2. 为 store 增加 `__repr__` 安全实现（屏蔽 password/auth_code），并补回归测试
     断言 `repr(store)` 不含明文密码。
  3. 复核 `configs/` 与 `examples/` 不含真实凭证。
- **边界**：不改登录/认证逻辑与字段名，只收敛"对外可见"的表面。
- **验收**：新增 `tests/unit/stores/test_credential_safety.py`，断言 repr/日志样本
  不含明文凭证；`pytest` 通过。

#### S-4 / S-5 低危 triage 闭环（弱随机 + SQL 拼接）

- **S-4 弱随机（B311）**：`btrun.py:555` 用 `random.choice` 生成临时模块名，
  **非安全用途**。处置：加 `# nosec B311` 并注明"仅用于生成内部模块名，无安全意义"，
  让 bandit Low 清单收敛为"全部已 triage"。
- **S-5 SQL 拼接（B608）**：`influxfeed.py` 的 `.format()` 已有 `# nosec B608`
  （字段名来自参数而非用户输入）。处置：复核确认字段来源可信、补一行来源说明注释，
  确认 `trade_logger` 全链路参数化无遗漏。
- **验收**：`bandit -r backtrader -c pyproject.toml` 的 Low 清单中，每一条都能对应到
  "已修复"或"已 triage（带理由 nosec）"，无无主项。

## 4. 单批次执行流程（沿用迭代5）

### Step 1：审查现状

1. 阅读目标文件、相关测试、主要调用方。
2. 列出本批问题并按 P0/P1/P2 分类，明确在范围/不在范围。

### Step 2：制定最小批次计划

- "1 个主题 + 1–3 个源码文件 + 必要测试文件"，可验证、可回退。
- 出现"多主题/跨多核心文件/改对外契约/无法定义最小验证集"任一信号时，先缩范围。

### Step 3：实施

- 安全敏感路径：补日志/脱敏/超时/repr 安全；热路径：补 nosec + 理由注释。
- **不改控制流、不改捕获范围、不改字段名与数值逻辑。**

### Step 4：验证

```bash
# 定向
pytest <相关测试> -q
bandit -r backtrader -c pyproject.toml          # 复核 Low 清单收敛
ruff check backtrader                            # 必须 All checks passed
mypy backtrader --config-file=pyproject.toml | grep -c 'error:'   # 不回流
# 提交前硬门禁
pytest tests -n 8 -q
```

### Step 5：提交

- 仅 `pytest tests -n 8` 全绿后 `git commit`；只提交本批已验证改动；commit message 标注主题（如 `sec(s1): ...`）。

### Step 6：结构化输出

本轮范围 / 发现问题 / 实施计划 / 实际改动 / 风险与未处理 / 验证结果 / 下一轮建议。

## 6. 风险与应对

| 风险 | 应对 |
|------|------|
