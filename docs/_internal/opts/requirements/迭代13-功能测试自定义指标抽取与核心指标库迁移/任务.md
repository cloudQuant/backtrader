## 0. 背景与定位

迭代 10/11 把 1,152 个功能测试的**注释完整性与可读性**推进到健康区间，迭代 12 做完
**安全加固**。本迭代换一个切入角度：治理功能测试套件里**内联自定义指标的重复与漂移**。

`tests/functional/strategies/` 下的每个回归测试都是自包含的——内联策略 + 内联指标 +
`cerebro.run()` + 对 master 烘焙指标的精确断言。这种自包含设计在"单测可独立运行"上是
优点，但代价是：**同一个指标被独立地重写了很多遍**，彼此之间出现命名相同、实现漂移、
注释风格不一的情况，维护成本和"改一处漏一处"的风险随测试规模线性增长。

之所以单独立项，是因为这是一类**结构性重复**问题，既不属于注释治理（迭代10/11），也
不属于安全（迭代12），需要专门的去重 + 可选收编流程，且**必须**在严格的数值等价门禁
下进行——功能测试断言的是 master 烘焙的精确数字（`final_value` 容差 ~1e-6、
`trade_count/win/loss` 完全相等），任何改变指标数值输出的重构都会直接点爆测试。

### 与既有迭代的边界

| 已有迭代 | 已覆盖 | 本迭代不重复 |
|----------|--------|--------------|
| 迭代10/11 | 功能测试注释完整性与可读性 | 本迭代只动**指标定义的物理位置与重复**，不重写注释规范 |
| 迭代12 | 安全敏感模块加固 | 本迭代不碰凭证/网络/持久化路径 |
| 迭代5 | 代码质量方法论与分级门禁 | 本迭代直接沿用其 P0/P1/P2 分级与提交门禁 |
| 迭代6/7 | 逐 bar 对比工具链（branch_compare + TradeLogger） | 本迭代**复用**该工具链做数值等价证明，不重建 |

### 贯穿硬约束（与迭代5/11/12 一致，不可违反）

```text
[强制] 既有公共 API、导入路径、类继承、参数默认值、数值结果、订单逻辑、事件顺序均不变。
[强制] backtrader/ 核心库的既有指标行为一律冻结；允许新增指标模块、兼容 alias、
       兼容 wrapper，但不得修改既有指标类的数值语义。
[强制] 任何"去重/合并/迁移"在落盘前，必须对每个受影响测试运行 `pytest` 并全绿；
       涉及合并多份同名实现时，必须先证明逐 bar / 逐指标数值等价（见 §5 门禁）。
[强制] 不允许通过降低断言、放宽容差、跳过测试、xfail 来制造假通过。
[强制] 已回归通过的既有测试断言值不得修改；只允许移动指标定义的物理位置。
[强制] 小批次、可回退；一次只处理一组指标，不做大而全重构。
[强制] 台账纳入 git 跟踪；每个文件同一时刻只允许一个执行者领取。
```

### v1.1 方案改进摘要（2026-06-01）

本版在 v1.0 的安全框架上做方向修正：用户目标不是只减少测试内重复，而是让可复用指标
最终可以通过 `bt.indicators.xxx` 使用。因此执行主线从"测试侧共享去重为主、核心库收编
为辅"调整为：

```text
核心库迁移为主线：通用、稳定、可证明等价的指标优先迁入 backtrader/indicators。
测试共享模块降级为暂存区：只承接尚未证明可入库、策略专属或不等价的指标。
兼容 wrapper 必须保留旧测试语义：line 名、默认参数、warmup、NaN/0 边界行为不漂移。
不以删测试、放宽断言、降低 CI 检查换取通过。
```

v1.1 不推翻 v1.0 的数值门禁；它修正的是迁移目标和落点。与 v1.1 冲突的旧表述，以
本文 §11 的修订实施方案为准。

### v1.2 方案改进摘要（2026-06-01）

本版继续修正 v1.1 的边界：不再保留“D 层不入库”的最终状态。所有从
`tests/functional/strategies` 抽取出的内联指标，都要迁入 `backtrader/indicators`
体系内，只是分成两级 API：

```text
通用、稳定、可复用指标：
  backtrader/indicators/<topic>.py
  默认通过 bt.indicators.Xxx 暴露。

专属、少用、MT5 移植、策略信号、颜色/箭头/代理类指标：
  backtrader/indicators/contrib/<topic>.py
  物理存放在 contrib，但仍默认通过 bt.indicators.Xxx 暴露；
  bt.indicators.contrib.Xxx 作为次级显式入口保留。
```

也就是说，`contrib` 不是“不迁移”，而是低通用性指标的正式归宿。全量迁移矩阵见同目录
`指标迁移矩阵.md`。与 v1.2 冲突的旧表述，以 §12 为准。

### v1.3 方案改进摘要（2026-06-01）

本版继续收敛 API 目标：`contrib` 只表示源码物理归档位置，不表示用户导入边界。所有迁移
后的指标，无论源码位于顶层模块还是 `backtrader/indicators/contrib/`，都必须支持：

```python
bt.indicators.Xxx(...)
```

`bt.indicators.contrib.Xxx(...)` 可以保留为次级显式入口，便于识别来源，但不是唯一入口。

同名多实现的处理也调整为“以逻辑为准”：源码文本不同不等于需要拆分。迁移前必须验证
公式逻辑、边界处理、warmup、line 输出和代表性数据结果；逻辑一致且结果一致的副本合并
为一个 canonical。只有逻辑或结果确实不一致时，才拆 wrapper / variant。

## 1. 当前基线（2026-06-01 实测）

> 所有数字均在 `dev` 分支干净工作区下实测，不沿用历史文档数字。
> 测量口径：直接继承 `bt.Indicator` / `Indicator` 的内联类定义。

### 1.1 重复规模

| 维度 | 命令口径 | 实测 |
|------|----------|------|
| 功能测试策略文件总数 | `find tests/functional/strategies -name 'test_*.py'` | **1,152** |
| 含内联指标定义的文件数 | `grep -l 'class …(bt.Indicator)'` | **288** |
| 内联指标类定义总数 | 同上，计总数 | **325** |
| 去重后唯一指标名 | 同上，`sort -u` | **273** |
| 出现 ≥2 次的指标名（重复名） | 同上，`uniq -c \| awk '$1>=2'` | **28** |
| 可由去重消除的冗余副本 | Σ(count-1) over 重复名 | **52** |

### 1.2 重复最严重的指标名（count ≥ 3）

| 指标名 | 副本数 | 初判归类 |
|--------|--------|----------|
| `MoneyFlowIndex` | 7 | 通用技术指标（P1 候选） |
| `WeightedPrice` | 5 | 通用价格变换（P1 候选） |
| `SuperTrendIndicator` | 5 | 通用技术指标（P1 候选） |
| `DeMarkerIndicator` / `DeMarker` | 5 / 4 | 同一指标两种命名（P1 候选，需先统一名） |
| `SkyscraperFixIndicator` | 4 | MT5 移植/策略专属（P2 不入库） |
| `MFI` | 3 | 与 `MoneyFlowIndex` 疑似同义（P1，需合并判定） |
| `AlligatorIndicator` | 3 | 通用技术指标（P1 候选） |
| `XPeriodCandleColor` / `X2MACandleApprox` / `ColorAMLIndicator` / `AppliedPriceCCI` | 3 | 策略专属/信号代理（P2） |

### 1.3 与核心库撞名（测试重写了核心库已有指标）

`backtrader/indicators/` 已存在多组核心指标模块与 alias。以下内联指标名与核心库**已有类同名**：

| 内联名 | 核心库对应 | 风险 |
|--------|-----------|------|
| `AwesomeOscillator` | `indicators/awesomeoscillator.py` | 高：内联版与核心版很可能数值不一致，否则不会重写 |
| `AccelerationDecelerationOscillator` | `indicators/accdecoscillator.py` | 高：同上 |
| `LaguerreFilter` | `indicators/lrsi.py`（Laguerre 家族） | 高：同上 |

### 1.4 关键证据：同名 ≠ 同实现（决定整个迭代的设计）

对重复 7 次的 `MoneyFlowIndex`，逐副本 AST 抽取对比发现：**同名副本彼此文本不同**——
变量名（`positive_flow` vs `positive`）、循环写法（`range(self.p.period)` vs
`range(0, self.p.period)`）、零判定（`negative_flow == 0.0` vs `negative == 0`）、
docstring 均有差异。语义上大概率等价，但**不是字节级相同**，且零判定/边界写法的差异
意味着**不能假设数值一定相同**。

> 结论：不存在"按名字一键去重"的安全路径。每一次合并都必须以**数值等价证明**为前提
> （见 §5 门禁），无法证明等价的副本一律保留原样、不强行合并。

### 1.5 结论

- 重复真实存在且规模可观（288 文件 / 325 定义 / 52 冗余副本），值得立项。
- 但"抽取"和"迁移核心库"是两种风险等级，**必须拆开分级**，且全程受数值门禁约束。
- v1.1 目标是把通用指标最终迁入 `backtrader/indicators/`，让策略可以直接
  `bt.indicators.Xxx(...)`；测试侧共享模块只作为暂存/隔离机制，不作为通用指标的最终归宿。
- 复用工具链（branch_compare + TradeLogger）已就绪，无需新建基础设施。

## 2. 核心风险与设计原则

本迭代最容易踩的三个坑，以及对应的设计原则：

1. **坑：按名字盲目去重 → 数值漂移点爆测试。**
   原则：**先验证、再合并、再迁移**。同名源码不同不代表逻辑不同；先验证公式、warmup、
   边界处理和逐 bar 结果。逻辑与结果一致则合并 canonical；确实不一致才拆 wrapper / variant。

2. **坑：先创建测试共享模块 → 多一层临时架构，迁移路径变长。**
   原则：**直接迁移到目标 indicators 文件**。本迭代不再以 `tests/functional/_shared/indicators/`
   作为前置步骤；每个指标从测试内联实现直接迁入 `backtrader/indicators/<topic>.py` 或
   `backtrader/indicators/contrib/<topic>.py`。

3. **坑：把低通用性指标混入顶层主题文件 → 顶层模块膨胀且难维护。**
   原则：**物理分层、API 统一**。通用指标放顶层主题模块；专属/少用/MT5 移植/策略信号
   指标放 `contrib`。无论物理位置在哪里，最终都必须支持 `bt.indicators.Xxx`。

4. **坑：改核心库既有指标实现去“对齐”内联版 → 破坏公共 API 与数值冻结。**
   原则：**既有行为冻结，只新增/包装**。已有核心指标不改数值语义；撞名内联实现先做等价
   验证，等价则复用既有核心指标，不等价则新增明确命名的兼容 variant，不能覆盖旧类。

## 3. 问题清单与分级

### P0 — 全量 inventory 与直接迁移矩阵（前置，必须先做）🟢

目标不是创建共享模块，而是确认每个内联指标的**最终 indicators 归宿**。

- **P0-a** 维护 `指标迁移矩阵.md`：覆盖 325 个内联定义 / 273 个去重类名 / 23 个目标文件。
- **P0-b** 每个指标必须标记目标物理文件：顶层主题模块或 `contrib` 模块。
- **P0-c** 每个重复名必须标记验证状态：`待验证 / 逻辑一致已合并 / wrapper / variant / 撞名复用核心`。
- **P0-d** 每个指标必须标记消费测试清单，作为后续删除内联实现的依据。
- **验收**：矩阵无“待归类”项；没有指标以测试共享模块作为最终目标。

### P1 — indicators 导出骨架（直接迁移的基础设施）🟢

先打通目标包结构和统一导入路径，再迁移具体指标。

- **P1-a** 建立 `backtrader/indicators/contrib/__init__.py`，聚合 contrib 指标的 `__all__`。
- **P1-b** 在 `backtrader/indicators/__init__.py` 统一导出顶层主题模块和 contrib 模块，确保
  所有迁移后指标都能通过 `bt.indicators.Xxx` 使用。
- **P1-c** 约定冲突处理：contrib 指标名不得覆盖既有核心顶层指标；撞名时必须改为明确
  variant 名。
- **P1-d** 建立新增指标模板：`__all__`、alias、wrapper、docstring、line/params 兼容、
  warmup/NaN/除零行为说明。
- **验收**：空骨架或首批模块导入路径可用；不引入测试共享模块。

### P2 — 顶层通用指标直接迁移（主线第一阶段）🟢

把高复用、标准化程度高的指标直接迁入顶层主题模块，并把消费测试改为 `bt.indicators.Xxx`。

- **P2-a** 价格变换：`priceops_ext.py`，例如 `WeightedPrice`、`TypicalPrice`、`MedianPrice`、
  `HighLowAverage`。
- **P2-b** 资金流：`moneyflow.py`，例如 `MoneyFlowIndex` / `MFI`、`AccumulationDistributionLine`、
  `ChaikinOscillator`。
- **P2-c** DeMarker/WPR/Delta：`demarker.py`，例如 `DeMarker`、`DeMarkerIndicator`、
  `WilliamsPercentR`、`DeltaRSI`、`DeltaWPR`。
- **P2-d** 通道/波动带：`channels_ext.py`，例如 `DonchianChannel`、`KeltnerChannelIndicator`、
  `ChandelierExitIndicator`。
- **P2-e** 趋势/动量/均线/波动扩展：`trend_ext.py`、`momentum_ext.py`、`ma_ext.py`、
  `volatility_ext.py`。
- **验收**：已迁移指标不再在测试文件内定义；受影响测试只通过 `bt.indicators.Xxx` 使用；
  数值结果不漂移。

### P3 — contrib 指标直接迁移（主线第二阶段）🟢

低通用性指标也直接迁入 `backtrader/indicators/contrib/`，不再先进入测试共享模块。

- **P3-a** 策略信号/代理：`contrib/signals.py`。
- **P3-b** 蜡烛/图形/形态：`contrib/candles.py`。
- **P3-c** 少用振荡/颜色/直方图/状态型指标：`contrib/oscillators.py`。
- **P3-d** 非标准趋势系统：`contrib/trend_systems.py`。
- **P3-e** 自适应/滤波状态系统：`contrib/adaptive.py`。
- **P3-f** 其它专属系统：`contrib/machine_learning.py`、`contrib/volatility_systems.py`、
  `contrib/volume_systems.py`、`contrib/patterns.py`、`contrib/misc.py`。
- **验收**：虽然物理在 contrib，但消费测试优先使用 `bt.indicators.Xxx`；必要时增加少量
  `bt.indicators.contrib.Xxx` 导入测试，证明次级入口可用。

### P4 — 同名多实现合并与 variant 管理（贯穿 P2/P3）🟡

同名多实现不是单独迁移前置阶段，而是每迁移一个指标组时必须执行的验证动作。

- **P4-a** 对同名实现做逻辑对比：公式、输入价格源、窗口、平滑、状态递推、边界处理。
- **P4-b** 对同名实现做结果对比：消费测试数据 + 代表性数据集的逐 bar 输出。
- **P4-c** 逻辑和结果一致：合并为一个 canonical，保留 alias。
- **P4-d** 逻辑一致但 line/params 不同：canonical + wrapper。
- **P4-e** 逻辑或结果不一致：拆明确 variant，回写 `指标迁移矩阵.md`。
- **验收**：不存在“因为源码文本不同就直接拆 variant”的情况；以逻辑和结果为准。

### P5 — 防回流、文档与 CI/CD 验收（收尾）🟢

- **P5-a** 更新开发说明：新功能测试不得新增可复用内联 `Indicator`，应直接复用
  `bt.indicators.Xxx`。
- **P5-b** 可选增加非阻塞统计脚本：报告新增内联 `Indicator` 数量，先提示不阻塞。
- **P5-c** 全量本地和远端 CI/CD 验收：不删减 Python/OS 矩阵，不降低 lint、typing、安全、
  依赖审计、复杂度等检查。
- **验收**：远端 GitHub Actions 当前分支通过；迁移过程没有放宽断言、没有 skip/xfail、
  没有修改 golden 制造通过。

## 4. 目标架构

```text
backtrader/indicators/
  priceops_ext.py          # 通用价格变换
  moneyflow.py             # MFI / 资金流
  demarker.py              # DeMarker / WPR / Delta 派生
  channels_ext.py          # 通道/包络/止损带
  supertrend.py            # SuperTrend 家族
  trend_ext.py             # 通用趋势扩展
  momentum_ext.py          # 动量/振荡扩展
  ma_ext.py                # 均线/滤波扩展
  volatility_ext.py        # 波动率扩展
  contrib/
    __init__.py            # contrib 聚合导出，并被顶层 __init__.py re-export
    signals.py             # 策略信号/箭头/代理
    candles.py             # 蜡烛/图形/价格形态
    oscillators.py         # 少用振荡器/颜色/直方图/状态
    trend_systems.py       # 非标准趋势系统
    adaptive.py            # 自适应/滤波状态系统
    ...
```

设计要点：
- 所有迁移后的指标都属于 `backtrader/indicators` 体系，不再以测试共享模块作为最终归宿。
- 通用指标放顶层主题模块；专属/少用指标物理放 `contrib`。
- 顶层 `backtrader/indicators/__init__.py` 统一 re-export，保证 `bt.indicators.Xxx` 可用。
- `bt.indicators.contrib.Xxx` 保留为次级入口，便于识别 contrib 来源。
- 同名多实现先验证逻辑和结果；一致合并，不一致才拆 wrapper / variant。

## 5. 单批次执行流程（沿用迭代5，新增数值门禁）

### Step 1：审查现状

### Step 2：制定最小批次计划

### Step 3：实施
- 选 canonical 实现放入共享模块；逐个消费测试改 import。

### Step 4：验证（**强制数值门禁**）

### 11.2 核心库新增指标 API 规范

新增到 `backtrader/indicators/` 的指标必须满足：

```text
- 单独模块或主题模块中定义，模块有 __all__。
- 在 backtrader/indicators/__init__.py 暴露，支持 bt.indicators.Xxx。
- 保留常用 alias，例如 MFI = MoneyFlowIndex、DeMarkerIndicator = DeMarker。
- 参数默认值等于迁移前 canonical 实现。
- line 名不破坏旧测试；若同一指标历史 line 名不同，提供兼容 wrapper。
- warmup、NaN、除零、边界 bar 行为必须与消费测试原实现一致。
- 不引入 pandas/numpy 作为指标运行时依赖，除非该依赖已是核心指标既有约定。
```

建议模块划分：


### 11.4 单个指标迁移流程

```text
1. inventory：列出所有内联副本、消费测试、lines、params、minperiod/warmup、边界处理。
2. canonical：选一个 canonical 实现，说明选择原因。
3. equivalence：对所有消费测试证明 canonical 与旧实现等价；不等价则拆 variant/wrapper。
4. core add：新增 backtrader/indicators 模块和 __init__.py 导出。
5. test rewrite：消费测试删除本地 class，改用 bt.indicators.Xxx。
6. validation：受影响测试逐个通过，必要时跑 category/full tests。
7. record：在台账记录 canonical 来源、变体处理、验证命令和结论。
```

### 11.5 兼容 wrapper 规则

同一指标历史实现常见差异是 line 名和参数名，而不是公式本身。v1.1 要求优先通过 wrapper
保留旧接口，避免为了迁移修改测试断言。

示例原则：

```python
class DeMarker(Indicator):
    lines = ("demarker",)
    params = (("period", 14),)

class DeMarkerIndicator(DeMarker):
    lines = ("dem",)
```

实际实现时不要求照搬上例，必须以 backtrader line 机制可行和测试等价为准。

### 12.5 同名多实现处理

全量迁移不等于强行拆分。同名多实现必须先做逻辑等价和结果等价验证，再按下面规则处理：

```text
1. 如果源码不同但公式逻辑、warmup、边界处理和逐 bar 结果一致：合并为一个 canonical。
2. 如果公式逻辑一致、结果一致，但 line/params 命名不同：canonical + wrapper/alias。
3. 如果公式逻辑一致但历史默认参数不同：canonical + 参数化 wrapper。
4. 如果逻辑或逐 bar 结果确实不一致但同属一个概念：canonical + 明确命名 variant。
5. 如果只是历史命名相同但语义不同：保留不同类名，测试使用对应 variant。
```

同名多实现的验证维度：

| 维度 | 要求 |
|------|------|
| 公式逻辑 | 对比输入价格源、窗口长度、平滑方法、递推状态、除零处理 |
| warmup | 对比 `addminperiod`、首个有效 bar、前置 NaN/0 输出 |
| lines | 对比 line 名和语义；名称不同但语义一致时用 wrapper/alias |
| params | 对比默认参数；默认不同但公式一致时用参数化 wrapper |
| 逐 bar 结果 | 在消费测试数据和至少一个代表性数据集上对比输出序列 |
| 策略结果 | 受影响功能测试的 `final_value` / `trade_count` / `win` / `loss` 不漂移 |

示例命名：

```python
MoneyFlowIndex
MoneyFlowIndexClassic
MoneyFlowIndexPatternVariant

SuperTrend
SuperTrendDirection
SuperTrendBands

DeMarker
DeMarkerLine
DeMarkerSignalVariant
```

具体命名以迁移时的数值证据为准，不能为了名字漂亮牺牲等价。
