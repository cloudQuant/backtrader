## 1. 设计概览

### 1.1 设计原则

1. **研究先行**：先明确对标结论、现状差距和性能基线，再落地结构性改造
2. **共享核心优先**：高频撮合语义不直接散落在 broker 中，而是先抽成共享核心
3. **Broker 职责收敛**：`TickBroker` 与 `MixBroker` 负责接入与调度，不再承担全部撮合细节
4. **可插拔架构**：延迟模型、撮合模型、队列模型、状态追踪均通过接口抽象，可独立替换
5. **最小侵入**：不修改 `bbroker.py`、尽量不破坏 `Order` 核心语义，优先通过扩展和组合实现
6. **性能敏感**：基础模式不应显著慢于当前实现，增强能力必须支持按需启用
7. **结构解耦优先于旧继承关系**：`MixBroker` 是否继续继承 `TickBroker` 不再是硬约束
8. **模块归位清晰**：共享高频模块优先收敛到 `backtrader/brokers/hft/`，避免污染通用 broker 层命名空间

### 1.3 重分析后的设计结论

1. **当前 `TickBroker` 需要从“全量逻辑承载者”降级为“事件入口协调者”**
   - 当前实现将订单管理、成交逻辑、状态回写直接堆叠在 broker 内部，不利于扩展延迟、队列、maker/taker 等复杂语义。

2. **`MixBroker` 不应再被视为 `TickBroker` 的被动派生物**
   - 当前它主要增加 timeout + bar fallback，但本轮目标是让它表达独立的混合模式策略。
   - 最终是否使用继承，取决于共享核心能否将二者共同逻辑充分抽离。

3. **本轮先统一时间语义与订单生命周期，再叠加复杂功能**
   - 如果先做延迟/队列而不先统一时间与状态流转，后续会反复推翻实现。

4. **性能优化必须围绕数据结构和热路径组织展开**
   - 核心问题不是是否引入 numpy，而是先避免“每个事件全扫描所有 pending order”的结构性低效。

5. **研发建议中的实现细节应分层吸收，而不是直接变成硬承诺**
   - `orders_by_symbol`、`heapq` 延迟队列、`hft` 子包落位属于应直接吸收的结构建议。
   - `RandomLatency`、轻量级通知对象、批处理模式则保留为增强/候选优化项。

6. **延迟能力按“基础能力优先、增强能力后置”分层落地**
   - Phase 1/2 先提供零延迟、固定延迟以及订单可见时间规则，保证 P0 时序语义可验证。
   - 插值延迟、随机延迟等高级能力放在 Phase 4a 扩展，不阻塞基础 broker 重建。

#### 2.1.2 延迟引擎集成

```python
class LatencyEngine:
    """管理延迟调度的内部引擎。
    
    维护一个基于 heapq 的时间优先队列，延迟的事件按其“可见时间”排序。
    - 对于 feed：event 在 local_ts = exch_ts + feed_latency 时对策略可见
    - 对于 order：order 在 exch_receive_ts = local_ts + entry_latency 时到达交易所
    
    使用 heapq 保证 O(log n) 插入和 O(k) 弹出（k 为可见订单数）。
    """

    def __init__(self, latency_model=None):
        import heapq
        self._model = latency_model
        self._pending_orders = []  # heapq: (visible_ts, seq, order)
        self._seq = 0  # 序号，用于稳定排序
        self._current_time = 0.0

    def delay_order(self, order, submit_ts):
        """将订单放入延迟队列。"""
        import heapq
        if self._model is None:
            return None  # 无延迟，立即可见
        symbol = getattr(order, 'data', None)
        order_symbol = getattr(symbol, '_name', '') if symbol else ''
        entry_lat = self._model.order_entry_latency(submit_ts, order_symbol)
        visible_ts = submit_ts + entry_lat
        heapq.heappush(self._pending_orders, (visible_ts, self._seq, order))
        self._seq += 1
        return visible_ts

    def get_visible_orders(self, current_ts):
        """获取在 current_ts 已经到达交易所的订单。
        
        使用 heapq 的小根堆特性，只需检查堆顶元素，
        无需遍历整个队列。
        """
        import heapq
        visible = []
        while self._pending_orders and self._pending_orders[0][0] <= current_ts:
            _, _, order = heapq.heappop(self._pending_orders)
            visible.append(order)
        return visible

    def apply_feed_latency(self, event):
        """为事件添加 local_ts 字段。"""
        if self._model is None:
            return event
        feed_lat = self._model.feed_latency(event.timestamp, event.symbol)
        event.local_time = event.timestamp + feed_lat
        return event
```

#### 2.2.1 接口定义

```python
# backtrader/brokers/hft/queue.py

from abc import ABC, abstractmethod

class QueueModel(ABC):
    """队列位置模型基类。"""

    @abstractmethod
    def estimate_queue_position(self, order, ob_snapshot):
        """估计订单在某价格层级的排队位置。
        
        Args:
            order: 限价单
            ob_snapshot: 当前 orderbook 快照
            
        Returns:
            float: 估计的排队位置（前方数量）
        """
        ...

    @abstractmethod
    def update_on_trade(self, order, trade_event):
        """当 trade 事件发生时，更新排队位置。
        
        Args:
            order: 被追踪的限价单
            trade_event: 发生的 trade
            
        Returns:
            float: 可成交数量（如果 trade 消耗到了该订单位置）
        """
        ...


class NoQueueModel(QueueModel):
    """无队列模型（当前行为）。价格触及即全量成交。"""

    def estimate_queue_position(self, order, ob_snapshot):
        return 0.0  # 排在最前面

    def update_on_trade(self, order, trade_event):
        return order.remaining_size  # 全量可成交


class ProbQueueModel(QueueModel):
    """概率队列模型。
    
    基于订单提交时该价格层级的深度估算排队位置。
    假设订单排在当前深度的末尾。
    """

    def __init__(self, behind_fraction=1.0):
        """
        Args:
            behind_fraction: 假设排在现有深度的百分比后面 (0.0~1.0)
                           1.0 = 排在最后，0.5 = 排在中间
        """
        self._behind_fraction = behind_fraction

    def estimate_queue_position(self, order, ob_snapshot):
        # 查找该价格层级的当前深度
        price = order.price
        if order.isbuy():
            for p, q in ob_snapshot.bids:
                if abs(p - price) < 1e-10:
                    return q * self._behind_fraction
        else:
            for p, q in ob_snapshot.asks:
                if abs(p - price) < 1e-10:
                    return q * self._behind_fraction
        return 0.0

    def update_on_trade(self, order, trade_event):
        # 通过 trade 消耗排队位置
        queue_ahead = getattr(order, '_queue_ahead', 0.0)
        trade_qty = trade_event.volume

        if queue_ahead > 0:
            consumed = min(queue_ahead, trade_qty)
            order._queue_ahead = queue_ahead - consumed
            trade_qty -= consumed

        if trade_qty > 0 and order._queue_ahead <= 0:
            # 排队位置被消耗完，可以成交
            fillable = min(trade_qty, self._get_remaining(order))
            return fillable

        return 0.0
```

from abc import ABC, abstractmethod
from enum import Enum

class FillRole(Enum):
    MAKER = "maker"
    TAKER = "taker"


@dataclass
class OrderResult:
    """统一的订单处理结果对象，避免混合返回类型。"""
    action: str  # 'FILL', 'REJECT', 'PENDING'
    fills: list = None  # list of (fill_price, fill_size, FillRole)
    reject_reason: str = ""

    def __post_init__(self):
        if self.fills is None:
            self.fills = []


class ExchangeModel(ABC):
    """交易所撮合模型基类。"""

    @abstractmethod
    def on_new_order(self, order, ob_snapshot):
        """处理新订单。
        
        Returns:
            OrderResult: action 为 'FILL'/'REJECT'/'PENDING'
        """
        ...

    @abstractmethod
    def on_trade(self, trade_event, pending_orders):
        """处理 trade 事件，检查是否消耗到 maker 订单。
        
        Returns:
            list of (order, fill_price, fill_size, FillRole)
        """
        ...

    @abstractmethod
    def on_depth_update(self, ob_event, pending_orders):
        """处理深度更新事件。
        
        Returns:
            list of (order, fill_price, fill_size, FillRole)
        """
        ...

    """简单交易所模型（当前 TickBroker 行为的封装）。
    
    - 市价单立即与深度匹配（taker）
    - 限价单价格触及即成交（不排队）
    """
    ...


class QueueExchangeModel(ExchangeModel):
    """带队列的交易所模型（借鉴 hftbacktest）。
    
    - 市价单立即与深度匹配（taker）
    - 限价单提交后进入队列，需要 trade through 才能成交（maker）
    - 支持 GTX/FOK/IOC 等 TIF
    """

    def __init__(self, queue_model=None):
        self._queue_model = queue_model or ProbQueueModel()

    def on_new_order(self, order, ob_snapshot):
        if order.exectype == Order.Market:
            # Taker: 立即与深度匹配
            return self._match_against_depth(order, ob_snapshot, FillRole.TAKER)

        if order.exectype == Order.Limit:
            # 检查是否会穿过对手盘
            if self._crosses_spread(order, ob_snapshot):
                # 处理 TIF
                tif = getattr(order, 'time_in_force', 'GTC')
                if tif == 'GTX':
                    return OrderResult(action='REJECT', reject_reason='GTX_CROSSED')
                # 其他 TIF: 作为 taker 成交
                fills = self._match_against_depth(order, ob_snapshot, FillRole.TAKER)
                if tif == 'FOK' and not self._is_fully_filled(fills, order):
                    return OrderResult(action='REJECT', reject_reason='FOK_INSUFFICIENT')
                if tif == 'IOC':
                    return OrderResult(action='FILL', fills=fills)  # IOC: 部分成交后剩余取消
                return OrderResult(action='FILL', fills=fills)
            else:
                # Maker: 进入队列
                queue_pos = self._queue_model.estimate_queue_position(order, ob_snapshot)
                order._queue_ahead = queue_pos
                order._fill_role = FillRole.MAKER
                return OrderResult(action='PENDING')  # 进入 pending

    def on_trade(self, trade_event, pending_orders):
        fills = []
        for order in pending_orders:
            if not self._price_matches_trade(order, trade_event):
                continue
            fillable = self._queue_model.update_on_trade(order, trade_event)
            if fillable > 0:
                fills.append((order, trade_event.price, fillable, FillRole.MAKER))
        return fills
```

### 2.4 StateTracker 状态追踪

> **与现有 `_cash`/`_positions` 的关系**：StateTracker 作为**只读聚合视图**，
> 从 Broker 的 `_cash` 和 `_positions` 派生统计数据（fee、num_trades、trading_volume、trading_value）。
> position 和 balance 的权威源仍然是 Broker 内部的 `_cash` 和 `_positions`，
> StateTracker 不独立维护这两个值，避免双重记账导致状态不一致。

```python
# backtrader/brokers/hft/state.py

class StateTracker:
    """追踪每个资产的交易统计（只读聚合视图）。
    
    position/balance 从 Broker._cash 和 Broker._positions 派生，
    本类仅维护累计统计数据。
    """

    def __init__(self):
        self._states = {}  # symbol -> StateValues

    def get_state(self, symbol):
        if symbol not in self._states:
            self._states[symbol] = {
                'fee': 0.0,
                'num_trades': 0,
                'trading_volume': 0.0,
                'trading_value': 0.0,
            }
        return self._states[symbol]

    def on_fill(self, symbol, price, size, is_buy, commission, role):
        _ = (is_buy, role)
        state = self.get_state(symbol)
        state['num_trades'] += 1
        state['trading_volume'] += abs(size)
        state['trading_value'] += abs(size) * price
        state['fee'] += commission

    def snapshot(self, symbol, position, balance, mid_price):
        state = self.get_state(symbol)
        return {
            'position': position,
            'balance': balance,
            'fee': state['fee'],
            'num_trades': state['num_trades'],
            'trading_volume': state['trading_volume'],
            'trading_value': state['trading_value'],
            'equity': balance + position * mid_price,
        }
```

### 2.5 MatchingCore 共享撮合核心

#### 2.5.2 核心接口

```python
class MatchingCore:
    """共享高频撮合核心。
    
    职责：
    1. 管理 pending orders 索引（按 symbol 组织）
    2. 处理 Stop/StopLimit 触发状态
    3. 委托 ExchangeModel 执行 Market/Limit 撮合
    4. 委托 LatencyEngine 管理延迟队列
    5. 通知 StateTracker 更新统计
    6. 生成 FillReport / MatchResult / CancelResult
    
    不做：
    - 不直接操作 Broker 的 _cash / _positions
    - 不处理事件分发（由 Broker 负责）
    - 不处理策略通知（由 Broker 负责）
    """

    def __init__(self, exchange_model=None, latency_engine=None,
                 state_tracker=None):
        from .exchange import SimpleExchangeModel
        self._exchange = exchange_model or SimpleExchangeModel()
        self._latency = latency_engine  # None = 无延迟
        self._state = state_tracker  # None = 不追踪
        self._pending = {}  # symbol -> list of orders
        self._stop_orders = {}  # symbol -> list of stop/stoplimit orders

    def submit_order(self, order, current_ts, ob_snapshot=None):
        """提交新订单。
        
        处理流程：
        1. 如有延迟，放入延迟队列，返回 ACCEPTED
        2. Stop/StopLimit 订单进入 _stop_orders 等待触发
        3. Market/Limit 委托 ExchangeModel 处理
        
        Returns:
            MatchResult
        """
        symbol = self._get_symbol(order)

        # 延迟队列
        if self._latency is not None:
            visible_ts = self._latency.delay_order(order, current_ts)
            if visible_ts is not None:
                return MatchResult(action='ACCEPTED')

        # Stop/StopLimit: 进入触发等待队列
        if order.exectype in (Order.Stop, Order.StopLimit):
            self._stop_orders.setdefault(symbol, []).append(order)
            return MatchResult(action='ACCEPTED')

        # Market/Limit: 委托 ExchangeModel
        return self._process_new_order(order, ob_snapshot)

    def on_tick(self, tick_event):
        """处理 tick 事件。
        
        Returns:
            List[FillReport]: 本次事件产生的所有成交
        """
        symbol = tick_event.symbol
        all_fills = []

        # 1. 检查延迟队列中到达的订单
        if self._latency:
            arrived = self._latency.get_visible_orders(tick_event.timestamp)
            for order in arrived:
                result = self._activate_order(order, tick_event)
                all_fills.extend(result)

        # 2. 检查 Stop/StopLimit 触发
        triggered_fills = self._check_stop_triggers(symbol, tick_event)
        all_fills.extend(triggered_fills)

        # 3. 对 trade 类型 tick，检查 maker 订单成交
        if self._is_trade_tick(tick_event):
            fills = self._exchange.on_trade(tick_event,
                                           self._pending.get(symbol, []))
            all_fills.extend(self._to_fill_reports(fills, tick_event.timestamp))

        return all_fills

    def on_orderbook(self, ob_event):
        """处理 orderbook 事件。
        
        Returns:
            List[FillReport]
        """
        symbol = ob_event.symbol
        fills = self._exchange.on_depth_update(ob_event,
                                              self._pending.get(symbol, []))
        return self._to_fill_reports(fills, ob_event.timestamp)

    def on_trade(self, trade_event):
        """处理独立的 trade 事件。
        
        Returns:
            List[FillReport]
        """
        symbol = trade_event.symbol
        fills = self._exchange.on_trade(trade_event,
                                       self._pending.get(symbol, []))
        return self._to_fill_reports(fills, trade_event.timestamp)

    def cancel_order(self, order):
        """取消订单。
        
        Returns:
            CancelResult
        """
        symbol = self._get_symbol(order)
        # 尝试从 pending 移除
        pending = self._pending.get(symbol, [])
        if order in pending:
            pending.remove(order)
            return CancelResult(success=True)
        # 尝试从 stop_orders 移除
        stops = self._stop_orders.get(symbol, [])
        if order in stops:
            stops.remove(order)
            return CancelResult(success=True)
        return CancelResult(success=False, reason='ORDER_NOT_FOUND')

    def get_pending_orders(self, symbol=None):
        """获取待撮合订单。"""
        if symbol:
            return list(self._pending.get(symbol, []))
        return [o for orders in self._pending.values() for o in orders]

    # --- 内部方法 ---

    def _check_stop_triggers(self, symbol, tick_event):
        """检查 Stop/StopLimit 订单是否触发。
        
        Stop buy: 价格 >= stop_price 时触发，转为 Market
        Stop sell: 价格 <= stop_price 时触发，转为 Market
        StopLimit buy: 价格 >= stop_price 时触发，转为 Limit(price)
        StopLimit sell: 价格 <= stop_price 时触发，转为 Limit(price)
        """
        fills = []
        remaining = []
        for order in self._stop_orders.get(symbol, []):
            triggered = self._is_stop_triggered(order, tick_event.price)
            if triggered:
                # 转换订单类型并重新提交
                result = self._activate_triggered_stop(order, tick_event)
                fills.extend(result)
            else:
                remaining.append(order)
        self._stop_orders[symbol] = remaining
        return fills

    def _is_stop_triggered(self, order, price):
        """判断 Stop 订单是否触发。"""
        if order.isbuy():
            return price >= order.created.price  # stop_price
        else:
            return price <= order.created.price

    def _get_symbol(self, order):
        """从订单中提取 symbol。"""
        data = order.data
        return getattr(data, '_name', str(data))
```

TickBroker                          MatchingCore
    |                                    |

    |                                    |-- [latency?] delay_order()
    |                                    |-- [stop?] add to _stop_orders

    |<-- 对每个 fill 调用 _execute()    |
```

### 2.6 TickBroker 增强后的核心流程

#### 2.6.1 process_tick 新流程

```
process_tick(tick_event):
    1. 应用 feed 延迟（设置 local_ts）
    2. 更新 _last_tick
    3. 从延迟队列中取出已到达交易所的订单 → 加入 pending
    4. 对于 trade 类型的 tick:
       a. 调用 exchange_model.on_trade() 检查 maker 订单
       b. 对匹配的 maker 订单执行 _execute()
    5. 对于 quote 类型的 tick:
       a. 无直接撮合（仅更新深度）
    6. 通知策略
```

#### 2.6.2 process_orderbook 新流程

```
process_orderbook(ob_event):
    1. 应用 feed 延迟
    2. 更新 _last_orderbook
    3. 调用 exchange_model.on_depth_update() 检查是否有订单可成交
    4. 执行匹配的订单
```

#### 2.6.3 submit 新流程

```
submit(order):
    1. 设置订单状态为 Submitted
    2. 如果有延迟模型:
       a. 计算 entry_latency
       b. 将订单放入 latency_engine 延迟队列
       c. 订单进入 "in-flight" 状态
    3. 如果无延迟:
       a. 直接调用 exchange_model.on_new_order()
       b. 如果有即时成交（taker），执行 _execute()
       c. 否则加入 pending_orders

## 5. 数据流设计

### 5.1 事件处理时序（有延迟模型）

```
时间轴 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

交易所:  tick@t0 ─────────────────── order到达@t2 ── 撮合 ── response@t2
                \                   /                        \
                 \ feed_latency    / entry_latency            \ resp_latency
                  \               /                            \
策略:           tick到达@t1 → 决策下单@t1 ────────────── 收到确认@t3

t0 = exch_ts (交易所产生时间)
t1 = t0 + feed_latency (策略看到的时间)
t2 = t1 + order_entry_latency (交易所收到订单的时间)
t3 = t2 + order_response_latency (策略收到确认的时间)
```

```
                    ┌─────────────────────────────────────┐

                    │                                     │

                                                    │
                                    ┌───────────────┼───────────────┐
                                    │               │               │
                                  TAKER           MAKER           REJECT

                                              (maker成交)
```

## 6. 关键设计决策

### D-01: 延迟模型放在共享撮合核心 / Broker 侧，而非 Cerebro 层

**理由**：
- 延迟直接影响订单撮合时机，属于高频撮合核心关注点
- Cerebro 保持简单的事件分发职责
- 策略层无需关心延迟细节

### D-02: 优先建立共享撮合核心，再由 Broker 组合使用

**理由**：
- 避免把高频撮合能力分别复制到 `TickBroker` 和 `MixBroker`
- 共享核心更适合作为延迟、队列、交易所模型、状态追踪的统一承载层
- 便于后续继续扩展新的 broker 模式或仿真模式

### D-03: 队列位置通过 Order 扩展属性追踪

**理由**：
- 不修改 Order 核心类，通过 `order._queue_ahead` 等动态属性存储
- Order 类来自 backtrader 核心，修改风险高
- 动态属性在 Python 中性能可接受

### D-04: StateTracker 独立于 Position 类

**理由**：
- 现有 Position 类管理持仓和均价，职责清晰
- StateTracker 追踪累计交易统计，是更高层次的状态
- 独立组件更易测试和复用

### D-05: 默认使用 SimpleExchangeModel

**理由**：
- 确保零配置时行为与当前完全一致
- 用户需要显式启用队列模拟，避免意外行为变更
- 渐进增强策略

### D-06: `MixBroker` 架构方案应在 Phase 0 产出倾向性结论

**理由**：
- 当前继承关系来自实现便利，而不是明确的领域建模结果
- 如果共享核心已经抽离，`MixBroker` 继续继承 `TickBroker` 的价值会下降
- **Phase 0 应产出倾向性方案（推荐组合模式）**，Phase 1 据此设计共享核心接口
- Phase 3 验证并固化最终方案
- 完全延后到 Phase 3 会导致 Phase 1 共享核心设计缺少方向性约束

### D-07: 共享高频模块优先收敛到 `backtrader/brokers/hft/`

**理由**：
- 高频撮合能力是 broker 体系中的专门子域，单独成包更利于隔离复杂度
- 可以避免 `brokers/` 根目录充斥高频专用模块，提升结构辨识度
- 后续若新增 recorder、notification、stats 等能力，也有清晰落位

### D-08: maker/taker 手续费优先通过 CommissionInfo 扩展实现

**理由**：
- 费率模型天然属于佣金抽象，而不是撮合 broker 的核心职责
- 通过 `CommInfoBase` 子类扩展更符合 backtrader 既有设计风格
- 若第一版确实受限，broker 层可做过渡性兜底，但不应成为长期方案

## 7. 性能设计

### 7.1 热路径优化

| 路径 | 优化策略 |
|------|----------|
| `process_tick` 循环 | 按 symbol 索引 pending_orders（dict 而非 list） |
| 队列位置更新 | 批量处理同价格层级的 trade |
| 延迟队列 | 使用 `heapq` 保证 O(log n) 插入和弹出 |
| 深度匹配 | 限制遍历层级（`max_depth_levels`） |
| 通知回写 | 评估轻量级 fill notification，减少高频 `order.clone()` 开销 |

### 7.2 内存优化

- OrderHistory: 使用环形缓冲区限制大小（可配置 `max_history_size`）
- StateTracker: 仅存储当前值，时序数据由 Recorder 管理
- Recorder: 预分配 numpy array（类似 hftbacktest 的 Recorder）

### 7.3 可选加速

- 深度匹配可通过 numpy 向量化加速
- 延迟队列可使用 `sortedcontainers.SortedList`
- 未来可考虑 Cython 编译热路径

## 8. 错误处理

| 场景 | 处理方式 |
|------|----------|
| 修改不存在的订单 | 返回错误状态，记录 warning 日志 |
| GTX 订单被拒绝 | 订单状态设为 Rejected，通知策略 |
| FOK 流动性不足 | 订单状态设为 Rejected，通知策略 |
| 延迟导致订单过期 | 订单状态设为 Expired，通知策略 |
| 现金不足 | checksubmit 模式下拒绝订单 |
| 无效的延迟参数 | 初始化时 ValueError |
