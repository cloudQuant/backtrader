## 1. 背景与动机

### 1.1 项目现状

backtrader 项目当前有 4 个 Broker 实现：

| Broker | 文件 | 用途 | 成熟度 |
|--------|------|------|--------|
| BackBroker | `brokers/bbroker.py` | Bar/KLine 级别回测 | ★★★★★ 完善 |
| TickBroker | `brokers/tickbroker.py` | Tick/OrderBook 高频回测 | ★★☆☆☆ 基础可用 |
| MixBroker | `brokers/mixbroker.py` | 混合级别回测（tick优先+bar回退） | ★★☆☆☆ 基础可用 |
| BtApiBroker | `brokers/btapibroker.py` | 实盘交易 | ★★★☆☆ 可用 |

**核心问题**：TickBroker 虽已实现基本的 tick 撮合和 orderbook 深度撮合，但与业界顶尖的高频回测框架（如 hftbacktest）相比，在以下关键维度存在明显差距：

- **延迟建模**：缺少 feed latency 和 order latency 的模拟
- **队列位置模拟**：没有模拟限价单在 orderbook 中的排队位置
- **交易所撮合模型**：撮合逻辑过于简单，缺少 maker/taker 区分
- **Time-In-Force**：仅支持 GTC，缺少 GTX (Post-Only)、FOK、IOC
- **性能**：Python 层纯循环匹配，百万级 tick 数据回测性能不足
- **状态追踪**：缺乏完善的交易统计（trading_volume, trading_value, fee 累计等）
- **多资产**：不原生支持多资产并行回测

### 1.2 参考框架：py-hftbacktest

`py-hftbacktest` 是一个专为高频交易设计的回测框架，核心引擎用 Rust 实现（通过 maturin/PyO3 暴露给 Python），具有以下关键特性：

| 特性 | 实现方式 |
|------|----------|
| **事件驱动** | 统一 `event_dtype` (ev, exch_ts, local_ts, px, qty, order_id, ival, fval)，8 字段 numpy 结构体 |
| **双时间戳** | 区分 `exch_ts`（交易所时间）和 `local_ts`（本地时间），精确到纳秒 |
| **延迟建模** | `IntpOrderLatency` 模型：插值历史延迟数据，模拟 order entry/response 延迟 |
| **Feed 延迟** | `feed_latency()` 返回 (exch_ts, local_ts) 对，模拟行情传输延迟 |
| **Order 延迟** | `order_latency()` 返回 (req_ts, exch_ts, resp_ts) 三元组 |
| **深度模型** | `HashMapMarketDepth`（哈希表）和 `ROIVectorMarketDepth`（向量）两种深度实现 |
| **队列位置** | 支持 `PartialFillExchange` 模型，模拟限价单排队位置和部分成交 |
| **TIF 支持** | GTC, GTX (Post-Only), FOK, IOC |
| **Order 类型** | LIMIT, MARKET |
| **状态追踪** | `StateValues`：position, balance, fee, num_trades, trading_volume, trading_value |
| **时间控制** | `elapse()`, `elapse_bt()`, `wait_next_feed()`, `wait_order_response()` |
| **多资产** | 原生 `num_assets` + `asset_no` 索引 |
| **录制器** | `Recorder` 记录时序状态快照，支持 npz 导出 |
| **统计指标** | `Stats` 模块：SR, Sortino, MaxDrawdown, AnnualReturn, Turnover 等 |

### 1.3 差距分析总结

| 维度 | backtrader TickBroker | py-hftbacktest | 差距等级 |
|------|----------------------|----------------|----------|
| 延迟建模 | ❌ 无 | ✅ feed + order 延迟 | **P0 关键** |
| 队列位置 | ❌ 无 | ✅ PartialFillExchange | **P1 首批增强**（非 P0） |
| TIF 支持 | GTC 仅 | GTC/GTX/FOK/IOC | **P1 重要** |
| 交易所模型 | 简单价格匹配 | maker/taker + 深度消耗 | **P0 核心重构** |
| 状态追踪 | 基础 cash/position | 完整 StateValues | **P1 重要** |
| 多资产 | 按 symbol 过滤 | 原生 asset_no 索引 | **P2 改进** |
| 统计指标 | 无内置 | SR/Sortino/MDD/Turnover | **P2 改进** |
| Order Modify | ❌ 无 | ✅ modify() | **P2 改进** |
| 执行回写完整度 | 简化（无开/平仓拆分、无PnL计算、无杠杆/保证金） | N/A（Rust核心处理） | **P0 核心重构** |

### 1.4 重分析结论

基于对初始需求、当前 `tickbroker.py` / `mixbroker.py` 实现以及现有测试的再次分析，本次迭代计划需要做以下修正：

1. **本次不是给现有 TickBroker 做局部补丁，而是重构高频回测内核**
   - 当前 `TickBroker` 本质上仍是“按 tick / orderbook 事件直接扫描 pending_orders 并尝试成交”的简单实现。
   - 当前 `MixBroker` 只是基于 `TickBroker` 增加了 timeout + bar fallback，耦合较深。
   - 因此，本轮更合理的目标是先重构共享的高频撮合基础能力，再重建 `TickBroker` 和 `MixBroker`。

2. **`TickBroker` 与 `MixBroker` 均属于本轮改造范围**
   - 不再将 `MixBroker` 视为只能“被动兼容”的下游影响对象。
   - 是否继续保持 `MixBroker(TickBroker)` 继承关系，不再是硬约束；若共享核心 + 组合关系更合理，可以调整架构。

3. **现有 `tickbroker` / `mixbroker` 相关测试不再是强兼容约束**
   - 本轮允许替换或重写这两类测试。
   - 真正的约束应转为：新架构下的能力覆盖、验收场景完整、对非目标模块无无谓破坏。

4. **对标 `py-hftbacktest` 需要分阶段实现，而不是一次性承诺完全同等性能**
   - `py-hftbacktest` 依赖 Rust 核心；本项目当前约束仍以 Python 实现为主。
   - 因此更现实的目标应为：先补齐关键微观结构语义和架构扩展点，再通过基准测试衡量性能改进，而不是在本轮直接承诺达到其全部性能上限。

5. **必须新增“研究与基线阶段”作为正式交付物**
   - 初始需求第 1、2 条首先要求研究 `py-hftbacktest` 和现有 backtrader 代码。
   - 因此，迭代计划中必须包含：对标分析、差距矩阵、当前瓶颈基线、建议采纳与不采纳项。

6. **P0 能力必须和实施顺序保持一致**
   - 延迟建模不能整体后置到 Phase 4 才开始，否则 `TickBroker` / `MixBroker` 的 P0 时序语义无法在重建阶段被验证。
   - 因此本轮应拆分为：Phase 1/2 落地“零延迟/固定延迟 + 订单可见时间规则”的基础能力；Phase 4a 再补齐插值延迟、随机延迟等增强模型。
   - `StateTracker` 应作为对 Broker 权威状态的聚合视图，不应与 `_cash` / `_positions` 双重记账。

### 1.5 研发建议采纳结论

结合研发工程师的《改进建议》以及当前计划口径，本轮采纳结果如下：

1. **采纳：共享高频撮合能力放入专属 `hft` 子包**
   - 共享核心建议统一落位到 `backtrader/brokers/hft/`，避免把高频专用模块散落在 `brokers/` 根目录。
 
2. **采纳：基础热路径优化应前置为架构要求，而不是实现尾项**
   - `orders_by_symbol` 索引、基于 `heapq` 的延迟队列应视为共享核心/基础 broker 重建的一部分。
   - 这两项直接决定新架构是否具备可接受的热路径成本。
 
3. **部分采纳：队列位置模拟是高价值能力，但不应阻塞基础 broker 重建**
   - 该能力业务价值高，但实现复杂度明显高于延迟建模和基础撮合抽象。
   - 因此实施顺序上应归入 Phase 4b 的首批增强项，而不是阻塞 Phase 1-3。

4. **采纳：maker/taker 手续费优先走 CommissionInfo 扩展路径**
   - 优先考虑 `MakerTakerCommInfo` 一类的佣金模型扩展。
   - 若落地成本或兼容性过高，才允许 broker 层做过渡性兜底。

5. **采纳：性能目标采用“硬门槛 + 伸缩目标”双层表达**
   - 硬门槛：基础模式不能显著慢于当前实现。
   - 伸缩目标：在多品种/多订单场景下，通过索引与延迟队列优化争取 2-5x 的结构性收益，但不把该数值写成硬验收承诺。

### 2. 需求定义

### 2.1 P0 — 必须实现（核心竞争力）
 
#### R-P0-01：延迟建模基础能力
 
**描述**：为 TickBroker 添加 feed 延迟和 order 延迟建模，使回测结果更贴近真实市场。
 
**功能要求**：
1. **Feed 延迟基础能力**：模拟行情数据从交易所到策略的传输延迟
   - P0 至少支持零延迟和固定延迟 (`ConstantLatency`)
   - 事件需要具备 `exch_ts` / `local_ts` 或等价的可见时间语义
2. **Order 延迟基础能力**：模拟下单请求从策略到交易所的延迟，以及响应返回的延迟
   - 三阶段延迟：request_ts → exchange_ts → response_ts
   - P0 至少支持固定延迟模型
3. **延迟对撮合的影响**：
   - 订单在延迟期间不可见给交易所撮合引擎
   - 行情数据按 local_ts 而非 exch_ts 对策略可见
4. **后续增强（Phase 4a）**：
   - 基于历史数据的插值延迟 (`IntpLatency`)
   - 随机延迟等可选模型
 
**验收标准**：
- Phase 2 前必须完成：零延迟、固定延迟、订单可见时间规则三类场景验证
- 配置固定延迟后，订单的撮合时间应晚于 `submit_ts + order_entry_latency`
- feed 延迟导致策略看到的行情滞后于实际交易所时间
- 插值延迟和随机延迟属于 Phase 4a 增强验收项，不阻塞基础 broker 重建
 
#### R-P0-02：执行回写完整度修复
 
**描述**：当前 TickBroker._execute() 相比 bbroker._execute() 存在严重功能缺失，导致期货场景下回测结果不正确。
 
**问题分析**：当前 TickBroker._execute() 将所有成交一律当作 opened 处理，未区分开仓/平仓；PnL 固定传 0；无杠杆/保证金检查；未调用 order.addcomminfo()。
 
**功能要求**：
1. **开/平仓拆分**：正确区分 opened 和 closed，参照 bbroker 中 `position.pseudoupdate()` 逻辑
2. **PnL 计算**：平仓时通过 `comminfo.profitandloss()` 计算盈亏
3. **杠杆与保证金**：支持 `comminfo.get_leverage()` 和现金不足时的 margin 拒绝
4. **order.addcomminfo()**：成交后附加佣金信息
5. **OCO/Bracket 支持（P1）**：在共享核心稳定后接入
 
**验收标准**：
- 期货场景下平仓 PnL 计算正确
- 杠杆交易时保证金计算正确
- 现金不足时订单被正确拒绝
- order.addcomminfo() 被调用
 
#### R-P0-03：增强的交易所撮合模型
 
**描述**：升级撮合逻辑，区分 maker/taker 角色。
 
**功能要求**：
1. **Maker 订单**（限价单未穿过对手盘）：
   - 进入 orderbook 排队
   - 当对手方 trade 消耗到该价格时成交
2. **Taker 订单**（市价单或限价单穿过对手盘）：
   - 立即与 orderbook 深度匹配
   - 按深度逐层消耗
3. 手续费区分 maker fee 和 taker fee
 

**验收标准**：
- maker 限价单不会在提交瞬间成交（除非价格已穿过）
- taker 订单按深度逐层成交，加权均价正确
- 手续费按 maker/taker 分别计算

#### R-P0-04：Stop/StopLimit 订单在新架构中的处理

**描述**：当前 TickBroker 支持 Market、Limit、Stop、StopLimit 四种订单类型。新的 ExchangeModel 架构必须保持对 Stop/StopLimit 的支持。

**功能要求**：
1. Stop 订单：价格触发后转为 Market 单执行
2. StopLimit 订单：价格触发后转为 Limit 单，按 Limit 逻辑处理
3. 触发状态管理应在 MatchingCore 层处理，ExchangeModel 只负责 Market/Limit 撮合

**验收标准**：
- Stop buy 在价格上穿 stop_price 时触发并成交
- Stop sell 在价格下穿 stop_price 时触发并成交
- StopLimit 在触发后按限价逻辑撮合
- 触发状态在延迟模型下的时序正确

### 2.2 P1 — 应当实现（功能完善）

#### R-P1-01：Time-In-Force 扩展

**描述**：支持 GTX (Post-Only)、FOK (Fill-or-Kill)、IOC (Immediate-or-Cancel)。

**功能要求**：
1. **GTX (Post-Only)**：如果订单会立即成交（taker），则自动取消
2. **FOK**：必须全量成交，否则全部取消
3. **IOC**：立即成交尽可能多的量，剩余取消

**验收标准**：
- GTX 订单在穿过对手盘时被拒绝
- FOK 订单在流动性不足时被完全取消
- IOC 订单部分成交后剩余部分被取消

#### R-P1-02：完善的状态追踪 (StateValues)

**描述**：借鉴 hftbacktest 的 `StateValues`，为 TickBroker 添加完整的交易状态追踪。

**功能要求**：
1. 追踪以下累计状态值：
   - `fee`：累计手续费
   - `num_trades`：交易次数
   - `trading_volume`：总成交量
   - `trading_value`：总成交额
2. `position`、`balance` 继续以 Broker 的 `_positions` / `_cash` 为权威来源
3. 提供 `state_values()` 接口供策略和录制器查询，对外返回聚合后的统一视图：
   - `position`：来自 Broker 当前持仓快照
   - `balance`：来自 Broker 当前现金快照
   - `fee` / `num_trades` / `trading_volume` / `trading_value`：来自 StateTracker 累计值
   - `equity`：由 `balance + position * mid_price` 计算
4. 避免 Broker 与 StateTracker 双重记账导致状态漂移

**验收标准**：
- `state_values()` 返回的 `position` / `balance` 与 Broker 权威状态一致
- `fee` / `num_trades` / `trading_volume` / `trading_value` 能在多次部分成交后正确累计
- `equity = balance + position * mid_price` 恒等式成立

#### R-P1-03：Order Modify 支持

**描述**：支持修改已提交订单的价格和数量。

**功能要求**：
1. `modify(order, new_price, new_qty)` 方法
2. 修改后订单在队列中的位置根据策略可配置（保持/重置）
3. 不支持修改订单方向（buy/sell）

**验收标准**：
- 修改价格后订单按新价格撮合
- 修改数量后剩余量正确更新
- 修改不存在的订单返回错误

#### R-P1-04：Maker/Taker 手续费模型

**描述**：扩展 CommissionInfo 支持 maker/taker 分层手续费。

**功能要求**：
1. 新增 `maker_commission` 和 `taker_commission` 参数
2. 负 maker fee（返佣）支持
3. 与现有 CommissionInfo 体系兼容

**验收标准**：
- maker 订单使用 maker fee，taker 订单使用 taker fee
- 负 maker fee 正确增加 balance
- 不设置分层费率时回退到统一费率

### 2.3 P2 — 可选实现（性能与体验）

#### R-P2-02：时序状态录制器 (Recorder)

**描述**：借鉴 hftbacktest 的 Recorder，记录时序状态快照。

**功能要求**：
1. 可配置采样间隔（每 N 个 tick 或每 T 毫秒）
2. 记录 timestamp, mid_price, position, balance, fee, equity
3. 支持导出为 CSV/NPZ 格式
4. 与 hftbacktest Stats 模块的数据格式兼容

#### R-P2-03：内置统计指标

**描述**：提供常用高频交易统计指标计算。

**功能要求**：
1. Sharpe Ratio、Sortino Ratio
2. Maximum Drawdown、Return over MDD
3. 日均交易量、日均换手率
4. 最大持仓、平均持仓

#### R-P2-04：多资产回测增强

**描述**：优化多资产并行回测性能和 API。

**功能要求**：
1. 多资产独立状态管理（借鉴 hftbacktest 的 `asset_no` 模式）
2. 跨资产信号支持（一个资产的行情触发另一个资产的交易）
3. 统一的多资产 position/balance 查询接口

## 4. 非功能需求

### 4.2 可观测性

- 关键路径日志：订单提交、撮合、成交使用 `logging.debug`
- 延迟相关事件使用独立 logger：`backtrader.latency`
- 性能瓶颈点提供 profiling hook

### 4.3 可扩展性

- 延迟模型可插拔（实现 `LatencyModel` 接口即可替换）
- 撮合模型可插拔（实现 `ExchangeModel` 接口即可替换）
- 队列模型可插拔（实现 `QueueModel` 接口即可替换）

### 4.4 性能目标修正

- 本轮性能目标从“直接达到 `py-hftbacktest` 水平”修正为“建立可量化基线并取得明确改进”
- 至少需要提供以下基准对比：
  - 当前 `TickBroker` 基线
  - 重构后基础模式（无延迟、无队列）
  - 重构后增强模式（启用延迟 / 队列 / maker-taker）
- 验收重点是：
  - 基础模式吞吐不应显著差于当前实现
  - 增强模式开销可解释、可配置、可关闭
- 可作为伸缩目标记录但不写成硬门槛的是：
  - 在多品种、多订单热路径下，通过 symbol 索引和延迟队列优化争取 2-5x 的结构性收益

## 5. 风险评估

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| TickBroker 与 MixBroker 同时重构导致范围膨胀 | 高 | 两个 broker 同时演进，若没有共享核心和阶段边界，容易失控 | 先做研究与共享核心，再分阶段重建两个 broker |
| TickBroker._execute() 功能缺失 | 高 | 期货场景下 PnL、杠杆、保证金计算均不正确，影响回测结果可信度 | Phase 2 中优先重构 _execute()，参照 bbroker 实现 |
| Order/Commission 扩展与现有抽象冲突 | 中 | 若直接改动核心类，可能波及非目标模块 | 优先使用 `addinfo()`、可选属性和 `CommInfoBase` 子类扩展 |
| 延迟建模增加撮合复杂度 | 中 | 可能引入难以调试的时序 bug | 设计清晰的时间线可视化/日志 |
| 性能下降（新增逻辑） | 中 | 延迟模拟和队列计算增加 CPU 开销 | 默认关闭，提供 bypass 路径 |
| 与 channel 模式集成 | 中 | Cerebro `_run_channel()` 需要配合新 broker 行为 | 早期集成测试 |
| 队列模型精度与复杂度不平衡 | 中 | 可能实现复杂但结果解释性不足 | 先落地可解释的基础队列模型，再逐步增强 |
| 参数爆炸 | 低 | 新增大量配置参数导致 API 复杂 | 合理分组，文档清晰 |

## 7. 术语表

| 术语 | 定义 |
|------|------|
| **Feed Latency** | 行情数据从交易所产生到策略接收的延迟 |
| **Order Latency** | 订单从策略发出到交易所确认的延迟 |
| **Queue Position** | 限价单在某价格层级 orderbook 中的排队位置 |
| **Maker** | 提供流动性的订单（限价单挂在盘口） |
| **Taker** | 消耗流动性的订单（市价单或穿过对手盘的限价单） |
| **GTX (Post-Only)** | 仅做 maker 的订单类型，如果会立即成交则取消 |
| **FOK (Fill-or-Kill)** | 全量成交或全量取消 |
| **IOC (Immediate-or-Cancel)** | 立即成交尽可能多，剩余取消 |
| **Trade Through** | 交易价格穿过某限价单的挂单价格 |
| **exch_ts / local_ts** | 交易所时间戳 / 本地接收时间戳 |
