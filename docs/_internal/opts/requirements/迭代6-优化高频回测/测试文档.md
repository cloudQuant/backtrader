## 1. 测试策略

### 1.1 总体原则

| 原则 | 说明 |
|------|------|
| **能力优先** | 验证目标是覆盖新架构能力，而不是维持旧测试文件不变 |
| **测试先行** | 共享核心、Broker 重建、增强能力都应先定义场景再实现 |
| **分层验证** | 采用组件测试、集成测试、端到端测试、基准测试四层结构 |
| **允许替换旧测试** | `tickbroker` / `mixbroker` 相关旧测试可被替换，但必须补齐新的能力测试 |
| **非目标模块稳态** | 除目标模块外，其他回归尽量保持稳定 |
| **基线对比** | 性能评估以“当前实现基线 vs 新实现”对比为主 |
| **可解释性** | 时序、拒单、撤单、部分成交、回退行为都必须可验证和可解释 |

### 1.2 测试分层

```text
Layer 4: 端到端 / 基准测试
  - 真实 tick / orderbook / bar 数据回测
  - 基线性能对比
  - 增强模式附加开销评估

Layer 3: 集成测试
  - MatchingCore + TickBroker
  - MatchingCore + MixBroker
  - 延迟 / 队列 / 交易所模型组合场景
  - Cerebro channel 模式集成

Layer 2: 组件测试
  - LatencyEngine
  - QueueModel
  - ExchangeModel
  - StateTracker
  - MatchingCore 生命周期流转

Layer 1: 单元测试
  - 固定延迟 / 插值延迟计算
  - 概率队列 / 无队列
  - maker / taker 判定
  - TIF 规则判断
```

### 1.3 测试口径修正

1. **`TickBroker` 与 `MixBroker` 都在本轮重构范围内**
   - 不要求沿用旧测试文件或旧内部实现路径。
   - 若外部行为契约调整，测试需同步重写。

2. **新的测试矩阵按能力组织**
   - 订单生命周期
   - tick / trade / orderbook / bar 事件驱动
   - tick-first + bar-fallback
   - 延迟、队列、maker-taker、TIF、状态追踪

3. **性能测试强调相对结果**
   - 当前实现基线
   - 新基础模式（无延迟、无队列）
   - 新增强模式（延迟 / 队列 / maker-taker / TIF）

4. **选择性采纳的建议同样需要测试表达**
   - `orders_by_symbol` 和 `heapq` 延迟队列属于结构性优化，应通过共享核心与热路径场景验证其行为正确。
   - `RandomLatency`、轻量级通知对象、批处理模式属于候选增强项，测试中先保留扩展位，不作为本轮硬门槛。

5. **延迟测试分两阶段推进**
   - Phase 1/2 必须覆盖零延迟、固定延迟和订单可见时间规则。
   - Phase 4a 再补齐插值延迟、随机延迟等增强场景。

## 3. 示例测试片段

```python
def test_tickbroker_market_path(data):
    broker = TickBroker(cash=1000.0)
    order = broker.buy(owner=None, data=data, size=2, price=100.0,
                       exectype=Order.Market)

    tick = TickEvent(timestamp=1.0, symbol=data._name, price=101.5, volume=3.0)
    broker.process_tick(tick)

    assert order.status == Order.Completed
    assert broker.getcash() == pytest.approx(1000.0 - (101.5 * 2))


def test_tickbroker_with_constant_latency(data):
    latency = ConstantLatencyModel(order_entry_latency_ms=100)
    broker = TickBroker(cash=10000.0, latency_model=latency)

    order = broker.buy(owner=None, data=data, size=1, price=100.0,
                       exectype=Order.Market)

    broker.process_tick(TickEvent(timestamp=1.0, symbol=data._name, price=100.0, volume=10.0))
    assert order.status != Order.Completed

    broker.process_tick(TickEvent(timestamp=1.2, symbol=data._name, price=100.5, volume=10.0))
    assert order.status == Order.Completed


def test_mixbroker_bar_fallback(data, bar_event):
    from backtrader.brokers.mixbroker import MixBroker

    broker = MixBroker(cash=5000.0, tick_timeout=1.0, bar_fallback=True)
    order = broker.buy(owner=None, data=data, size=2, price=100.0,
                       exectype=Order.Limit)

    broker.process_bar(bar_event)
    assert order.status == Order.Completed
```

## 7. 覆盖率与风险

### 7.1 覆盖率目标

| 模块 | 目标覆盖率 |
|------|-----------|
| `matching_core.py` | > 85% |
| `latency.py` | > 90% |

| `tickbroker.py` 新增代码 | > 80% |
| `mixbroker.py` 新增代码 | > 80% |

### 7.2 已知测试风险

| 风险 | 缓解 |
|------|------|
| 浮点误差 | 所有数值比较使用 `pytest.approx()` |
| 时间戳边界问题 | 包含相同时间戳、刚好超时等边界场景 |
| Mock 与真实 Order 行为差异 | Broker 集成测试尽量使用真实订单对象 |
| 性能基准受机器负载影响 | 记录基线并做相对对比，不只看绝对阈值 |
| `MixBroker` 可能从继承改为组合 | 测试按行为契约编写，不绑定类层次 |
