# 迭代6：高频回测基准结果

## 1. 基准脚本

本轮基准脚本位于：

- `tests/bench/bench_tickbroker.py`

执行命令：

```bash
python tests/bench/bench_tickbroker.py
```

## 2. 实测结果

本轮在当前开发环境下执行结果如下：

```text
TickBroker benchmark
[baseline-100k] ticks=100000 orders=1000 elapsed=0.2334s rate=428363.31/s
[enhanced-100k] ticks=100000 orders=1000 elapsed=0.2797s rate=357555.58/s
[baseline-1m] ticks=1000000 orders=10000 elapsed=2.1862s rate=457420.26/s
[enhanced-1m] ticks=1000000 orders=10000 elapsed=2.6188s rate=381854.81/s
```

## 3. 结果解读

### 3.1 基础模式

- `100K` tick：约 `42.8 万 events/s`
- `1M` tick：约 `45.7 万 events/s`

说明基础模式下，新实现已经具备稳定的热路径处理能力。

### 3.2 增强模式

增强模式启用：

- 固定订单入场延迟
- `QueueExchangeModel`
- 更完整的订单语义路径

结果：

- `100K` tick：约 `35.8 万 events/s`
- `1M` tick：约 `38.2 万 events/s`

### 3.3 相对开销

相对于基础模式：

- `100K` 场景下降约 `16.5%`
- `1M` 场景下降约 `16.5%`

该开销与新增的延迟、队列和 TIF 语义一致，属于可解释范围。

## 4. 结论

- 新基础模式没有出现不可接受的性能退化
- 新增强模式的附加成本与特性复杂度匹配
- 当前结果满足“以相对基线为目标”的迭代要求

## 5. 后续优化方向

可继续优化的点：

- 将更多撮合判断下沉到共享核心并减少 broker 层重复判断
- 对高频订单簿路径做更细粒度的价格层级索引
- 进一步减少通知与历史记录对象构造开销
- 在未来迭代评估批量事件处理/向量化优化
