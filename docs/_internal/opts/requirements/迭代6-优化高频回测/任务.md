## 1. 实施总策略

### 1.1 本轮执行定位

本轮迭代不再按“给现有 `TickBroker` 打补丁”的方式推进，而是按以下路径实施：

1. **先研究，再重构**
   - 先产出 `py-hftbacktest` 对标结论、当前 backtrader 高频回测差距矩阵、性能与语义基线。

2. **先抽共享核心，再重建 Broker**
   - 优先抽离高频撮合公共能力，再分别重建 `TickBroker` 和 `MixBroker`。

3. **允许重写目标模块测试**
   - `tickbroker` / `mixbroker` 旧测试不再作为硬门槛。
   - 验收依据切换为新的能力场景测试矩阵。

4. **以“可验证的阶段结果”为主，而非一次性堆满功能点**
   - 先完成共享内核、基础时间/延迟语义与基础 broker 重建，再追加插值延迟、队列、maker/taker、TIF 等增强能力。

### 1.2 强制约束

```text
[强制] 不修改 bbroker.py 的既有职责边界
[强制] 保持 TickEvent / OrderBookSnapshot 等核心事件接口兼容
[强制] 保持 cerebro channel 运行入口不变
[强制] tickbroker / mixbroker 相关旧测试允许替换，但必须补足新的场景测试
[强制] 每个阶段结束时都要有可运行、可验证、可回顾的交付物
```

### 1.3 阶段门禁原则

| 门禁项 | 含义 |
|------|------|
| **研究门禁** | 没有完成对标分析与基线报告前，不进入结构性重构 |
| **架构门禁** | 没有稳定的共享撮合核心前，不分别重建 TickBroker / MixBroker |
| **能力门禁** | 没有完成基础撮合语义前，不叠加复杂微观结构特性 |
| **验证门禁** | 没有新的测试矩阵和基准结果，不视为阶段完成 |

## Phase 1：共享高频撮合核心重构（预计 4-5 天）

### 目标

抽离出可供 `TickBroker` 与 `MixBroker` 共同依赖的高频撮合核心，解决当前实现中逻辑散落、流程耦合、扩展点不足的问题，并前置落地 P0 所需的基础时间/延迟语义。

### 任务清单

| 编号 | 任务 | 输出 | 验收方式 |
|------|------|------|----------|
| 1.1 | 设计共享撮合核心的职责边界 | 核心模块职责表 | 职责不与 broker 混淆 |
| 1.2 | 统一订单生命周期状态机 | 状态流转定义 | 提交/排队/成交/取消语义明确 |
| 1.3 | 统一事件时间与基础延迟语义 | `exch_ts` / `local_ts` / 可见时间规则 | 时序逻辑可落地 |
| 1.4 | 落地 P0 延迟基础设施 | `LatencyModel` / `LatencyEngine` 的零延迟与固定延迟能力 | Phase 2 可直接消费 |
| 1.5 | 设计通用撮合结果对象 | fill report / reject report / cancel report | Broker 可消费 |
| 1.6 | 重构 pending order 组织方式 | symbol 维度或价格层级索引 | 具备后续性能优化空间 |
| 1.7 | 实现 MatchingCore 核心接口 | submit_order/on_tick/on_orderbook/cancel_order | 能独立驱动订单生命周期 |
| 1.8 | 实现 Stop/StopLimit 触发状态管理 | _check_stop_triggers / _is_stop_triggered | Stop 触发后正确转换为 Market/Limit |
| 1.9 | 实现 FillReport/MatchResult/CancelResult | 统一结果对象 | Broker 可消费且类型安全 |

### 建议模块划分

- `backtrader/brokers/hft/matching_core.py`：共享撮合主流程
- `backtrader/brokers/hft/exchange.py`：交易所撮合规则
- `backtrader/brokers/hft/queue.py`：队列位置模型
- `backtrader/brokers/hft/latency.py`：延迟模型与调度
- `backtrader/brokers/hft/state.py`：状态累计与快照
- `backtrader/brokers/hft/recorder.py`：录制器（P2）

### 阶段完成标准

- 共享核心能够独立驱动订单提交、进入待撮合、返回成交结果
- 共享核心具备零延迟与固定延迟下的订单可见性控制
- MatchingCore 能处理 Market/Limit/Stop/StopLimit 四种订单类型
- 不依赖 `TickBroker` / `MixBroker` 的具体实现细节
- 新增核心模块具备独立组件测试
- 相关文档已同步更新

## Phase 2：TickBroker 重建（预计 4-5 天）

### 目标

基于共享核心重建 `TickBroker`，让它成为“tick / trade / orderbook 高频回测入口”，而不是直接承担所有撮合细节。

### 任务清单

| 编号 | 任务 | 输出 | 验收方式 |
|------|------|------|----------|
| 2.1 | 重构 `submit()` / `cancel()` / 通知流程 | 新订单入口实现 | 生命周期一致 |
| 2.2 | 重构 `process_tick()` | tick 事件驱动路径 + 延迟到达订单激活路径 | 可驱动基础撮合 |
| 2.3 | 重构 `process_orderbook()` | 深度驱动路径 | 可处理多层深度成交 |
| 2.4 | 前置热路径优化 | symbol 索引 + `heapq` 延迟队列 | 不再依赖全量扫描 |
| 2.5 | 重构 `_execute()` 执行回写 | 支持开/平仓拆分、PnL计算、杠杆/保证金、addcomminfo | cash/position/order/PnL 状态全部正确 |
| 2.5a | — P0: 开/平仓拆分和 PnL 计算 | 参照 bbroker position.pseudoupdate() | 平仓时 pnl 正确 |
| 2.5b | — P0: 杠杆和保证金检查 | comminfo.get_leverage() + margin check | 期货场景正确 |
| 2.5c | — P1: OCO/Bracket 支持 | _ococheck / _bracketize | 在共享核心稳定后接入 |
| 2.6 | 为 maker/taker 手续费与 modify 预留挂接点 | Commission/Order 扩展接口 | Phase 4 可平滑接入 |
| 2.7 | 重建 `TickBroker` 专属测试 | 新测试文件与场景 | 覆盖基础能力 |

### 本阶段验收重点

- 市价单、限价单、止损单、止损限价单基础语义成立
- tick 与 orderbook 两条路径都可驱动成交
- 固定延迟下的订单可见时间规则成立
- `TickBroker` 自身不再包含难以复用的复杂撮合分支

## Phase 3：MixBroker 重建（预计 2-4 天）

### 目标

重构 `MixBroker`，明确其职责是“协调 tick 优先与 bar 回退”，而不是复制 `TickBroker` 的撮合细节。

### 关键决策点

| 决策项 | 备选方案 | 推荐方向 |
|------|----------|----------|
| 结构关系 | 继续继承 `TickBroker` / 改为组合共享核心 | 优先评估组合 |
| 回退策略 | 固定 timeout / 配置化 timeout + 模式 | 配置化 |
| 回退边界 | 全部订单回退 / 仅部分订单类型回退 | 明确按订单语义定义 |

### 任务清单

| 编号 | 任务 | 输出 | 验收方式 |
|------|------|------|----------|
| 3.1 | 明确 `MixBroker` 架构方案 | 继承或组合的决策记录 | 设计一致 |
| 3.2 | 实现 tick 优先 + bar fallback 主流程 | 新 `MixBroker` 实现 | 时序行为可验证 |
| 3.3 | 定义 timeout、回退窗口、状态同步规则 | 回退策略说明 | 无二义性 |
| 3.4 | 重建 `MixBroker` 测试 | 新混合场景测试 | 覆盖混合能力 |

### 本阶段验收重点

- tick 路径优先
- 超时或缺失 tick 时可落到 bar 回退
- 回退执行不破坏订单状态与持仓状态

## 4. 阶段依赖关系

```text
Phase 0 研究与基线
    ↓
Phase 1 共享高频撮合核心重构（含 MatchingCore + 基础延迟语义）
    ↓
Phase 2 TickBroker 重建（含 _execute() 重构）
    ↓
Phase 3 MixBroker 重建（依赖 Phase 2 稳定）
    ↓
Phase 4a 微观结构增强（延迟增强 + 状态视图）
    ↓
Phase 4b 微观结构增强（队列 + maker/taker + TIF + modify）
    ↓
Phase 5 验证、基准与收尾
```

补充依赖说明：

- `LatencyModel`、`LatencyEngine`、`QueueModel`、`ExchangeModel` 应依附共享核心，不应直接散落在 broker 层。
- 零延迟/固定延迟能力应在 Phase 1 落地，插值/随机延迟在 Phase 4a 完整落地。
- `StateTracker` 可以在 Phase 1 预留接口，在 Phase 4a 完整落地统一视图。
- maker/taker 手续费优先通过 `CommInfoBase` 子类扩展；broker 层内联计算只作为过渡性兜底方案。
- `modify` 语义优先定义为“对外单次操作、内部 cancel + re-submit”，避免在第一版引入过重的原地改单状态机。
- `Recorder`、统计指标、多资产增强建议放在本轮尾部或后续迭代，不阻塞主链路。
- Phase 2 和 Phase 3 为串行依赖：MixBroker 依赖 TickBroker 稳定后才能开始。
- 尾部性能优化（轻量级 fill notification、numpy 批处理）并入 Phase 5。

## 6. 风险与应对

| 风险 | 等级 | 影响 | 应对策略 |
|------|------|------|----------|
| 过早实现复杂特性导致核心结构反复推倒 | 高 | 进度失控 | 强制先完成 Phase 0/1 |
| `TickBroker` / `MixBroker` 同时重构导致范围膨胀 | 高 | 交付失焦 | 以共享核心为主线拆分阶段 |
| 组合与继承方案摇摆不定 | 中 | 设计反复 | 在 Phase 0/3 固化 ADR |
| Order/Commission 扩展破坏既有抽象 | 中 | 波及非目标模块 | 优先使用可选属性、`addinfo()` 与子类化扩展 |
| 延迟与队列逻辑导致时序 bug 难调试 | 中 | 验证成本高 | 增加 timeline 场景测试和 debug 日志 |
| 增强能力引入明显性能退化 | 中 | 无法投入使用 | 引入基线对比与可关闭开关 |
| 队列模型过度复杂而解释性不足 | 中 | 功能可用但难验证 | 先交付可解释的基础模型，再逐步增强 |
| 新测试体系不完整 | 中 | 质量无保证 | 用“能力矩阵”替代“旧文件是否通过” |

## 8. 建议执行顺序

1. 先做 Phase 0，把目标和边界真正定义清楚。
