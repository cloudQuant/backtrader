# 迭代6：`py-hftbacktest` 对标分析

## 1. 分析目标

本分析用于把 `py-hftbacktest` 的核心设计要点映射到当前 `backtrader` 迭代6的高频回测实现，明确哪些能力已经采纳，哪些能力延后处理，以及为什么这样裁剪。

## 2. 对标维度

| 维度 | `py-hftbacktest` | 迭代6实现 | 结论 |
|------|------------------|------------|------|
| 事件时间模型 | `exch_ts` / `local_ts` 双时间语义 | `LatencyEngine.apply_feed_latency()` 为事件补充 `local_time` | 已采纳基础语义 |
| 订单在途延迟 | request/exchange/response 三段延迟 | `ConstantLatencyModel` + `IntpLatencyModel` + 可见时间队列 | 已采纳基础能力 |
| 共享撮合核心 | 独立回测核心 | `backtrader/brokers/hft/matching_core.py` | 已采纳 |
| maker/taker | 独立角色语义 | `QueueExchangeModel` + `FillRole` + `TickBroker` 角色化手续费 | 已采纳 |
| 队列位置 | Partial fill / queue ahead | `ProbQueueModel` / `NoQueueModel` | 已采纳基础版本 |
| TIF | GTX / IOC / FOK | `QueueExchangeModel` + `TickBroker` 路径支持 | 已采纳 |
| 状态聚合 | `StateValues` | `StateTracker` + `state_values()` | 已采纳 |
| 录制器 | Recorder | `Recorder` | 已采纳基础版本 |
| 多资产索引 | 原生 `asset_no` | 当前为按 `symbol` 组织 | 部分采纳 |
| 极限性能 | Rust 核心 | 纯 Python 实现 | 明确不在本轮硬承诺 |

## 3. 关键采纳结论

### 3.1 已采纳

- 共享高频撮合核心独立为 `hft` 子包
- 基础延迟语义前置，不等到尾部阶段才落地
- `TickBroker` 不再内联承担全部高频撮合职责
- `MixBroker` 保持 tick-first + bar-fallback 语义
- 状态追踪、录制器、队列和交易所模型按可插拔方向实现

### 3.2 延后或降级采纳

- 未引入 Rust/C 扩展
- 未实现 `asset_no` 级别的原生多资产索引
- 未对齐 `hftbacktest` 全量统计模块
- 队列模型先交付可解释版本，不追求复杂概率市场微结构

## 4. 与当前实现最相关的借鉴点

### 4.1 时间语义

`hftbacktest` 的核心价值之一是区分“交易所发生时间”和“本地可见时间”。迭代6在 `LatencyEngine` 中保留了这一语义：事件仍有原始 `timestamp`，同时在需要时附加 `local_time` 作为策略可见时间。

### 4.2 撮合职责分层

`hftbacktest` 倾向于把撮合、状态和时间控制拆开。迭代6对应拆分为：

- `MatchingCore`
- `LatencyEngine`
- `ExchangeModel`
- `QueueModel`
- `StateTracker`
- `Recorder`

### 4.3 手续费与角色语义

`hftbacktest` 的 maker/taker 语义必须体现在成交成本上。迭代6通过 `CommInfoBase` 增加 `maker_commission` / `taker_commission` 参数，并由 `TickBroker` 在成交时把角色透传给手续费体系，保持与现有 broker / comminfo API 兼容。

## 5. 本轮边界

本轮的目标不是复制 `py-hftbacktest` 的全部实现，而是：

- 先补齐高频回测主链路语义
- 建立共享核心与扩展位
- 用测试与基准证明当前实现可用
- 把更重的性能和市场微结构增强留在后续迭代

## 6. 结论

迭代6对 `py-hftbacktest` 的借鉴已经覆盖“共享核心、延迟、队列、maker/taker、状态聚合、录制器、能力测试与基准验证”这些关键维度。对未采纳部分也做了明确裁剪，因此本轮交付是一个有边界、有验证、可继续演进的高频回测版本。
