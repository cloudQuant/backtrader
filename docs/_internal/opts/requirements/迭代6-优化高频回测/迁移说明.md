# 迭代6：高频回测迁移说明

## 1. 迁移目标

本次迁移将高频回测能力从以 `TickBroker` / `MixBroker` 内联撮合逻辑为主，迁移为以 `backtrader.brokers.hft` 共享核心为主的结构。

## 2. 新增模块

- `backtrader/brokers/hft/latency.py`
- `backtrader/brokers/hft/state.py`
- `backtrader/brokers/hft/matching_core.py`
- `backtrader/brokers/hft/queue.py`
- `backtrader/brokers/hft/exchange.py`
- `backtrader/brokers/hft/recorder.py`

## 3. Broker 侧主要变化

### 3.1 TickBroker

- 支持 `LatencyEngine` 控制订单可见时间
- 支持 `MatchingCore` 管理 pending 订单索引
- 支持 `StateTracker` 聚合视图查询
- 支持 `Recorder` 记录成交时间线
- 支持可选 `exchange_model`，提供 maker/taker、GTX/FOK/IOC 等语义
- 支持 `CommInfoBase` 的 `maker_commission` / `taker_commission` 角色化手续费
- `_execute()` 已补齐开平仓拆分、PnL、margin、`addcomminfo()`
- 新增 `modify()`，语义为 `cancel + re-submit`

### 3.2 MixBroker

- 保留 `TickBroker` 继承结构
- 职责聚焦为 `tick 优先 + bar fallback`
- bar fallback 路径复用增强版 `_execute()`

## 4. 兼容性说明

### 4.1 保持不变

- `Cerebro` 入口不变
- `TickEvent` / `OrderBookSnapshot` / `BarEvent` 核心事件接口不变
- `bbroker.py` 职责边界不变

### 4.2 行为增强

以下行为相比旧实现更加明确：

- 拒单原因会写入 `order.info.reject_reason`
- IOC 剩余撤单原因会写入 `order.info.cancel_reason`
- GTX / FOK / IOC 在启用 `QueueExchangeModel` 时生效
- maker/taker 手续费优先走 `CommInfoBase` 扩展，不在 broker 层内联硬编码费率
- `order_history` 不再只记录成交，也会记录关键拒单/撤单语义
- `state_values()` 提供统一聚合状态视图

## 5. 迁移建议

### 5.1 旧代码调用方

如果你只是：

- 调用 `buy()` / `sell()`
- 使用 `process_tick()` / `process_orderbook()` / `process_bar()`
- 读取 `getcash()` / `getposition()`

则通常不需要修改。

### 5.2 如需启用增强能力

可以显式注入：

- `latency_model`
- `exchange_model`
- `state_tracker`
- `recorder`
- 角色化手续费的 `CommissionInfo` / `CommInfoBase`

例如：

```python
broker = TickBroker(
    cash=100000.0,
    latency_model=ConstantLatencyModel(order_entry_latency_ms=5),
    exchange_model=QueueExchangeModel(),
)
```

也可以直接通过 broker 配置角色化费率：

```python
broker.setcommission(
    commission=0.001,
    maker_commission=-0.0005,
    taker_commission=0.002,
    name="BTC/USDT",
)
```

或者显式注入 `CommissionInfo` / `CommInfoBase` 子类：

```python
broker.addcommissioninfo(
    CommissionInfo(
        commission=0.001,
        maker_commission=-0.0005,
        taker_commission=0.002,
    ),
    name="BTC/USDT",
)
```

### 5.3 关于 modify

`modify()` 不是原地改单，而是：

- 取消旧单
- 复制关键字段
- 重新提交新单

因此：

- 原订单会进入 `Canceled`
- 新订单会带 `modified_from` 信息
