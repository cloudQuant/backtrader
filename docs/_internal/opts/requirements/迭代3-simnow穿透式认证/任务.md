# 迭代 3 - SimNow 穿透式认证任务说明

## 1. 背景

本任务的原始输入来自以下材料：

1. `docs/_internal/opts/requirements/迭代 1-穿透式认证/期货程序化交易系统功能测试过程记录报告.docx`
2. `docs/_internal/opts/requirements/迭代 1-穿透式认证/readme.md`
3. 当前 `backtrader` 项目中已经存在的 CTP / SimNow live 示例与日志能力

经核对 `docx`，本次需要覆盖 **33 个测试点**，其中：

1. **必做 28 个**
2. **选做 5 个**

本轮不是从零实现 CTP 接入，而是基于现有项目能力，把“监管测试点”沉淀为一套**可单独运行、可重复验证、可沉淀证据、失败可追踪**的 SimNow 7x24 穿透式认证场景集。

## 2. 当前项目现状

结合当前代码与已有示例，项目已经具备以下基础：

1. `examples/live_tests/test_simnow_ctp.py`
   - 已有 SimNow 连接、行情、账户、持仓、下单、撤单等 live 用例骨架
   - 内置多套 SimNow 环境配置，默认可指向 `new_7x24`

1. `examples/live_tests/test_simnow_trade_logger_certification.py`
   - 已有监管日志方向的 live 验收用例
   - 已覆盖 `system.log`、`monitor.log`、`error.log`、`order.log` 等关键证据文件

1. `backtrader/stores/btapistore.py`
   - CTP 路径已经支持 `app_id`、`auth_code`
   - 已存在 `store_auth_success`、`store_login_success`、`order_submit_request`、`order_cancel_request` 等运行时事件

1. `backtrader/observers/trade_logger.py`
   - 已存在 `system.log`、`monitor.log`、`error.log`
   - 已存在监测统计与阈值相关能力入口

1. `backtrader/brokers/btapibroker.py`
   - 已存在本地下单校验与本地拒单事件基础

因此，迭代 3 的重点不是“先做能力再写验证”，而是：

1. **整理 33 个场景**
2. **把现有能力转成一场景一入口的可执行示例**
3. **补齐缺口场景**
4. **建立统一结果记录与失败归档机制**

## 3. 目标

本轮目标如下：

1. 基于 SimNow 7x24 环境，产出一套覆盖 33 个测试点的穿透式认证场景集
2. 每个场景都能通过单独入口执行，做到“运行一个例子 = 验证一个场景”
3. 每个场景都输出明确结果：
   - `PASS`
   - `FAIL`
   - `BLOCKED`
1. 每个场景都沉淀可追溯证据：
   - 控制台输出
   - 结构化结果文件
   - 相关日志文件
   - 失败原因 / 阻塞原因
1. 对无法稳定在 SimNow 复现的场景，不允许伪造“成功”，必须落为阻塞项并记录后续修复建议

## 4. 默认运行环境与约束

### 4.1 默认环境

本轮默认使用 **SimNow 7x24 看穿式前置环境**，即当前代码中的 `new_7x24`：

1. `td_address = tcp://182.254.243.31:40001`
2. `md_address = tcp://182.254.243.31:40011`

同时保留通过环境变量切换其他 SimNow 环境的能力。

### 4.2 运行凭据

默认从环境变量读取账户信息：

1. `SIMNOW_USER_ID` / `simnow_user_id`
2. `SIMNOW_PASSWORD` / `simnow_password`

必要时补充：

1. `SIMNOW_ENV`
2. `SIMNOW_TICK_SYMBOL`
3. `SIMNOW_ORDER_SYMBOL`

### 4.3 运行约束

1. 所有 live CTP 场景必须**串行执行**
2. macOS 下 CTP native 扩展退出阶段存在不稳定风险，场景执行需延续当前示例中的**子进程隔离**策略
3. 不能依赖人工临时修改代码来触发场景
4. 不能通过伪造远端回报来冒充 SimNow 验证成功
5. 对市场状态、资金、持仓等远端错误场景，如果 SimNow 无法稳定复现，允许判定为 `BLOCKED`，但必须留下证据

## 5. 交付形式

本轮交付不是单一测试文件，而是一套“场景目录 + 公共运行骨架 + 结果汇总”的结构。

建议交付结构如下：

```text
examples/live_certification/simnow_penetration/
├── README.md
├── common/
│   ├── config.py
│   ├── runtime.py
│   ├── result.py
│   └── helpers.py
├── cases/
│   ├── C01_connect_and_login.py
│   ├── T01_open_order.py
│   ├── ...
│   └── L04_error_log_record.py
├── reports/
│   ├── latest/
│   └── history/
└── run_case.py

```bash
要求如下：

1. **一场景一入口**
   - 可以是一个独立文件
   - 也可以是独立目录
   - 但必须保证用户可以明确运行某一个场景

1. **公共能力复用**
   - SimNow 配置读取
   - Store / Broker / Data 初始化
   - 子进程执行
   - 结果落盘
   - 日志目录创建

1. **结果归档统一**
   - 每个场景执行后必须输出结构化结果文件
   - 所有场景执行后必须输出汇总文件

## 6. 33 个测试场景清单

### 6.1 连通性

1. `C01`：验证登录测试账号通过柜台认证并完成账号登录

### 6.2 基础交易功能

1. `T01`：验证能正常下达开仓指令
2. `T02`：验证能正常下达平仓指令
3. `T03`：验证能正常下达撤单指令

### 6.3 系统连接异常监测

1. `M01`：验证连接成功时能正常显示连接成功
2. `M02`：验证连接断开时能正常显示连接断开
3. `M03`：验证连接断开后能正常显示重连成功

### 6.4 报撤单笔数监测

1. `M04`：验证能正常统计报单笔数
2. `M05`：验证能正常统计撤单笔数

### 6.5 重复报单监测（选测）

1. `O01`：验证能统计重复开仓单报单笔数
2. `O02`：验证能统计重复平仓单报单笔数
3. `O03`：验证能统计重复撤单报单笔数

### 6.6 阈值设置及预警

1. `TH01`：验证提供报单笔数统计阈值设置功能
2. `TH02`：验证报单笔数达到或超过阈值时会预警
3. `TH03`：验证提供报撤单总数统计与阈值设置功能
4. `TH04`：验证报撤单总数达到或超过阈值时会预警
5. `TH05`：验证提供重复报单笔数统计与阈值设置功能（选测）
6. `TH06`：验证重复报单笔数达到或超过阈值时会预警（选测）

### 6.7 错误防范

1. `V01`：验证订单合约代码错误时系统能检查并拒绝报单
2. `V02`：验证订单价格最小变动价位错误时系统能检查并拒绝报单
3. `V03`：验证订单委托数量超过单笔最大委托数量时系统能检查并拒绝报单

### 6.8 错误提示

1. `E01`：验证系统能接收并展示柜台返回的资金不足错误码
2. `E02`：验证系统能接收并展示柜台返回的持仓不足错误码
3. `E03`：验证系统能接收并展示柜台返回的市场状态错误码

### 6.9 应急处理

1. `EM01`：验证系统可通过限制账号交易权限方式暂停交易
2. `EM02`：验证系统可通过暂停策略执行方式暂停交易
3. `EM03`：验证系统可通过强制账号退出方式暂停交易

### 6.10 批量撤单（选测）

1. `B01`：验证系统支持将多笔部分成交报单进行批量撤单
2. `B02`：验证系统支持将多笔已报单进行批量撤单

### 6.11 日志记录

1. `L01`：验证系统日志中会记录交易信息
2. `L02`：验证系统日志中会记录系统运行信息
3. `L03`：验证系统日志中会记录监测信息
4. `L04`：验证系统日志中会记录错误提示信息

## 7. 每个场景的统一交付要求

每个场景至少要包含以下内容：

1. **场景元信息**
   - 场景编号
   - 场景名称
   - 所属分类
   - 是否选测

1. **执行前置条件**
   - 所需环境变量
   - 所需合约
   - 是否需要账户有持仓 / 可用资金
   - 是否要求特定交易时段

1. **执行步骤**
   - 使用哪个 Store / Broker / Feed
   - 发起什么动作
   - 预期收到什么事件 / 日志 / 回报

1. **通过标准**
   - 必须命中的事件
   - 必须生成的日志
   - 必须满足的统计值

1. **失败与阻塞判定**
   - 本地逻辑失败记为 `FAIL`
   - 远端条件不足 / 市场关闭 / SimNow 不稳定导致无法完成记为 `BLOCKED`

1. **结果输出**
   - `result.json`
   - `stdout.log`
   - 如涉及日志验证，还需保留 `system.log`、`monitor.log`、`error.log`、`order.log`

## 8. 统一结果模型

每个场景执行后必须输出统一格式结果，至少包含：

1. `case_id`
2. `case_name`
3. `status`
4. `started_at`
5. `finished_at`
6. `duration_seconds`
7. `simnow_env`
8. `evidence`
9. `failure_reason`
10. `next_action`

其中 `status` 只允许以下取值：

1. `PASS`
2. `FAIL`
3. `BLOCKED`

建议约定退出码：

1. `0` = `PASS`
2. `1` = `FAIL`
3. `2` = `BLOCKED`

## 9. 场景实现原则

### 9.1 优先复用现有能力

优先在以下基础上扩展，不要重复造轮子：

1. `examples/live_tests/test_simnow_ctp.py`
2. `examples/live_tests/test_simnow_trade_logger_certification.py`
3. `backtrader/stores/btapistore.py`
4. `backtrader/brokers/btapibroker.py`
5. `backtrader/observers/trade_logger.py`

### 9.2 先主链路，后难场景

建议实现顺序：

1. 连通性 + 基础交易 + 日志记录
2. 监测统计 + 阈值预警 + 本地错误防范
3. 远端错误提示 + 应急处理 + 批量撤单 + 重连

### 9.3 不允许伪成功

以下做法不可接受：

1. 仅打印“成功”但无柜台或日志证据
2. 用 mock 结果替代 SimNow 回报却标记为 live 验证通过
3. 发现问题后跳过记录，直接人工口头说明

## 10. 当前已知高风险场景

以下场景预计存在较高不确定性，需要在实现时优先设计阻塞记录机制：

1. `M02` / `M03`：断开与重连成功
2. `E01` / `E02` / `E03`：远端错误码稳定复现
3. `EM03`：强制账号退出后的验证链路
4. `B01`：多笔部分成交后批量撤单

这些场景如果不能在 SimNow 稳定复现，不应强行伪造通过，而应输出：

1. 已执行步骤
2. 已观察到的真实回报
3. 缺失的条件
4. 建议后续修复或补充方式

## 11. 最终验收要求

本任务完成时，至少要满足以下验收标准：

1. 33 个场景均有独立编号和对应入口
2. 每个场景都能单独运行
3. 每个场景运行后都有明确结果与证据目录
4. 至少形成一份总表，汇总 33 个场景的当前状态
5. 对失败或阻塞场景，必须形成可复现的问题记录
6. 不允许只交付代码，不交付执行说明与结果说明

## 12. 本轮明确不做的事情

除非实现过程中被证明是主链路阻塞项，否则本轮不额外展开以下内容：

1. 非 SimNow / 非 CTP provider 的认证场景
2. Web 管理页面
3. MySQL 日志平台化改造
4. 与本次 33 个测试点无关的 UI 或回测功能重构
