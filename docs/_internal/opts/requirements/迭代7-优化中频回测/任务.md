 ## 1. 实施总策略

### 1.1 本轮执行定位

本轮迭代不按“单点修 bug”方式推进，而按以下主线实施：

1. **先定义中频回测，再进入实现讨论**
   - 先把中频回测的边界、时间语义和执行语义说清楚。
2. **先统一数据视图，再推进策略样例**
   - 先定义 `tick/orderbook/k线` 如何共同形成可见状态，再谈策略如何下单。
3. **先收敛 `MixBroker` 职责，再评估实现路径**
   - 避免继续让 `MixBroker` 成为“什么都往里塞”的混合层。
4. **先打通单品种闭环，再扩展到多品种联动**
   - 单品种是主链路，多品种是本轮的重要增强。
5. **以可验证交付物为阶段门禁**
   - 每个阶段结束都要有文档、样例或测试矩阵作为成果。

### 1.2 强制约束

```text
[强制] 不以多继承方式同时承接 TickBroker 和 BackBroker 的完整语义
[强制] 不修改 bbroker.py 的既有职责边界
[强制] 中频回测必须坚持“高频优先执行、bar默认只参与信号”的基本原则
[强制] 所有数据对齐与可见性规则都必须文档化
[强制] 多品种场景的结论必须落实到统一时间调度语义，而不是仅停留在概念层
```

### 1.3 阶段门禁原则

| 门禁项 | 含义 |
|------|------|
| 需求门禁 | 没有定义清楚中频回测边界前，不进入结构设计 |
| 时间门禁 | 没有统一时间线和bar可见性规则前，不进入策略样例固化 |
| 架构门禁 | 没有明确 `MixBroker` 职责边界前，不进入执行链路定稿 |
| 多品种门禁 | 没有单品种闭环前，不扩展套利和联动样例 |
| 验证门禁 | 没有测试矩阵和验收标准前，不视为阶段完成 |

## Phase 3：执行链路与 MixBroker 职责收敛（预计 4-5 天）

### 目标

让中频策略在执行层保持高频优先语义，同时明确 `MixBroker` 的定位。

### 关键决策点

| 决策项 | 备选方案 | 推荐方向 |
|------|----------|----------|
| `MixBroker` 结构 | 继续深度继承 `TickBroker` / 协调层化 | 优先职责收敛为协调层 |
| bar 的作用 | 既做信号又做默认执行 / 默认只做信号 | 默认只做信号 |
| 无高频数据时的订单处理 | 自动 bar 降级 / 保持未成交 | 保持未成交 |
| 订单执行入口 | 直接混合撮合 / 高频优先 + 未成交边界 | 高频优先 |

### 任务清单

| 编号 | 任务 | 输出 | 验收方式 |
|------|------|------|----------|
| 3.1 | 明确 `MixBroker` 职责边界 | 结构决策说明 | 不与 `BackBroker` 职责混淆 |
| 3.2 | 定义执行链路主流程 | 执行流程图或步骤说明 | 高低频职责清晰 |
| 3.3 | 定义无高频数据时的订单边界 | 未成交规则说明 | 语义明确 |
| 3.4 | 梳理订单回写语义 | 账户更新说明 | 现金/持仓/盈亏口径一致 |
| 3.5 | 完成单品种闭环口径 | 单品种验收标准 | 能支撑后续测试矩阵 |

### 阶段完成标准

- `MixBroker` 的结构方案有明确结论
- 中频执行链路清晰
- 单品种闭环可作为后续实现依据

### 9.2 Phase 1 文件级任务分解

**目标**：实现统一时间线，支持 tick/orderbook/bar 混合数据的归并排序和同时刻事件排序。

| 任务编号 | 文件 | 具体任务 | 预估工时 | 依赖 |
|----------|------|----------|----------|------|
| T1-01 | 新文件：`backtrader/feeds/mixed_channel.py` | 实现 `build_mixed_channel()` 函数，负责读取 tick/orderbook 和 bar 两路数据，按 timestamp 归并排序生成统一 ChannelEvent 流 | 1-2 天 | 无 |
| T1-02 | 同上 | 实现同时刻事件排序：同一时间戳下按 `tick(1) > orderbook(2) > bar(3)` 优先级排列 | 0.5 天 | T1-01 |
| T1-03 | `backtrader/events.py` | 复用并确认 `BarEvent` 数据结构契约（`symbol`、`timestamp`、`open`、`high`、`low`、`close`、`volume`），不再额外拆分 `bar_event.py` | 0.5 天 | 无 |
| T1-04 | `tests/test_mixed_channel.py` | 统一时间线的单元测试：时间戳非递减、同时刻排序正确、多品种归并正确 | 1-2 天 | T1-01 |

- `build_mixed_channel()` 输出的事件流时间戳严格非递减
- 同一时刻 tick 事件排在 bar 事件之前
- 多品种事件按全局时间戳统一排列

### 9.7 风险与应对（补充）

| 风险 | 等级 | 影响 | 应对策略 |
|------|------|------|----------|
| channel 归并排序对大文件性能差 | 中 | 回测启动慢 | 使用 `heapq.merge` 实现惰性归并，不全量加载到内存 |
| `MidFreqContext` 接口频繁变更 | 中 | 样例和测试需要反复修改 | 在 Phase 2 早期冻结核心接口签名 |
| 多品种事件时间精度不足 | 中 | 不同品种事件错序 | 使用 nanosecond 精度时间戳 + 品种名作为次级排序键 |
