### 背景
现在低频回测(`bbroker.py`)和高频回测(`tickbroker.py`)已经有基础能力，但中频回测场景还没有形成完整、可复用、可验证的方案。本轮希望围绕“多源数据联合计算信号、以高频数据驱动撮合、统一账户回写”的链路，完成中频回测的需求梳理与迭代规划。

### 需求

中频回测不是简单地把低频回测和高频回测拼接在一起，而是要在同一套回测链路中同时支持 `tick`、`orderbook`、`k线` 三类数据的协同使用：信号层可以联合使用高频与低频特征，执行层以 `tick/orderbook` 为主进行撮合，账户层要基于真实成交结果回写持仓、现金、手续费和盈亏。

### 希望实现的例子

1. 使用过去30个orderbook的数据，计算前10档的买单金额与卖单金额的比值，如果值大于1.5，则做多，如果值小于0.7，则做空。
2. 做多和做空还需要满足1分钟K线的条件，需要1分钟k线的收盘价大于过去20周期的1分钟K线的均线，才允许做多；反之，才允许做空。

### 需求梳理与迭代7实施计划

#### 1. 业务定义

1. 中频回测的本质，是在一个策略里同时使用高频市场微观结构信息和较低频的K线状态。
2. 信号层需要同时支持：
   - 基于 `orderbook`、`tick` 的短窗口统计特征计算
   - 基于 `1分钟K线`、`5分钟K线` 等bar数据的趋势、过滤、确认逻辑
3. 执行层不能退化成只按bar成交，而应继续以 `tick/orderbook` 为主进行撮合，保证成交价格、成交时点和部分成交语义更真实。
4. 账户层需要沿用 broker 的真实成交回写能力，正确更新现金、持仓、手续费、盈亏、订单状态。
5. 场景层不仅要支持单品种择时，还要支持多品种联动、对冲、套利等中频策略。

#### 2. 对当前问题的分析结论

1. 当前 `MixBroker` 已经存在，但旧实现更接近“TickBroker 的执行扩展层”，还不足以完整表达中频回测。
2. 因此，迭代7不应只理解为“给 `MixBroker` 补几个功能”，而应明确以下三件事：
   - 如何统一管理 `tick/orderbook/k线` 的时间对齐和可见性
   - 如何定义“信号计算”和“订单撮合”之间的职责边界
   - 如何让单品种与多品种场景都能复用同一套中频回测框架
3. 本轮的核心不是简单追求更多订单类型，而是先把“混合数据驱动的策略上下文”和“稳定可解释的执行语义”建立起来。

#### 3. 架构判断

1. `mixbroker` 不建议同时继承 `tickbroker` 和 `bbroker`。
   - 原因是两者分别代表不同的撮合语义和时间推进方式，多继承会让职责边界更混乱。
   - 更合理的方向是：保留 `MixBroker` 作为混合模式协调层，底层通过共享执行核心或组合方式复用高频撮合能力。
2. 中频回测的数据对齐建议采用“统一事件时间线 + 多源状态快照”的模式。
   - `tick/orderbook` 负责推进执行时钟和盘口状态
   - `k线` 负责推进指标状态和过滤条件
   - 策略在任意时点看到的是“该时点之前已经完成的bar状态 + 当前可见的高频状态”
3. `k线` 默认用于信号计算和过滤，不直接替代高频撮合。
   - 本轮不引入任何 bar 降级撮合语义，订单执行只走 `tick/orderbook` 路径。
4. 多品种场景建议采用“全局时间调度 + 品种级状态 + 组合级账户”的设计。
   - 每个品种维护自己的 `tick/orderbook/bar` 状态
   - 组合账户统一管理资金、持仓、盈亏和风险
   - 跨品种策略通过共享上下文访问多个品种的状态快照

#### 4. 迭代7目标

1. 明确中频回测的定义、边界、数据流和执行语义。
2. 形成统一的数据对齐方案，解决 `tick/orderbook/k线` 在混合模式下的时钟推进问题。
3. 形成 `MixBroker` 的职责收敛方案，明确它到底是：
   - 继续作为 `TickBroker` 的扩展层
   - 还是调整为基于共享执行核心的协调层
4. 建立单品种中频策略样例，验证“高频信号 + K线过滤 + 高频撮合”的完整闭环。
5. 建立多品种场景样例，验证中频回测在套利/联动场景下的可行性。

#### 5. 本轮纳入范围

1. `tick`、`orderbook`、`k线` 的混合时间对齐规则
2. 中频策略上下文的数据访问方式

#### 6. 本轮非目标范围

1. 不重写 `bbroker.py` 的既有bar回测职责.
2. 不把本轮目标定义为追平极限高频性能.

### 技术经理评审补充（基于代码审查）

> 以下内容基于对现有代码的深入审查，对上述3个问题给出结论性建议，并补充需求分析师文档中未覆盖的技术要点.

#### 问题1结论：MixBroker 的继承策略

**现状分析**：

当前 `MixBroker`（`backtrader/brokers/mixbroker.py`）已经继承 `TickBroker`，旧版结构清晰但职责过于靠近执行层扩展：

- 复用 `TickBroker` 全部高频撮合能力（`process_tick`、`process_orderbook`）
- 旧版 `process_bar` 曾承担 bar 级撮合语义
- 旧版通过 `_order_submit_ts`、`_try_match_bar` 等机制引入了降级执行路径

**结论：继续继承 `TickBroker`，不做多继承，不退回 `BrokerBase`。**

理由如下：

1. **继承 `TickBroker` 是正确的基线**。`TickBroker` 内部已经封装了 `MatchingCore`、`LatencyEngine`、`ExchangeModel`、`StateTracker`、`Recorder` 等核心执行组件，这些正是中频回测所需的"共享执行核心"。如果退回 `BrokerBase`，就需要重新组装这些组件，属于不必要的重复工作.
2. **不做 `TickBroker + BackBroker` 的多继承**。两者都直接继承 `BrokerBase`，各自维护独立的 `_cash`、`_positions`、`_pending_orders` 状态，多继承会导致 MRO 冲突和状态管理混乱。`BackBroker` 的 bar 撮合逻辑（1822 行）与 `TickBroker` 的 tick 撮合逻辑（1181 行）在时间推进、订单生命周期、成交回写方面语义差异过大.
3. **MixBroker 的扩展方向是职责收敛，而不是继承变更**。本轮的重点是：
   - 在 `MixBroker` 中新增**统一时间线管理**能力
   - 在 `MixBroker` 中新增**统一策略上下文生成**能力
   - 将 `process_bar` 固化为仅更新低频状态，不做任何 bar 撮合
   - 保持高频撮合路径完全复用 `TickBroker` 的已有能力

**继承关系图**：

```text
BrokerBase (broker.py)
  ├── BackBroker (bbroker.py)        ← 低频回测，不改动
  └── TickBroker (tickbroker.py)     ← 高频回测
        └── MixBroker (mixbroker.py) ← 中频协调层（本轮重点）
```

#### 问题2结论：tick / orderbook / K线 的数据对齐方案

**现状分析**：

当前 Cerebro 已有 channel 事件循环（`cerebro.py:1006-1033`），按事件到达顺序分发 `tick`、`orderbook`、`bar` 事件给 broker 和策略：

```python
for event in channel:
    ch = event.channel_type    # "tick" / "orderbook" / "bar"
    evdata = event.data
    if ch == "tick":    broker.process_tick(evdata)
    elif ch == "orderbook": broker.process_orderbook(evdata)
    elif ch == "bar":   broker.process_bar(evdata)
    # ... 然后分发给策略
```

策略侧已有对应回调：`notify_tick()`、`notify_orderbook()`、`notify_bar()`。

**数据对齐方案**：

1. **数据源层面**：中频回测需要至少两个数据 channel 同时输入：
   - 高频 channel：按时间排序的 `tick` + `orderbook` 事件流
   - 低频 channel：按时间排序的已完成 `bar` 事件流（外部输入，非实时聚合）
   - 两路数据在进入 Cerebro 前需要按照 **事件时间戳** 进行归并排序（merge sort），形成统一时间线

2. **bar 数据来源策略**（本轮的关键决策）：
   - **推荐方案：bar 由外部独立输入**。即用户在回测时同时提供 tick/orderbook 数据文件和 1 分钟 K 线数据文件，由 channel 层负责按时间戳归并。
   - **不推荐方案：从 tick 实时聚合生成 bar**。原因是实时聚合增加了回测复杂度和不确定性，且在中频场景下用户通常已有 K 线数据。如果确实需要从 tick 聚合 bar，可在后续迭代中作为 P2 特性支持.

3. **bar 可见性规则**：
   - bar 事件的时间戳 = bar 的结束时间（如 10:01:00 代表 10:00-10:01 的 1 分钟 K 线）
   - 只有当 channel 推进到 bar 结束时间时，该 bar 才对策略可见
   - bar 事件到达时，`MixBroker` 更新内部低频状态（已完成 bar 列表、均线等指标缓存），但**不默认**触发 bar 级撮合

4. **同时刻事件处理顺序**（当 tick/orderbook 和 bar 的时间戳相同时）：
   ```text
   ① 先处理高频事件（tick / orderbook）→ 更新盘口状态
   ② 再处理 bar 事件 → 更新已完成 bar 状态
   ③ 生成统一策略上下文快照
   ④ 触发策略决策
   ⑤ 策略产生的订单进入执行链路
   ```
   该顺序需要在 channel 的归并排序层或 `MixBroker` 内部保证.

#### 问题3结论：多品种回测方案

**现状分析**：

当前 Cerebro 的 channel 事件循环本身就支持多品种——事件通过 `event.data.symbol` 区分品种。`TickBroker` 内部已经用 `_orders_by_symbol`、`_last_tick[data_name]`、`_last_orderbook[data_name]`、`_positions[name]` 做了品种级状态隔离.

**多品种方案**：

1. **数据层**：多品种的 tick/orderbook/bar 数据在进入 channel 前，按照**全局时间戳**统一归并排序。所有品种的事件进入同一个事件循环.
2. **状态层**：`MixBroker` 为每个品种维护独立的：
   - 高频状态：`_last_tick[symbol]`、`_last_orderbook[symbol]`、`_ob_window[symbol]`（orderbook 窗口缓存）
   - 低频状态：`_completed_bars[symbol]`（已完成 bar 列表）、`_bar_indicators[symbol]`（均线等指标）
3. **策略层**：策略在任一事件到达时，可以读取**所有品种**在当前全局时点已可见的最新状态快照。跨品种决策示例：
   ```python
   def notify_orderbook(self, ob):
       # 读取品种A的orderbook比值
       ratio_a = self.context.get_ob_ratio("FutureA")
       # 读取品种B的orderbook比值
       ratio_b = self.context.get_ob_ratio("FutureB")
       # 读取品种A和B的价差
       spread = self.context.get_last_price("FutureA") - self.context.get_last_price("FutureB")
       # 套利逻辑
       if spread > threshold:
           self.sell(data=self.data_a, size=1)
           self.buy(data=self.data_b, size=1)
   ```
4. **账户层**：所有品种共享同一个组合账户（`_cash`、`_positions`、`_value`），这是 `TickBroker` 已有的行为，`MixBroker` 直接继承.
5. **风险点**：多品种事件交错密度高时，需要注意内存占用（每个品种都有 orderbook 窗口缓存）。建议增加 `max_ob_window` 参数控制窗口大小，默认 100.

#### 补充要点：需求分析师文档未覆盖的技术风险

1. **Cerebro 集成改造风险**
   - 当前 channel 事件循环在 `cerebro.py:1006-1033`，是一个简单的 for 循环，不保证同时刻事件的处理顺序.
   - 本轮需要在 channel 层或 `MixBroker` 内部增加**同时刻事件排序**逻辑，这是一个跨模块改动.
   - 建议将排序逻辑放在 `MixBroker` 内部，通过 event buffer 机制实现，避免改动 Cerebro 核心循环.

2. **策略上下文与现有 Strategy 类的集成**
   - 当前 Strategy 类已经有 `notify_tick()`、`notify_orderbook()`、`notify_bar()` 回调.
   - "统一策略上下文"不应替换这些回调，而应作为一个**附加的状态查询接口**，挂载到 Strategy 实例上（如 `self.context`）。
   - 上下文对象由 `MixBroker` 维护，通过 Cerebro 注入到 Strategy.

3. **OrderBook 窗口统计的内存开销**
   - "过去 30 个 orderbook 样本"意味着每个品种需要缓存 30 个 `OrderBookSnapshot`.
   - 每个 snapshot 的 bids/asks 通常 20 档，每档 2 个 float，单个 snapshot 约 640 字节.
   - 30 个样本 × 2 品种 ≈ 38.4 KB，内存开销可接受.
   - 但如果窗口扩大到 1000 或品种数量增加到 50，需要警惕内存增长，建议提供 `max_ob_window` 参数.

4. **工时评估调整建议**
   - 原计划总工时 16-22 天偏乐观。基于代码量和跨模块改动复杂度，建议调整为 **18-25 天**.
   - Phase 1（统一时间线）是技术难度最高的阶段，建议预留 **4-6 天**.
   - Phase 3（MixBroker 职责收敛）需要重构现有 `process_bar` 逻辑并新增上下文管理，建议 **4-6 天**.

5. **MixBroker 不需要向后兼容**
   - `MixBroker` 本身是迭代中新引入的实验性组件，尚无外部稳定用户依赖其行为.
   - 本轮直接将 `MixBroker` 重构为中频协调层，移除原有 bar 降级执行 / `tick_timeout` 机制，`process_bar` 默认仅更新低频状态，不做 bar 级撮合.
   - 不引入 `mode` 参数，避免增加不必要的分支复杂度。需要纯高频回测的用户直接使用 `TickBroker`，需要纯低频回测的用户使用 `BackBroker`.
