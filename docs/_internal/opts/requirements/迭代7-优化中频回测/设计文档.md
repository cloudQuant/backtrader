## 1. 设计概览

### 1.1 设计原则

1. **时间语义先于实现技巧**
   - 中频回测最大的风险不是功能点缺失，而是时间对齐失真。
2. **信号层与执行层解耦**
   - bar 主要服务于信号过滤，高频数据主要服务于执行和盘口状态。
3. **MixBroker 职责收敛**
   - `MixBroker` 是中频混合模式协调层，不应再成为无限扩张的聚合实现。
4. **可见性规则显式化**
   - 策略能看到什么数据，必须先于实现被清晰定义。
5. **单品种闭环优先，多品种扩展其次**
   - 先跑通主链路，再叠加套利和联动。
6. **复用已有能力，避免重复造轮子**
   - 尽量复用 `TickBroker` 的高频执行能力与 `BackBroker` 的bar语义边界，而不是平行再造新的大而全引擎。

### 1.2 架构总览

```text
┌───────────────────────────────────────────────────────────────┐
│                         Cerebro / Channel                      │
│             负责驱动数据源事件，按时间顺序分发给 Broker          │
└──────────────┬────────────────────────────────────────────────┘
               │
               ▼
┌───────────────────────────────────────────────────────────────┐
│                    中频回测协调层：MixBroker                    │
│                                                               │
│  1. 接收 tick / orderbook / bar 事件                           │
│  2. 维护统一时间线下的可见状态                                 │
│  3. 向策略暴露统一上下文                                       │
│  4. 接收策略订单并路由到高频优先执行链路                        │
│  5. 明确无高频数据时的未成交边界                               │
└──────────────┬────────────────────────────────────────────────┘
               │
      ┌────────┴─────────┐
      ▼                  ▼
┌───────────────┐   ┌────────────────────┐
│ 数据状态层     │   │ 执行协调层           │
│               │   │                    │
│ - 高频状态     │   │ - 订单接收          │
│ - 低频状态     │   │ - 高频优先撮合       │
│ - 已完成bar    │   │ - 未成交边界管理     │
│ - 多品种快照   │   │ - 账户回写          │
└───────────────┘   └────────────────────┘
```

### 1.3 重分析后的设计结论

1. **中频回测不是第三套完全独立的撮合模型**
   - 它是“多源数据状态 + 高频优先执行 + 统一账户回写”的协调机制。
2. **bar 的价值主要在信号层，而不是默认执行层**
   - 如果 bar 默认替代执行，则中频回测会退化为低频回测。
3. **`MixBroker` 需要从“补丁式混合实现”转成“中频协调层”**
   - 重点不在增加更多 if/else，而在明确职责边界。
4. **多品种问题本质上是全局时间调度问题**
   - 如果没有统一时间顺序和可见性定义，多品种联动策略无法稳定落地。

## 2. 核心设计

### 2.1 统一时间线设计

#### 2.1.1 基本原则

统一时间线需要满足以下约束：

1. 任意时点的策略决策只依赖于该时点已经可见的数据。
2. 高频状态和低频状态都必须映射到同一条逻辑时间线上。
3. 已完成bar只能在bar结束后对策略可见。
4. 同一时间戳下的事件处理顺序必须固定。

#### 2.1.2 推荐时间语义

在设计层，建议将中频回测的时间语义定义为：

1. `tick` / `orderbook` 事件使用事件时间直接推进高频状态。
2. `bar` 事件代表一个区间结束后的聚合结果，只在区间完成时进入可见状态。
3. 策略每次决策时读取的是：
   - 当前时点之前最新的高频状态
   - 当前时点之前已经完成的bar状态
4. 不允许把“正在形成中的bar”直接作为确认信号输入，除非未来有单独的增量bar语义设计。

#### 2.1.3 同时刻事件优先级建议

为了避免未来函数和行为不稳定，建议同一时间戳下采用以下顺序：

1. 更新该时点已经到达的高频状态（`orderbook`、`tick`）
2. 将已完成的bar结果写入低频状态
3. 生成对策略可见的统一快照
4. 触发策略决策
5. 将策略产生的订单送入执行链路

该顺序的核心目的是：

- 保证策略看到的是“当前时点完整可见的数据”。
- 保证bar完成后的状态不会被提前读取。
- 保证策略信号和执行之间的先后顺序是稳定的。

### 2.2 数据状态与快照设计

#### 2.2.1 状态分层

建议将状态分为三层：

1. **高频状态**
   - 最新 `tick`
   - 最新 `orderbook`
   - 高频窗口统计结果
2. **低频状态**
   - 已完成 `1分钟K线`
   - 已完成 `5分钟K线`
   - 均线等衍生指标
3. **账户状态**
   - 当前现金
   - 当前持仓
   - 已实现盈亏
   - 手续费、成交信息

#### 2.2.2 策略可见快照

策略每次决策应基于一份统一快照，而不是直接自行从多个数据源临时拼接。统一快照建议包含：

- 当前品种的高频窗口统计
- 当前品种最新高频市场状态
- 当前品种最新已完成bar状态
- 组合账户视图
- 若是多品种策略，还应包含其他相关品种在该时点的已可见快照

#### 2.2.3 快照生成原则

1. 快照必须在统一时间线下生成。
2. 快照一旦生成，在本次决策链路内保持稳定。
3. 快照不应混入任何尚未可见的数据。
4. 快照既要适用于单品种，也要适用于多品种。

### 2.3 中频策略上下文设计

#### 2.3.1 上下文职责

中频策略上下文需要负责回答三个问题：

1. 当前时点我看到了什么？
2. 当前时点我能基于哪些状态做决策？
3. 当前时点哪些信息还不可见？

#### 2.3.2 上下文内容

建议中频策略上下文至少包含以下内容：

- 当前事件时间
- 当前品种高频统计结果
- 当前品种最新高频状态
- 当前品种已完成bar指标状态
- 当前持仓和账户状态
- 多品种策略所需的关联品种快照

#### 2.3.3 业务样例映射

对业务样例，统一上下文需要支持：

1. 读取过去 30 个 `orderbook` 样本构成的窗口统计。
2. 读取前 10 档买卖盘金额比值。
3. 读取最近已完成的 1 分钟bar及其20周期均线。
4. 在同一时点将以上条件组合成做多或做空触发逻辑。

### 2.4 执行链路设计

#### 2.4.1 基本原则

中频策略的执行层必须满足以下原则：

1. 信号可以来自多频数据，但撮合优先走高频路径。
2. `tick/orderbook` 是默认执行来源。
3. 本轮不提供任何 bar 降级执行路径。
4. 成交后的账户回写仍遵循既有 broker 的权威更新语义。

#### 2.4.2 执行主流程

推荐的执行主流程如下：

1. 策略在统一快照下生成下单意图。
2. `MixBroker` 接收订单。
3. 优先将订单送入高频执行链路。
4. 若高频数据可驱动成交，则按高频路径完成撮合与回写。
5. 若当前没有可用高频数据，则订单保持未成交并等待后续高频事件或由策略撤单。
6. 成交结果统一回写到账户状态，并对策略/记录器可见。

#### 2.4.3 无高频数据时的订单边界

当高频数据缺失时，本轮执行语义固定如下：

- 不引入任何 bar 降级执行开关或超时降级参数。
- `process_bar` 只更新低频状态，不参与任何订单撮合。
- 未成交订单保持原有生命周期，等待后续 `tick/orderbook` 事件驱动成交。
- 若策略认为订单应取消，由策略显式撤单，而不是由 broker 自动降级成交。

因此，以下场景都不允许以 bar 替代执行：

- 仅因为策略使用了bar过滤
- 高频数据暂时缺失
- 为了规避高频撮合语义的复杂性

### 2.5 MixBroker 职责设计

#### 2.5.1 建议职责

`MixBroker` 在本轮中的建议职责包括：

1. 接收多源市场事件。
2. 维护统一时间线下的可见状态。
3. 生成统一策略上下文。
4. 接收策略订单并协调执行链路。
5. 管理高频优先执行与未成交边界。
6. 统一暴露账户层结果。

#### 2.5.2 不建议承担的职责

`MixBroker` 不应承担：

1. 完整复制 `BackBroker` 的bar撮合语义。
2. 承接所有高频撮合细节实现。
3. 同时作为时间协调器、策略引擎和底层撮合核心的全能容器。

#### 2.5.3 结构建议

从设计上，建议 `MixBroker` 以协调层角色存在：

- 对上承接 `Cerebro` 的事件驱动。
- 对内协调高频状态、低频状态和执行链路。
- 对下尽量复用已有高频执行能力，而不是重新复制一套底层撮合实现。

### 2.6 多品种调度设计

#### 2.6.1 设计目标

多品种场景需要解决以下问题：

1. 不同品种事件交错到达时，如何保证全局顺序一致。
2. 跨品种信号在某一时点读取到的快照是否一致。
3. 多品种订单执行后如何统一回写组合账户。

#### 2.6.2 调度原则

建议采用“全局时间顺序 + 品种局部状态”的方式：

1. 全局上按时间顺序推进事件。
2. 每个品种维护自己的高频状态和低频状态。
3. 当策略需要跨品种决策时，读取的是当前全局时点各品种已经可见的最新状态。
4. 订单回写统一进入组合账户视图。

#### 2.6.3 套利场景说明

以两个期货合约套利为例：

- 合约A和合约B有各自独立的 `tick/orderbook/bar` 状态。
- 策略在某一全局时点读取 A、B 两个品种的状态快照。
- 策略同时对两个品种发出交易指令。
- 执行结果仍按各自品种高频路径进行撮合。
- 账户层统一汇总两个品种的持仓和盈亏。

### 2.7 账户与结果回写设计

#### 2.7.1 权威状态

账户层仍应保持 broker 的权威语义：

- 现金、持仓、已实现盈亏、手续费等状态需要统一回写。
- 单品种和多品种场景使用一致的账户口径。

#### 2.7.2 中频回测结果

中频回测最终至少需要对外提供：

- 订单状态
- 成交结果
- 当前持仓
- 当前账户价值
- 关键决策链路的状态快照

## 3. 关键流程设计

### 3.1 单品种中频决策流程

```text
市场事件到达
    ↓
更新高频状态
    ↓
若bar完成，则更新低频状态
    ↓
生成统一快照
    ↓
策略读取 orderbook 窗口统计 + 已完成bar指标
    ↓
生成做多/做空/不操作决策
    ↓
订单进入 MixBroker
    ↓
优先走高频执行链路
    ↓
成交并回写账户状态
```

### 3.2 多品种联动流程

```text
全局时间推进
    ↓
更新各品种可见状态
    ↓
生成多品种统一快照
    ↓
策略读取 A/B 两个品种状态
    ↓
产生联动或套利指令
    ↓
各品种按自身高频路径执行
    ↓
统一回写组合账户
```

### 3.3 无高频数据时的执行流程

```text
订单进入执行链路
    ↓
高频路径优先处理
    ↓
若当前缺少可用高频事件
    ↓
订单保持未成交，等待后续 tick / orderbook
    ↓
若后续高频事件到达，则继续撮合并回写账户状态
```

## 5. 设计决策摘要

| 决策项 | 结论 |
|------|------|
| 中频回测定义 | 多源数据状态 + 高频优先执行 + 统一账户回写 |
| bar 的默认职责 | 信号过滤与状态确认，不默认承担执行 |
| `MixBroker` 定位 | 中频混合模式协调层 |
| 多继承方案 | 不建议同时继承 `TickBroker` 和 `BackBroker` |
| 多品种方案 | 全局时间调度 + 品种局部状态 + 组合账户 |
| 无高频数据边界 | 不提供 bar 降级执行，订单保持未成交直到后续高频事件到达或策略撤单 |

## 6. 设计风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 时间线定义不够精确 | 策略结果不可信 | 以统一快照和bar完成语义约束实现 |
| 快照口径不统一 | 单品种和多品种逻辑割裂 | 用统一上下文抽象读路径 |
| `MixBroker` 职责继续扩张 | 架构失控 | 明确只承担协调层职责 |
| 无高频数据时的订单边界不清 | 执行语义混乱 | 文档化“保持未成交”的行为契约 |
| 多品种调度未统一 | 套利结果不可复现 | 强制采用全局时间顺序 |

## 8. 技术经理补充：具体接口定义与实现方案

> 以下内容基于对 `mixbroker.py`（217行）、`tickbroker.py`（1181行）、`bbroker.py`（1822行）、`cerebro.py`（2369行）的代码审查，补充需求分析师设计稿中缺少的具体接口定义、伪代码、Cerebro 集成方案。

### 8.1 MixBroker 新增接口定义

基于继续继承 `TickBroker` 的结论，`MixBroker` 需要新增以下接口和内部状态：

```python
class MixBroker(TickBroker):
    """中频混合模式协调层。

    继承 TickBroker 的全部高频撮合能力，新增：
    - 统一时间线管理（event buffer + 同时刻排序）
    - 低频状态管理（已完成 bar 缓存 + 指标计算）
    - OrderBook 窗口统计（滑动窗口缓存）
    - 统一策略上下文生成
    """

    # === 新增参数 ===
    max_ob_window = ParameterDescriptor(
        default=100,
        doc="每品种 OrderBook 窗口缓存上限"
    )
    max_bar_history = ParameterDescriptor(
        default=200,
        doc="每品种已完成 bar 缓存上限"
    )

    # === 新增内部状态 ===
    # _ob_window: dict[str, deque[OrderBookSnapshot]]
    #   品种级 orderbook 滑动窗口
    # _completed_bars: dict[str, deque[BarEvent]]
    #   品种级已完成 bar 缓存
    # _bar_indicators: dict[str, dict[str, float]]
    #   品种级 bar 衍生指标（如均线）
    # _context: MidFreqContext
    #   统一策略上下文对象
    # _event_buffer: list[ChannelEvent]
    #   同时刻事件排序缓冲区

    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self._ob_window = defaultdict(
            lambda: deque(maxlen=self.get_param("max_ob_window"))
        )
        self._completed_bars = defaultdict(
            lambda: deque(maxlen=self.get_param("max_bar_history"))
        )
        self._bar_indicators = defaultdict(dict)
        self._context = MidFreqContext(broker=self)
        self._event_buffer = []
        self._last_buffer_ts = 0.0

    # === 重写的核心方法 ===

    def process_tick(self, tick_event, data=None):
        """重写：先更新 ob_window 相关的高频状态，再调用父类撮合。"""
        super().process_tick(tick_event, data)
        # 高频状态已由父类更新（_last_tick[symbol]）

    def process_orderbook(self, ob_event, data=None):
        """重写：在父类撮合基础上，追加 orderbook 窗口缓存。"""
        super().process_orderbook(ob_event, data)
        symbol = ob_event.symbol
        self._ob_window[symbol].append(ob_event)

    def process_bar(self, bar_event, data=None):
        """重写：bar 仅更新低频状态，不参与撮合。

        MixBroker 中所有订单撮合均走 tick/orderbook 路径（继承自 TickBroker），
        bar 数据仅用于策略信号计算（如均线过滤）。
        """
        symbol = bar_event.symbol
        self._completed_bars[symbol].append(bar_event)
        self._update_bar_indicators(symbol)

    # === 新增方法 ===

    def _update_bar_indicators(self, symbol):
        """根据已完成 bar 列表更新衍生指标（如均线）。"""
        bars = self._completed_bars[symbol]
        if len(bars) >= 20:
            closes = [b.close for b in list(bars)[-20:]]
            self._bar_indicators[symbol]["sma_20"] = sum(closes) / 20
        # 可扩展更多指标

    def get_context(self) -> "MidFreqContext":
        """返回统一策略上下文（供 Cerebro 注入到 Strategy）。"""
        return self._context

    def get_ob_window(self, symbol, n=30):
        """获取指定品种最近 n 个 orderbook 快照。"""
        window = self._ob_window.get(symbol, deque())
        return list(window)[-n:] if len(window) >= n else list(window)

    def get_completed_bars(self, symbol, n=20):
        """获取指定品种最近 n 个已完成 bar。"""
        bars = self._completed_bars.get(symbol, deque())
        return list(bars)[-n:] if len(bars) >= n else list(bars)

    def get_bar_indicator(self, symbol, indicator_name):
        """获取指定品种的 bar 衍生指标值。"""
        return self._bar_indicators.get(symbol, {}).get(indicator_name)
```

### 8.2 MidFreqContext 统一策略上下文

策略上下文是一个**只读查询接口**，挂载到 Strategy 实例上（`self.context`），不替换现有的 `notify_tick` / `notify_orderbook` / `notify_bar` 回调。

```python
class MidFreqContext:
    """中频策略统一上下文。

    提供对 MixBroker 内部状态的只读查询视图，
    策略通过 self.context.xxx() 读取数据。
    """

    def __init__(self, broker: MixBroker):
        self._broker = broker

    # --- 高频状态查询 ---

    def get_last_tick(self, symbol: str):
        """获取指定品种最新 tick。"""
        return self._broker._last_tick.get(symbol)

    def get_last_orderbook(self, symbol: str):
        """获取指定品种最新 orderbook 快照。"""
        return self._broker._last_orderbook.get(symbol)

    def get_last_price(self, symbol: str) -> float:
        """获取指定品种最新成交价。"""
        tick = self._broker._last_tick.get(symbol)
        return tick.price if tick else None

    def get_ob_window(self, symbol: str, n: int = 30):
        """获取最近 n 个 orderbook 快照列表。"""
        return self._broker.get_ob_window(symbol, n)

    def get_ob_ratio(self, symbol: str, levels: int = 10, window: int = 30) -> float:
        """计算买卖盘金额比值（核心业务接口）。

        Args:
            symbol: 品种标识
            levels: 取前几档盘口（默认10档）
            window: 过去几个 orderbook 样本（默认30）

        Returns:
            买盘总金额 / 卖盘总金额 的比值; 无数据时返回 None
        """
        snapshots = self._broker.get_ob_window(symbol, window)
        if not snapshots:
            return None

        total_bid_amount = 0.0
        total_ask_amount = 0.0
        for ob in snapshots:
            for i, (price, qty) in enumerate(ob.bids or []):
                if i >= levels:
                    break
                total_bid_amount += float(price) * float(qty)
            for i, (price, qty) in enumerate(ob.asks or []):
                if i >= levels:
                    break
                total_ask_amount += float(price) * float(qty)

        if total_ask_amount < 1e-12:
            return None
        return total_bid_amount / total_ask_amount

    # --- 低频状态查询 ---

    def get_completed_bars(self, symbol: str, n: int = 20):
        """获取最近 n 个已完成 bar。"""
        return self._broker.get_completed_bars(symbol, n)

    def get_sma(self, symbol: str, period: int = 20) -> float:
        """获取指定品种的简单移动均线值。"""
        if period == 20:
            return self._broker.get_bar_indicator(symbol, "sma_20")
        # 通用计算
        bars = self._broker.get_completed_bars(symbol, period)
        if len(bars) < period:
            return None
        return sum(b.close for b in bars) / period

    def get_last_bar(self, symbol: str):
        """获取指定品种最新已完成 bar。"""
        bars = self._broker.get_completed_bars(symbol, 1)
        return bars[0] if bars else None

    # --- 账户状态查询 ---

    def get_cash(self) -> float:
        return self._broker.getcash()

    def get_position(self, symbol: str):
        return self._broker._positions.get(symbol)

    def get_portfolio_value(self) -> float:
        return self._broker.getvalue()

    # --- 多品种快照 ---

    def get_symbols(self) -> list:
        """返回所有已注册品种列表。"""
        symbols = set()
        symbols.update(self._broker._last_tick.keys())
        symbols.update(self._broker._last_orderbook.keys())
        symbols.update(self._broker._completed_bars.keys())
        return sorted(symbols)

    def snapshot(self, symbol: str) -> dict:
        """生成指定品种在当前时点的完整状态快照。"""
        return {
            "symbol": symbol,
            "last_tick": self.get_last_tick(symbol),
            "last_orderbook": self.get_last_orderbook(symbol),
            "last_bar": self.get_last_bar(symbol),
            "sma_20": self.get_sma(symbol, 20),
            "ob_ratio": self.get_ob_ratio(symbol),
            "position": self.get_position(symbol),
        }
```

### 8.3 Cerebro 集成方案

当前 Cerebro 的 channel 事件循环（`cerebro.py:1006-1033`）已经支持 `tick/orderbook/bar` 三种事件类型的分发。本轮需要做的改动**尽量局限于 MixBroker 内部**，对 Cerebro 的改动仅限于上下文注入。

#### 8.3.1 Cerebro 最小改动

在 `cerebro.py` 的 `run(channel=...)` 路径中，仅增加一段上下文注入逻辑：

```python
# cerebro.py 中 channel 事件循环之前，增加上下文注入
if hasattr(self._broker, "get_context"):
    context = self._broker.get_context()
    for strat in runstrats:
        strat.context = context
```

这样策略就可以通过 `self.context` 访问统一上下文，无需改动 Strategy 基类。

#### 8.3.2 同时刻事件排序方案

**问题**：当 tick 和 bar 的时间戳完全相同时（如 bar 结束时刻恰好有 tick 到达），需要保证处理顺序固定。

**方案 A（推荐）：channel 层排序**

在数据归并时，对相同时间戳的事件按类型排序：
```text
排序优先级：tick(1) > orderbook(2) > bar(3)
```

即同一时刻先处理高频事件，再处理 bar。这在 channel 的 merge-sort 层实现，对 Cerebro 和 MixBroker 透明。

**方案 B（备选）：MixBroker 内部 buffer**

如果无法修改 channel 层，则在 `MixBroker` 内部增加 event buffer：
```python
def _buffer_event(self, event):
    """缓存同时刻事件，等时间推进时统一处理。"""
    ts = event.data.timestamp
    if ts != self._last_buffer_ts and self._event_buffer:
        self._flush_buffer()
    self._event_buffer.append(event)
    self._last_buffer_ts = ts

def _flush_buffer(self):
    """按优先级排序后统一处理缓冲区内事件。"""
    priority = {"tick": 1, "orderbook": 2, "bar": 3}
    self._event_buffer.sort(key=lambda e: priority.get(e.channel_type, 9))
    for event in self._event_buffer:
        ch = event.channel_type
        if ch == "tick":
            self.process_tick(event.data)
        elif ch == "orderbook":
            self.process_orderbook(event.data)
        elif ch == "bar":
            self.process_bar(event.data)
    self._event_buffer.clear()
```

**推荐方案 A**，因为更简洁且对 broker 无侵入。

### 8.5 业务样例的完整伪代码

以初始需求中的业务样例为基准，给出策略的完整实现框架：

```python
class MidFreqDemoStrategy(bt.Strategy):
    """中频回测业务样例策略。

    信号逻辑：
    - 过去30个orderbook的前10档买卖盘金额比值 > 1.5 → 做多信号
    - 过去30个orderbook的前10档买卖盘金额比值 < 0.7 → 做空信号
    - 做多还需1分钟收盘价 > 20周期均线
    - 做空还需1分钟收盘价 < 20周期均线
    """

    def __init__(self):
        self._symbol = "BTCUSDT"  # 或从 data 获取

    def notify_orderbook(self, ob):
        """每个 orderbook 事件触发决策。"""
        if ob.symbol != self._symbol:
            return

        # 1. 读取高频因子
        ratio = self.context.get_ob_ratio(self._symbol, levels=10, window=30)
        if ratio is None:
            return  # 数据不足，跳过

        # 2. 读取低频过滤条件
        sma = self.context.get_sma(self._symbol, period=20)
        last_bar = self.context.get_last_bar(self._symbol)
        if sma is None or last_bar is None:
            return  # bar 数据不足，跳过

        last_close = last_bar.close
        position = self.context.get_position(self._symbol)
        pos_size = position.size if position else 0

        # 3. 信号判定
        if ratio > 1.5 and last_close > sma and pos_size <= 0:
            # 做多
            if pos_size < 0:
                self.close()  # 先平空
            self.buy(size=1)

        elif ratio < 0.7 and last_close < sma and pos_size >= 0:
            # 做空
            if pos_size > 0:
                self.close()  # 先平多
            self.sell(size=1)

    def notify_bar(self, bar):
        """bar 到达时不做交易决策，仅记录日志。
        bar 状态已由 MixBroker 自动更新到 context。
        """
        pass

    def notify_tick(self, tick):
        """tick 事件到达时不做额外操作。
        tick 状态已由 MixBroker 自动更新。
        """
        pass
```

### 8.7 关键技术风险与缓解

| 风险 | 具体表现 | 缓解措施 |
|------|----------|----------|
| `process_bar` 行为变更 | 旧版 `MixBroker` 的 bar 降级执行行为被移除 | `MixBroker` 是实验性组件，无需向后兼容；纯高频用 `TickBroker`，纯低频用 `BackBroker` |
| `_ob_window` 内存泄漏 | 长时间回测时 deque 不断增长 | 使用 `maxlen` 参数控制，默认 100 |
| `MidFreqContext` 暴露 broker 内部状态 | 策略可能通过 context 修改 broker 状态 | context 方法全部为只读查询，不暴露可变引用 |
| 同时刻事件顺序不稳定 | 不同运行结果不同 | channel 层保证归并排序稳定性（stable sort） |
| bar 指标计算性能 | 每个 bar 到达都重新计算全部指标 | 采用增量计算（滑动窗口），避免重复遍历全部历史 |
