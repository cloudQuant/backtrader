## 1. 交接结论

经过对项目现状、代码实现和迭代7文档的复核，本轮已形成可维护、可回归、可继续扩展的冻结口径：

1. 当前仓库已经具备迭代7中频回测的完整主链路。
   - `backtrader/cerebro.py` 已有 channel 事件循环，可分发 `tick`、`orderbook`、`bar` 事件。
   - `backtrader/brokers/tickbroker.py` 已具备高频撮合、账户回写、订单通知等核心能力。
   - `backtrader/brokers/mixbroker.py` 已是独立 broker 入口，适合作为中频协调层的承载点。
2. 旧版 `MixBroker` 的 bar 降级执行 / `tick_timeout` 语义已经被移除。
   - `process_bar` 仅更新低频状态。
   - `MixBroker` 不再提供 bar 降级执行路径。
   - `tests/unit/brokers/test_mixbroker_enhanced.py`、`tests/unit/brokers/test_mixbroker_more.py` 已迁移到新契约。
3. 迭代7新增能力已经落地。
   - 已提供 `MidFreqContext`、`build_mixed_channel()`、单品种样例、多品种套利样例、集成测试与性能基线测试。
   - 同时刻事件顺序已固定为 `tick -> orderbook -> bar`。
   - 策略统一通过 `self.context` 读取中频状态。

本交接说明的目标，是把迭代7从“实现已完成”进一步收敛为“后续维护和增强可直接沿用冻结口径”。

## 2. 当前项目判断

### 2.1 现状优势

- `Cerebro -> Broker -> Strategy` 主链路已经存在，不需要重做引擎入口。
- `TickBroker` 已经覆盖订单提交、订单通知、撮合与账户回写，具备复用价值。
- 项目测试体系、示例目录、需求文档体系已经成型，适合按增量方式推进。

### 2.2 现状不足

- 后续扩展若新增文档或测试，仍需持续遵守当前冻结口径，避免重新引入 bar 降级执行或旧兼容分支表述。
- 多品种与性能相关能力已经落地，但后续若继续扩展，需要坚持现有时间语义和只读上下文约束。
- 性能基线当前以吞吐和延迟为主，若后续强化内存约束，需要新增专门回归项。

### 2.3 对迭代7的产品判断

迭代7方向是正确的，但如果直接按“全量目标一次性开工”，研发会面临三个问题：

1. P0 和 P1 混在同一执行批次，容易让主链路失焦。
2. 旧测试契约未先废弃，容易在重构中出现“代码改对了、测试却按旧逻辑失败”。
3. 多品种、性能、样例、上下文、时间线如果并行推进，回归成本会明显放大。

因此，本轮交接重点不再是“怎么开始做”，而是“后续继续做时不能突破哪些冻结边界”。

## 3. 优化后的范围冻结

### 3.1 P0 — 本轮必须完成

P0 只保留“单品种中频闭环 + 正确行为契约”四个结果：

1. **统一时间线最小闭环**
   - 支持 `tick/orderbook/bar` 的统一事件流。
   - 同一时间戳下固定优先级：`tick -> orderbook -> bar`。
   - `bar.timestamp` 明确定义为 bar 结束时间。
2. **MixBroker 收敛为中频协调层**
   - 继续继承 `TickBroker`。
   - `process_bar` 仅更新低频状态，不做任何订单撮合。
   - 不引入 `mode` 参数，不保留旧兼容分支。
3. **MidFreqContext 上下文闭环**
   - 策略通过 `self.context` 读取高频窗口、最新高频状态、已完成 bar、均线、账户状态。
   - 上下文必须是只读查询视图。
4. **单品种业务样例与测试闭环**
   - 实现 `orderbook` 比值 + `1分钟 SMA` 过滤 + 高频执行的最小样例。
   - 配套规则测试、场景测试、端到端测试至少各 1 组。

### 3.2 P1 — 本轮建议完成，但不阻塞 P0 交付

P1 作为 P0 完成后的增强项：

1. 多品种统一调度。
2. 双品种联动或套利样例。
3. 性能基线扩展与更细粒度回归。
4. 更多快照接口与多品种查询能力。

### 3.3 暂不纳入本轮

以下内容统一延后，不得混入本轮主开发：

1. `BackBroker` 重构。
2. 从 tick 实时聚合 bar。
3. 任何 bar 降级执行设计。
4. 实盘 broker 改造。
5. 复杂可视化回放。
6. 为兼容旧 `MixBroker` 行为而增加额外分支。

## 4. 冻结后的关键产品决策

以下决策视为本轮冻结，不再反复讨论：

| 决策项 | 冻结结论 |
|------|------|
| `MixBroker` 继承关系 | 继续继承 `TickBroker` |
| `MixBroker` 产品定位 | 中频协调层，不是降级执行 broker |
| `bar` 默认职责 | 仅参与低频状态更新与信号过滤 |
| `process_bar` 行为 | 不撮合、不回写成交、不激活 bar 成交路径 |
| 旧兼容策略 | 不做向后兼容，不保留旧兼容模式 |
| `mode` 参数 | 不引入 |
| bar 数据来源 | 外部独立输入 |
| 同时刻事件顺序 | `tick -> orderbook -> bar` |
| 策略数据读取入口 | `self.context` |
| 多品种是否阻塞本轮 | 不阻塞 P0，上移为 P1 |

## 7. 研发容易踩坑的点

### 7.1 不要把 `process_bar` 做成“顺手再撮合一下”

这是本轮最容易回退到旧实现思路的地方。只要 `process_bar` 继续承担撮合职责，整个中频方案就会重新退化为“bar 参与执行”的混合 broker，与本轮目标冲突。

### 7.2 不要把多品种能力提前并入 P0 主链路

多品种是重要增强，但不是本轮首个验收门槛。研发应先拿到单品种正确性，再进入跨品种扩展。

### 7.3 不要在 `Cerebro` 做过重改造

`Cerebro` 现有 channel 循环已经可用。本轮应把改动尽量限制在上下文注入和必要衔接，不在主循环中堆叠中频业务逻辑。

### 7.4 不要让 `MidFreqContext` 暴露内部可变结构

上下文如果直接返回内部 `deque` 或可写字典，策略侧可能无意中污染 broker 内部状态，后续调试会变得困难。

### 7.5 不要让旧测试继续定义新行为

本轮属于行为契约切换。若保留旧降级执行测试为“必须通过”，研发一定会被迫回填旧逻辑。

## 8. Definition of Done

研发完成本轮后，至少需要同时满足以下条件：

1. `MixBroker` 不再以 bar 作为默认成交来源。
2. `process_bar` 仅更新低频状态。
3. `self.context` 可供策略稳定读取中频状态。
4. 单品种业务样例可运行并复现。
5. 同时刻事件顺序固定且有测试覆盖。
6. 无高频数据时订单保持未成交的行为有测试覆盖。
7. 文档、任务、测试口径一致。

---

## 9. 后续增强建议顺序

后续若继续增强，建议按以下顺序推进，不建议并行打散：

```text
A. 统一时间线与 mixed_channel
    ↓
B. MixBroker 重构 + MidFreqContext + Cerebro 注入
    ↓
C. 单品种样例 + 端到端测试
    ↓
D. 多品种 + 性能基线（P1）
```

若 A 或 B 所对应的冻结契约被破坏，应先修复回归再继续扩展。

---

## 10. 产品经理最终交接意见

迭代7已经完成主链路实现，后续继续开发时应严格遵守本说明中的冻结范围：

- 先交付 **正确的中频行为契约**，再扩展多品种和性能。
- 先坚持 **bar 不参与执行** 的契约，再补充新样例和新测试。
- 先打通 **单品种闭环**，再做跨品种联动。

按此方案执行，研发会更容易在 17-25 天窗口内交付一个可验证、可解释、可扩展的中频回测第一版，而不是做出一个继续膨胀的 `MixBroker`。
