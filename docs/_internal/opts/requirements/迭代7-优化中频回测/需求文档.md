> 版本：v1.0  
> 状态：初稿  
> 关联文件：初始需求.md、设计文档.md、任务.md、测试文档.md

---

## 1. 背景与动机

### 1.1 项目现状

backtrader 当前已经具备两类相对明确的回测能力：

| Broker | 文件 | 主要用途 | 当前问题 |
|------|------|------|------|
| BackBroker | `backtrader/brokers/bbroker.py` | Bar/KLine 级别回测 | 适合低频，不适合需要盘口或逐笔驱动的场景 |
| TickBroker | `backtrader/brokers/tickbroker.py` | Tick / OrderBook 高频撮合 | 更适合执行层，但不直接解决多周期信号协同问题 |
| MixBroker | `backtrader/brokers/mixbroker.py` | 中频协调层（高频执行 + 低频状态） | 需要冻结并保持“bar 不参与撮合”的行为契约 |

当前业务提出的“中频回测”需求，本质上不是新增第三种完全独立的撮合引擎，而是希望在同一套回测链路中，把以下能力组合起来：

1. 使用 `tick`、`orderbook` 做短窗口特征或微观结构信号。
2. 使用 `1分钟K线`、`5分钟K线` 等bar数据做趋势过滤、方向确认或持仓管理。
3. 订单仍尽量通过高频数据进行撮合，而不是退化成仅按bar撮合。
4. 多品种场景下能够共享一个组合账户，支持联动、对冲和套利类逻辑。

### 1.2 中频回测的业务定义

本轮定义的中频回测，指的是：

- 策略在信号层同时消费高频和低频数据。
- 策略在执行层优先使用 `tick/orderbook` 做成交判定。
- 策略在账户层基于真实成交结果更新现金、持仓、手续费和盈亏。
- 策略在调度层能够处理多个品种、多个时间粒度数据共同推进的时间线。

### 1.3 典型业务样例

业务给出的最小样例包括两段逻辑：

1. 高频因子：使用过去 30 个 `orderbook` 样本，统计前 10 档买卖盘金额比值。
2. 低频过滤：使用 1 分钟 K 线收盘价与过去 20 周期均线做方向过滤。

形成的交易逻辑如下：

- 当买卖盘金额比值大于 `1.5`，且 `1分钟收盘价 > 20周期均线` 时，允许做多。
- 当买卖盘金额比值小于 `0.7`，且 `1分钟收盘价 < 20周期均线` 时，允许做空。
- 下单后的成交，仍以 `tick/orderbook` 驱动，不以bar价格直接替代。

### 1.4 当前核心问题

结合业务诉求和现有实现，迭代7需要解决以下问题：

1. `MixBroker` 的职责边界不清。
   - 当前它更像 `TickBroker` 的扩展版本，而不是完整的中频回测协调层。
2. `tick/orderbook/k线` 的时间对齐规则没有被系统性定义。
   - 若时序规则不清，很容易引入未来函数问题。
3. 策略上下文缺少统一表达。
   - 当前没有一个明确的方式同时向策略暴露“高频窗口状态 + 已完成bar状态”。
4. 多品种场景的统一调度和统一账户语义仍然不清晰。
   - 特别是两个期货合约套利场景，对事件顺序、上下文快照和账户回写要求更高。

## 2. 重分析结论

### 2.1 本轮不是简单补丁优化

本轮不应理解为“给 `MixBroker` 增加几个参数”或“给 `TickBroker` 临时加个bar过滤”。更合理的目标是：

1. 明确中频回测的数据视图和决策时点。
2. 明确中频回测的执行优先级和无高频数据时的未成交边界。
3. 明确 `MixBroker` 在整个链路中的职责定位。
4. 明确单品种和多品种场景下的统一调度方式。

### 2.2 架构方向结论

1. 不建议让 `MixBroker` 同时继承 `TickBroker` 和 `BackBroker`。
   - 两种 broker 的时间推进方式和撮合语义不同，多继承会带来职责混杂。
2. 更合理的方向是将 `MixBroker` 定位为“中频混合模式协调层”。
   - 策略信号使用多源数据状态。
   - 执行优先走高频路径。
   - `bar` 只更新低频状态，不承担任何降级执行职责。
3. 中频回测的关键，不是引入一个新数据源，而是建立“统一时间线 + 多源状态快照 + 高频执行”的组合机制。

### 2.3 本轮目标修正

本轮优先目标不是：

- 追求极限性能。
- 一次性补齐所有高级微观结构特性。
- 改造实盘 broker。

本轮优先目标是：

- 定义清楚中频回测的能力边界。
- 打通业务样例的完整回测闭环。
- 为多品种中频策略建立统一框架。

## 3. 需求定义

### 3.1 P0 — 必须实现

#### R-P0-01：统一时间线与可见性语义

**描述**：在同一回测中，统一 `tick`、`orderbook`、`k线` 的时间推进与数据可见性。

**功能要求**：

1. 策略只能读取当前时点之前已经可见的数据。
2. `orderbook` 和 `tick` 的状态更新必须与执行层时钟保持一致。
3. `bar` 数据必须以“已完成bar”的方式对策略可见，不允许提前读取未完成bar。
4. 同一时点下的数据处理顺序必须是确定的、可复现的。

**验收标准**：

- 同一组输入数据重复回放时，决策结果一致。
- 不出现明显的未来函数问题。
- 业务样例能够稳定读取“过去30个orderbook窗口 + 已完成的1分钟均线状态”。

#### R-P0-02：统一中频策略上下文

**描述**：向策略提供统一的数据访问视图，使其能同时访问高频状态和低频状态。

**功能要求**：

1. 策略能够读取最近 N 个 `orderbook` 样本的窗口统计结果。
2. 策略能够读取最新 `tick` 状态和最近成交相关信息。
3. 策略能够读取已经完成的 `1分钟K线`、`5分钟K线` 及其衍生指标。
4. 策略能够明确区分“当前事件驱动下的可见状态”和“未来才会完成的数据”。

**验收标准**：

- 业务样例可以直接表达为策略逻辑。
- 高频因子和低频过滤能够在同一时点被稳定组合。
- 数据读取口径对单品种和多品种场景一致。

#### R-P0-03：高频优先的执行语义

**描述**：中频策略虽然联合使用bar数据，但订单成交仍应优先以高频数据驱动。

**功能要求**：

1. 策略信号产生后，订单优先使用 `tick/orderbook` 进行撮合。
2. bar 数据默认只参与信号和过滤，不直接替代高频成交语义。
3. 当高频数据缺失时，订单保持未成交，等待后续高频事件。
4. 成交结果需要正确回写现金、持仓、手续费、订单状态和盈亏。

**验收标准**：

- 单品种策略可完成“混合信号 + 高频撮合 + 账户回写”闭环。
- 不会因为使用bar过滤而将执行路径整体降级为bar撮合。
- 无高频数据时不会自动进入 bar 撮合，订单与账户状态仍保持一致。

#### R-P0-04：MixBroker 职责收敛

**描述**：明确 `MixBroker` 在中频回测中的职责，而不是继续让它成为模糊的混合实现。

**功能要求**：

1. `MixBroker` 负责协调多源数据状态与执行链路。
2. `MixBroker` 不承担与 `BackBroker` 完全重叠的bar撮合职责。
3. `MixBroker` 不以多继承方式同时承接 `TickBroker` 和 `BackBroker` 的全部语义。
4. `MixBroker` 的对外行为契约需要文档化。

**验收标准**：

- 能给出清晰的职责边界说明。
- `MixBroker` 的结构方案有明确结论。
- 单品种混合场景下的行为稳定、可解释。

#### R-P0-05：单品种中频策略样例闭环

**描述**：提供最小可用样例，证明中频回测的核心链路已经打通。

**功能要求**：

1. 使用 `orderbook` 计算买卖盘金额比值。
2. 使用 1 分钟均线做方向过滤。
3. 使用高频数据驱动撮合。
4. 输出订单状态、持仓和账户结果。

**验收标准**：

- 业务样例可运行。
- 结果可复现。
- 核心中间状态能够解释。

### 3.2 P1 — 应当实现

#### R-P1-01：多品种统一调度

**描述**：支持多个品种事件共同推进，形成全局一致的回测时间线。

**功能要求**：

1. 多个品种的数据事件按全局时间顺序处理。
2. 每个品种维护自己的高频状态和bar状态。
3. 跨品种信号读取的是“该时点各品种已经可见的最新快照”。
4. 多品种下共享统一组合账户。

**验收标准**：

- 两个期货合约联动或套利场景能够稳定运行。
- 不会出现明显的事件错序问题。
- 组合账户的持仓和盈亏回写正确。

#### R-P1-02：无高频数据场景下的订单边界

**描述**：当 `tick/orderbook` 暂时缺失时，本轮不引入 bar 降级执行，需要明确定义订单保持未成交的边界。

**功能要求**：

1. 不提供任何 bar 降级执行开关、超时窗口或自动降级撮合参数。
2. 缺少高频事件时，订单保持未成交，等待后续高频事件或由策略自行撤单。
3. 未成交状态不会触发虚构成交、虚构现金回写或伪造持仓变化。

**验收标准**：

- tick 可用时优先走高频路径。
- tick / orderbook 缺失时不会进入 bar 撮合。
- 未成交状态不破坏订单和账户状态一致性。

#### R-P1-03：中频策略调试与结果解释能力

**描述**：中频回测需要比纯bar回测更强的时序解释能力。

**功能要求**：

1. 能记录关键事件时点下的策略输入快照。
2. 能定位信号触发、下单、成交、回写的关键时间点。
3. 能区分高频状态变化和bar状态变化。

**验收标准**：

- 样例策略的关键决策链路可回溯。
- 关键场景失败时可以定位是哪一类数据状态导致的问题。

### 3.3 P2 — 可选实现

#### R-P2-01：更多时间粒度和组合策略支持

- 在 1 分钟之外，支持更多bar周期协同。
- 支持更复杂的跨品种过滤、配对和仓位管理逻辑。

#### R-P2-02：中频回测性能优化

- 针对多品种、多窗口统计、多事件调度的场景做结构性优化。
- 验收重点是可解释的性能改进，而不是绝对吞吐承诺。

#### R-P2-03：可视化回放或状态导出

- 为中频策略输出可读的状态快照或时间线报告。
- 为后续策略调参与问题分析提供支持。

## 5. 非功能需求

### 5.1 确定性

- 同一份输入数据重复回放应得到一致结果。
- 同一时间戳下的事件优先级必须是确定的。

### 5.2 可观测性

- 能区分数据状态更新、信号计算、下单、成交、回写几个关键阶段。
- 核心时点应支持调试与回溯。

### 5.3 可扩展性

- 后续可以增加更多bar周期。
- 后续可以增加更多多品种协同模式。
- 后续可以替换更复杂的执行和撮合细节，而不推翻中频回测的总体时间语义。

### 5.4 性能目标

- 本轮先建立中频场景的正确性和结构稳定性。
- 性能以“可接受开销 + 后续可优化”为目标，不将极限吞吐写成硬门槛。

### 5.5 性能基线

- 单品种中频回测（10 万 tick + 1000 bar）运行时间应在 **30 秒以内**（基准机器：普通开发机 4 核 8G）。
- `MidFreqContext.get_ob_ratio()` 单次调用耗时应在 **1 毫秒以内**。
- OrderBook 窗口 100 个 snapshot 的内存开销应在 **100 KB 以内**。

### 5.6 内存约束

- 单品种回测内存增量（相对纯 TickBroker）不超过 **5 MB**。
- 10 品种回测内存增量不超过 **50 MB**。
- 通过 `max_ob_window` 和 `max_bar_history` 参数提供可调控手段。

## 6. 风险评估

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| `tick/orderbook/k线` 可见时点定义不清 | 高 | 容易引入未来函数 | 先固化统一时间线和bar完成语义 |
| `MixBroker` 职责持续膨胀 | 高 | 后续维护和扩展成本快速上升 | 在设计阶段先收敛职责边界 |
| 多品种事件顺序处理不一致 | 高 | 套利和联动场景结果不可信 | 采用全局时间调度和统一快照语义 |
| 信号层和执行层混在一起 | 中 | 文档和代码都难维护 | 强制区分策略上下文与执行协调职责 |
| 过早叠加高级特性 | 中 | 影响本轮主链路交付 | 先交付最小可用闭环，再逐步增强 |

## 8. 术语表

| 术语 | 定义 |
|------|------|
| 中频回测 | 同时使用高频市场状态与低频bar状态进行决策，并以高频路径优先执行的回测模式 |
| 高速状态 | `tick`、`orderbook` 等较高频率更新的数据状态 |
| 低频状态 | 已完成bar及其衍生指标状态 |
| 可见性 | 某时点下策略可以合法读取到的数据范围 |
| 已完成bar | 已经结束、可用于决策的bar，不包含未来尚未完成的数据 |
| 高频优先执行 | 策略信号可来自多频数据，但订单撮合尽量优先走 `tick/orderbook` 路径 |
| 未成交边界 | 在高频数据缺失场景下，订单保持未成交并等待后续高频事件，而不是退化为 bar 撮合 |
| 全局时间调度 | 多品种事件按统一时间顺序推进的调度机制 |
| MidFreqContext | 中频策略统一上下文对象，提供对 MixBroker 状态的只读查询接口 |
| channel | Cerebro 的事件驱动数据通道，负责按时间顺序向 broker 和策略分发 tick/orderbook/bar 事件 |
| event buffer | 同时刻事件缓冲机制，用于保证同一时间戳下不同类型事件的处理顺序固定 |

## 9. 技术经理补充需求

> 以下需求基于对现有代码（`mixbroker.py`、`tickbroker.py`、`cerebro.py`）的审查，补充需求分析师文档中未覆盖但对工程实施至关重要的需求项。

### 9.1 P0 补充

#### R-P0-06：Cerebro channel 事件循环集成

**描述**：中频回测依赖 Cerebro 的 channel 事件循环（`cerebro.py:1006-1033`），需要确保该循环能正确支持混合数据流。

**功能要求**：

1. Cerebro 在 channel 事件循环启动前，需要将 `MixBroker` 的统一上下文注入到所有策略实例。
2. channel 层需要支持将多源数据（tick/orderbook 文件 + bar 文件）按时间戳归并为统一事件流。
3. 对 Cerebro 核心循环的改动控制在最小范围（建议不超过 10 行）。

**验收标准**：

- 策略可通过 `self.context` 访问统一上下文。
- 现有非 channel 模式（`_runnext` / `_runonce`）不受影响。
- 归并后的事件流时间戳严格非递减。

#### R-P0-07：Bar 数据来源与可见性规则

**描述**：明确中频回测中 bar 数据的来源方式和可见性约束。

**功能要求**：

1. 本轮 bar 数据采用**外部独立输入**方式（用户提供已完成的 K 线数据文件）。
2. bar 事件的 `timestamp` 字段定义为 bar 的**结束时间**。
3. `MixBroker` 收到 bar 事件后仅更新低频状态（`_completed_bars`、`_bar_indicators`），**不默认触发 bar 撮合**。
4. 从 tick 实时聚合生成 bar 作为 P2 特性，本轮不实现。

**验收标准**：

- 策略读取的均线、已完成 bar 收盘价等指标均基于已结束的 bar，无未来函数。
- bar 到达时不会触发订单撮合。

### 9.2 P1 补充

#### R-P1-04：OrderBook 窗口缓存管理

**描述**：策略需要读取"过去 N 个 orderbook 样本"，需要 MixBroker 维护品种级滑动窗口。

**功能要求**：

1. `MixBroker` 为每个品种维护 `_ob_window[symbol]` 滑动窗口（`deque` 实现）。
2. 窗口大小通过 `max_ob_window` 参数控制，默认 100。
3. 每次 `process_orderbook` 后自动追加到窗口。
4. 策略通过 `context.get_ob_window(symbol, n)` 或 `context.get_ob_ratio(symbol, levels, window)` 读取。

**验收标准**：

- 窗口不超过 `max_ob_window` 限制。
- 内存开销可预估（单品种 100 个 snapshot ≈ 64 KB）。
- 多品种场景下各品种窗口独立。

#### R-P1-05：Bar 指标增量计算

**描述**：中频模式下每个 bar 到达都需要更新衍生指标（如 SMA），需要高效实现。

**功能要求**：

1. `MixBroker` 每次收到 bar 事件后增量更新 `_bar_indicators[symbol]`。
2. 至少支持 SMA（简单移动均线）的增量计算。
3. 指标计算不重复遍历全部历史 bar。
4. 已完成 bar 缓存通过 `max_bar_history` 参数控制大小，默认 200。

**验收标准**：

- 连续输入 10000 个 bar 事件后，SMA 计算耗时线性增长而非二次增长。
- 策略通过 `context.get_sma(symbol, period)` 可稳定获取正确值。

### 9.3 约束条件补充

#### 4.1 兼容性约束（新增）

| 约束 | 说明 |
|------|------|
| C-06 | `MixBroker` 继续继承 `TickBroker`，不改变继承关系 |
| C-07 | 不对 `TickBroker` 的已有公共接口做破坏性改动 |
| C-08 | `MixBroker` 是实验性组件，本轮直接重构为中频协调层，不需要向后兼容旧版 bar 降级执行行为 |

#### 4.2 技术约束（新增）

| 约束 | 说明 |
|------|------|
| T-05 | bar 数据本轮采用外部独立输入，不从 tick 实时聚合 |
| T-06 | Cerebro 核心循环改动不超过 10 行 |
| T-07 | `MidFreqContext` 只提供只读查询，不暴露可变内部状态 |
| T-08 | 同时刻事件排序在 channel 层或 MixBroker event buffer 中保证，不修改 Cerebro 主循环逻辑 |

### 9.4 非功能需求补充

#### 5.5 性能基线

- 单品种中频回测（10 万 tick + 1000 bar）运行时间应在 **30 秒以内**（基准机器：普通开发机 4 核 8G）。
- `MidFreqContext.get_ob_ratio()` 单次调用耗时应在 **1 毫秒以内**。
- OrderBook 窗口 100 个 snapshot 的内存开销应在 **100 KB 以内**。

#### 5.6 内存约束

- 单品种回测内存增量（相对纯 TickBroker）不超过 **5 MB**。
- 10 品种回测内存增量不超过 **50 MB**。
- 通过 `max_ob_window` 和 `max_bar_history` 参数提供可调控手段。
