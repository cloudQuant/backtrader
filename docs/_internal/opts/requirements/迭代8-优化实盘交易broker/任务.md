## 1. 实施总策略

### 1.1 本轮执行定位

本轮迭代不按“单点优化某个 live 文件”的方式推进，而按以下主线实施：

1. **先定义统一运行契约，再进入实现讨论**
   - 先把“同一策略如何无代码切换回测/实盘”说清楚。
2. **先收敛 broker / store / feed 边界，再讨论 provider 细节**
   - 避免一开始就陷入底层 provider 分支扩展。
3. **先打通高频 live 的最小验证链路，再扩展低频和中频**
   - 高频小样例和快速测试是本轮第一落点。
4. **先建立配置驱动切换方案，再评估运行装配细节**
   - 模式切换由配置承担，而不是由策略源码承担。
5. **以可验证交付物为阶段门禁**
   - 每个阶段结束都要有文档、样例、能力矩阵或测试作为成果。

### 1.2 强制约束

```text
[强制] 不通过修改策略源码来实现 backtest/live 切换
[强制] 不重写 bbroker.py、tickbroker.py、mixbroker.py 的核心回测语义
[强制] 现有稳定测试不能被破坏
[强制] 高频快速测试单例执行时间需控制在 15 秒以内
[强制] BtApiFeed 已存在，本轮按增强方式处理，不重写其核心 bar 回填和 tick 聚合逻辑
```

### 1.3 阶段门禁原则

| 门禁项 | 含义 |
|------|------|
| 契约门禁 | 没有定义清楚统一运行契约前，不进入 broker/store/feed 细化设计 |
| 配置门禁 | 没有明确 profile 和配置边界前，不进入切换方案定稿 |
| live 门禁 | 没有明确 live 订单链路和账户同步语义前，不进入样例固化 |
| 测试门禁 | 没有高频快速测试基线前，不视为 live 主链路可验证 |
| 兼容门禁 | 没有兼容性边界清单前，不推进大范围回归调整 |
