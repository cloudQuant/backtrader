## 1. 设计概览

### 1.1 设计原则

1. **统一契约先于实现细节**
   - 本轮最大的风险不是某个类少了一个方法，而是回测/实盘的策略运行契约不清。
2. **配置切换优先于策略适配**
   - 不允许通过修改策略源码来适配 live。
3. **对外语义统一，对内实现分层**
   - 策略看到的下单、通知、查询语义尽量一致；broker/store/feed 内部实现可以分层演进。
4. **先主链路，后扩展能力**
   - 先打通高频 live 的最小可验证链路，再逐步增强低频、中频和更多 provider。
5. **职责收敛优先**
   - `BtApiBroker`、`BtApiStore`、`BtApiFeed` 的职责必须先收敛，再谈功能扩展。
6. **测试资产与设计同步建设**
   - 没有 15 秒以内的快速测试基线，本轮 live 改造就难以持续回归。

### 1.2 架构总览

```text
┌───────────────────────────────────────────────────────────────┐
│                        Strategy Layer                         │
│                                                               │
│  同一份策略源码：                                             │
│  - 不写 if live / if backtest                                 │
│  - 通过统一 broker/feed 语义运行                               │
└──────────────┬────────────────────────────────────────────────┘
               │
               ▼
┌───────────────────────────────────────────────────────────────┐
│                 Runtime Profile / Config Layer                │
│                                                               │
│  负责根据配置装配：                                            │
│  - BackBroker / TickBroker / MixBroker / BtApiBroker          │
│  - Backtest Feed / 可选 BtApiFeed / 其他 live 数据桥接          │
│  - BtApiStore / provider 参数                                  │
└──────────────┬────────────────────────────────────────────────┘
               │
      ┌────────┴─────────┐
      ▼                  ▼
┌───────────────┐   ┌────────────────────┐
│ Backtest Path │   │    Live Path        │
│               │   │                    │
│ - bbroker     │   │ - BtApiBroker      │
│ - tickbroker  │   │ - BtApiStore       │
│ - mixbroker   │   │ - BtApiFeed(可选)  │
│ - 历史数据输入  │   │ - provider / 行情桥接 │
└───────────────┘   └────────────────────┘
```

### 1.3 重分析后的设计结论

1. **本轮不是做“一个统一 broker”**
   - 回测和实盘继续保留各自实现，但通过统一契约和配置装配形成同一套使用方式。
2. **配置层是回测/实盘切换的核心**
   - 策略不能承担模式切换复杂度，profile 才是边界层。
3. **`BtApiBroker` 是对策略的 live 语义出口**
   - 它负责对策略暴露订单状态、账户状态、持仓状态的统一语义。
4. **`BtApiStore` 是对底层 provider 的归一化入口**
   - 它负责接入和归一化，不应与策略对外语义混杂。
5. **`BtApiFeed` 是已存在的 live 数据桥接层**
    - 本轮按增强方式处理，重点是补齐 `orderbook` 事件通道并保持既有 bar/tick 能力稳定。

## 2. 核心设计

### 2.1 统一运行契约设计

#### 2.1.1 契约目标

统一运行契约需要回答三个问题：

1. 策略在 backtest 和 live 中可以依赖哪些稳定接口？
2. 哪些差异由配置承担，哪些差异由 broker/feed/store 承担？
3. 三类策略在 live 模式下的最小可运行条件是什么？

#### 2.1.2 契约范围

建议把统一运行契约限定在以下边界：

1. **订单能力**
   - 提交订单
   - 撤销订单
   - 接收订单状态通知
2. **账户能力**
   - 查询现金
   - 查询持仓
   - 查询账户价值
3. **数据能力**
   - 策略能消费 bar / tick / orderbook 等自己所需的数据事件
4. **装配能力**
   - broker / feed / store / provider 通过配置装配

#### 2.1.3 不纳入统一契约的部分

本轮不建议把以下内容写入强统一契约：

- 回测与实盘逐笔成交一致性
- 所有 provider 的高级细节字段
- 全量风控、断路器、监控平台能力
- 所有异常恢复细节的统一实现方式

### 2.2 运行 profile 设计

#### 2.2.1 设计目标

运行 profile 用于表达同一策略在不同模式下的装配方式。建议按“数据粒度 + 运行模式”来组织。

#### 2.2.2 推荐 profile

建议至少定义以下几类 profile：

| Profile | 说明 |
|---------|------|
| `lowfreq-backtest` | 低频 bar 回测 |
| `midfreq-backtest` | 中频混合回测 |
| `hft-backtest` | 高频 tick/orderbook 回测 |
| `lowfreq-live` | 低频实盘 |
| `midfreq-live` | 中频实盘 |
| `hft-live` | 高频实盘 |

#### 2.2.3 profile 应包含的内容

一个 profile 至少应描述：

- 运行模式：`backtest` 或 `live`
- 使用的 broker 类型
- 使用的 feed 类型
- 使用的 store 类型
- provider 名称与连接参数
- 数据粒度与事件类型
- 是否启用 mock / fake 测试装配

### 2.3 broker / store / feed 职责设计

#### 2.3.1 BtApiBroker 职责

建议 `BtApiBroker` 负责：

1. 对策略暴露 live broker 语义。
2. 接收策略订单。
3. 承担订单状态的对外生命周期管理。
4. 将 store 归一化后的账户、持仓、订单、成交信息映射为策略可见结果。
5. 控制对策略的通知节奏与通知口径。

不建议 `BtApiBroker` 负责：

- 底层 provider 具体协议适配
- 复杂行情解析和字段归一化
- 把所有 provider 差异直接暴露给策略

#### 2.3.2 BtApiStore 职责

建议 `BtApiStore` 负责：

1. 连接到底层 provider。
2. 归一化账户、持仓、订单、成交、行情等底层事件。
3. 为 broker 和 feed 提供统一的 live 数据来源。
4. 管理底层连接状态与 provider 级接口细节。

不建议 `BtApiStore` 负责：

- 直接承担策略可见语义定义
- 直接替代 broker 做订单生命周期对外通知
- 承担策略运行模式切换逻辑

#### 2.3.3 BtApiFeed 职责

本轮将 `BtApiFeed` 定位为：**已存在并按增强方式处理的 live 数据桥接层**。

本轮建议其职责包括：

1. 将 store 提供的实时行情或增量事件桥接成策略可消费的数据形式。
2. 为低频、中频、高频策略提供对应的数据输入适配。
3. 在不污染策略源码的前提下承接 live 数据桥接工作。

不建议 `BtApiFeed` 负责：

- 直接实现 provider 协议接入
- 直接定义 broker 对外订单语义
- 兼任 store 的数据归一化职责

### 2.4 三类策略 live 能力设计

#### 2.4.1 低频策略 live profile

低频策略的重点是：

- 以 bar / kline 为主的数据输入
- live 下单与账户同步
- 保持和 `bbroker` 回测相近的策略使用方式

建议低频策略 live 至少具备：

- 已完成 bar 数据输入
- 基本订单提交流程
- 账户与持仓查询
- 订单状态通知

#### 2.4.2 中频策略 live profile

中频策略的重点是：

- 同时消费高频状态和低频状态
- 不修改策略源码前提下保留上下文访问方式
- live 侧能够提供中频策略所需的状态组合能力

建议中频策略 live 至少具备：

- 高频输入：tick / orderbook 或其等价状态
- 低频输入：bar / kline
- 统一上下文或等价访问视图
- live 下单与账户同步

#### 2.4.3 高频策略 live profile

高频策略是本轮优先验证对象。重点是：

- tick / orderbook 输入
- live 订单状态通知
- 小样例和快速测试可落地

建议高频策略 live 至少具备：

- 实时 tick / orderbook 输入
- 订单提交流程与撤单流程
- 部分成交 / 完成 / 取消通知
- 账户与持仓查询

### 2.5 配置切换设计

#### 2.5.1 切换原则

配置切换的基本原则：

1. 策略源码不包含模式分支。
2. 模式切换在配置文件中表达。
3. 配置文件决定装配哪一种 broker/feed/store/profile。
4. 配置差异不应泄漏成策略内部的显式判断。

#### 2.5.2 推荐配置结构

推荐配置至少包含以下部分：

- `runtime.mode`
- `runtime.profile`
- `broker.class`
- `store.class`
- `feed.class`
- `provider.name`
- `provider.connection`
- `data.channels`
- `test.mock_enabled`

#### 2.5.3 配置驱动效果

配置层最终应能支持如下切换：

- 同一策略 + `hft-backtest` 配置 = 高频回测
- 同一策略 + `hft-live` 配置 = 高频实盘
- 同一策略 + `lowfreq-live` 配置 = 低频实盘
- 同一策略 + `midfreq-live` 配置 = 中频实盘

### 2.6 live 数据链路设计

#### 2.6.1 数据流目标

设计上需要把 live 数据流抽象成：

```text
Provider -> BtApiStore -> (可选 BtApiFeed) -> Cerebro/Strategy
```

或在某些场景下：

```text
Provider -> BtApiStore -> Broker / Strategy Notification
```

#### 2.6.2 事件类型

建议按以下事件类型来组织 live 数据输入：

- `bar`
- `tick`
- `orderbook`
- `order_status`
- `trade_fill`
- `account_update`
- `position_update`

#### 2.6.3 设计要求

1. 时间戳语义必须稳定。
2. symbol 映射必须稳定。
3. 事件顺序必须可解释。
4. 策略回调与 broker 状态更新顺序必须可解释。

### 2.7 live 订单链路设计

#### 2.7.1 主流程

推荐的 live 订单主流程如下：

```text
策略发起下单
    ↓
BtApiBroker 接收订单
    ↓
BtApiBroker 调用 BtApiStore 提交到底层 provider
    ↓
BtApiStore 返回提交结果 / 外部单号 / 初始状态
    ↓
BtApiBroker 对策略发出接受/拒绝通知
    ↓
后续订单状态、成交、取消由 store 回传并由 broker 统一映射
    ↓
Broker 更新账户/持仓可见状态
```

#### 2.7.2 订单状态边界

本轮需要优先文档化以下状态边界：

- 已提交
- 已接受
- 已拒绝
- 部分成交
- 已完成
- 已取消

#### 2.7.3 账户同步边界

账户同步至少应覆盖：

- 现金变化
- 持仓变化
- 账户价值变化
- 成交结果可见性

### 2.8 快速测试设计原则

#### 2.8.1 测试目标

快速测试不是验证所有 provider 细节，而是验证统一契约和 live 主链路是否成立。

#### 2.8.2 测试对象优先级

建议优先测试：

1. 高频策略样例
2. `BtApiBroker` 的订单生命周期语义
3. `BtApiStore` 的归一化输出语义
4. `BtApiFeed` 的最小桥接能力（若本轮需要创建/增强）

#### 2.8.3 测试方法

建议大量使用：

- 小数据集
- fake provider
- mock store
- 可控事件序列

以保证单个用例可以稳定控制在 15 秒以内。

## 3. 关键流程设计

### 3.1 backtest/live 配置切换流程

```text
选择策略类
    ↓
读取 runtime profile 配置
    ↓
根据配置装配 broker / store / feed
    ↓
启动 Cerebro
    ↓
策略在统一契约下运行
```

### 3.2 live 数据输入流程

```text
provider 推送行情/状态
    ↓
BtApiStore 归一化事件
    ↓
若需要，BtApiFeed 进一步桥接为策略数据输入
    ↓
Cerebro / Broker / Strategy 接收事件
```

### 3.3 live 订单与账户回写流程

```text
策略提交订单
    ↓
BtApiBroker 对外管理订单生命周期
    ↓
BtApiStore 与底层 provider 交互
    ↓
订单/成交/账户更新回传
    ↓
BtApiBroker 更新对策略可见状态
```

### 3.4 高频快速测试流程

```text
选择高频策略样例
    ↓
裁剪小数据集或构造 fake 事件流
    ↓
mock/fake provider 或 store
    ↓
运行高频测试
    ↓
验证执行时间、状态通知、账户更新语义
```

| 决策项 | 结论 |

## 6. 设计风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 统一契约定义不足 | 策略层后续会出现大量隐式分支 | 先冻结最小统一契约 |
| broker/store/feed 边界不清 | live 主链路难维护 | 在设计阶段收敛职责 |
| `BtApiFeed` 定位模糊 | 数据链路方案反复摇摆 | 先明确是否创建/增强 |
| 快速测试过慢 | 回归成本过高 | 采用小数据集和 mock/fake |
| 低频/中频/高频需求混在一起 | 设计无法落地 | 使用 profile 和能力矩阵拆分 |

---

## 7. 建议的后续实现顺序

1. 先落统一运行契约和配置 profile。
2. 再定义三类策略的 live 能力矩阵。
3. 再收敛 `BtApiBroker`、`BtApiStore`、`BtApiFeed` 的职责与边界。
4. 然后打通高频 live 的最小样例和快速测试链路。
5. 最后扩展低频/中频 live 支持和更多 provider 能力。

## 8. 技术经理评审补充

> 以下内容由技术经理基于完整代码审查后补充，用于修正设计假设、给出具体接口定义和实现方案。

### 8.1 事实修正与设计假设调整

#### 8.1.1 BtApiFeed 定位修正

设计文档 §1.3.5 和 §2.3.3 将 `BtApiFeed` 定位为"可创建/可增强的 live 数据桥接层"。**事实上该文件已存在**（`backtrader/feeds/btapifeed.py`，320 行），需将设计定位调整为：

> `BtApiFeed` 是已存在的 live 数据桥接层，本轮按增强方式处理。核心增强项为 orderbook 事件通道。不重写其已稳定的 bar 回填和 tick→bar 聚合逻辑。

#### 8.1.2 broker/store/feed 职责现状

代码审查确认三组件职责分工**已基本合理**，无重大交叉：

| 组件 | 实际职责 | 是否需要重大调整 |
|------|---------|----------------|
| BtApiBroker | 订单校验→store 路由→远程状态同步→策略通知→账户/持仓缓存→交易控制 | ❌ 补容错细节即可 |
| BtApiStore | 连接管理→多 provider 支持→数据归一化→订单/成交桥接→runtime 事件审计 | ❌ 补 orderbook polling 接口 |
| BtApiFeed | 历史回填→live bar 轮询→tick→bar 聚合→tick/bar 事件分发 | ⚠️ 补 orderbook 事件通道 |

因此设计文档中关于"职责收敛"的大量讨论（§2.3 整节）可简化为：**确认现有分工、补齐缺口、文档化边界**。

#### 8.1.3 Cerebro channel 事件循环

设计文档未提及 Cerebro 的 channel 事件循环对 BtApiBroker 的影响。关键事实：

```python
# cerebro.py:1006-1033 — channel 事件循环
if ch == "tick" and hasattr(self._broker, "process_tick"):
    self._broker.process_tick(evdata)
elif ch == "orderbook" and hasattr(self._broker, "process_orderbook"):
    self._broker.process_orderbook(evdata)
elif ch == "bar" and hasattr(self._broker, "process_bar"):
    self._broker.process_bar(evdata)
```

BtApiBroker **不实现**这三个方法 → channel 事件对 broker 层透传 → 策略仍通过 `notify_tick()` / `notify_orderbook()` / `notify_bar()` 收到事件。

**设计决策**：这是正确的。实盘 broker 不做本地撮合，订单执行由交易所完成。本轮不改变此设计。

#### 8.2.3 build_cerebro 工厂函数

```python
def build_cerebro(profile: LiveProfile, strategy_cls, **strategy_kwargs):
    """根据 LiveProfile 装配 Cerebro，策略类不感知运行模式。"""
    cerebro = bt.Cerebro()

    if profile.mode == "backtest":
        # 回测路径
        broker_cls = _resolve_broker_class(profile.broker_class)
        cerebro.setbroker(broker_cls(**profile.broker_params))
        for symbol in profile.symbols:
            feed_cls = _resolve_feed_class(profile.feed_class)
            cerebro.adddata(feed_cls(
                dataname=profile.data_path or symbol,
                **profile.feed_params,
            ))
    elif profile.mode == "live":
        # 实盘路径
        store = BtApiStore(
            provider=profile.store_provider,
            **profile.store_params,
        )
        cerebro.setbroker(store.getbroker(**profile.broker_params))
        for symbol in profile.symbols:
            cerebro.adddata(store.getdata(
                dataname=symbol,
                **profile.feed_params,
            ))

    cerebro.addstrategy(strategy_cls, **strategy_kwargs)
    return cerebro
```

#### 8.2.4 BtApiFeed orderbook 增强

需要在 `BtApiFeed._load()` 的 live 分支中增加 orderbook 轮询和分发。伪代码：

```python
# btapifeed.py — 在 _drain_live_ticks() 之后增加
def _drain_live_orderbooks(self):
    """Poll and dispatch orderbook events from the store."""
    while True:
        ob = self.store.poll_orderbook(self._dataname)
        if ob is None:
            break
        self._dispatch_event(
            channel_type="orderbook",
            priority=EventPriority.ORDERBOOK,
            event_data=ob,
        )
```

对应需要在 BtApiStore 中增加 `poll_orderbook(symbol)` 方法，以及在 FakeBtApiClient 中增加 `live_orderbooks` 队列。

#### 8.2.5 BtApiBroker cancel() 语义调整

当前 `cancel()` 立即将本地订单标记为 Canceled，不等待远程确认。建议调整为：

```python
def cancel(self, order):
    """Request cancellation — local state waits for remote confirmation."""
    self.store.cancel_order(order)
    # 不立即 order.cancel()，而是标记为 PendingCancel
    order.status = bt.Order.Cancelled  # 保持兼容：仍立即标记
    # 后续通过 _drain_store_updates() 收到远程确认后再最终确认
    self.notify(order)
    return order
```

为保持兼容性，建议增加 `cancel_wait_remote=False` 参数，默认行为不变，opt-in 等待远程确认。

### 8.3 三类策略 live 链路具体设计

#### 8.3.1 低频策略 live 链路

```text
BtApiStore.fetch_history(symbol)
    → BtApiFeed._load_history() → 回填 line 数据
    → BtApiFeed._load() [live 分支] → store.poll_live(symbol) → 新 bar
    → Cerebro.next() → Strategy.next()
    → Strategy.buy/sell → BtApiBroker.submit → BtApiStore.submit_order
    → BtApiStore.poll_broker_update → BtApiBroker._drain_store_updates
    → Strategy.notify_order(order)
```

**状态**：已基本可用，无阻塞项。

#### 8.3.2 高频策略 live 链路

```text
BtApiStore → BtApiFeed._drain_live_ticks()
    → _dispatch_event(channel_type="tick") → Cerebro → Strategy.notify_tick()

BtApiStore → BtApiFeed._drain_live_orderbooks()  [新增]
    → _dispatch_event(channel_type="orderbook") → Cerebro → Strategy.notify_orderbook()

Strategy.buy/sell → BtApiBroker.submit → BtApiStore.submit_order
BtApiStore.poll_broker_update → BtApiBroker._drain_store_updates
    → Strategy.notify_order(order)
```

**已解决项**：orderbook 通道已完成，BtApiFeed 与 BtApiStore 均已具备对应能力。

#### 8.3.3 中频策略 live 链路

```text
同时具备：
- 低频链路：bar 数据 → next()
- 高频链路：tick/orderbook → notify_tick()/notify_orderbook()
- 中频上下文：MidFreqContext（需要 BtApiBroker 或外部注入机制支持）
```

**状态**：复杂度最高。本轮仅做 POC 级验证——用 FakeBtApiClient 注入 tick+bar 混合数据，验证策略同时收到 `next()` 和 `notify_tick()` 回调。不要求 MidFreqContext 在实盘侧完整可用。
