## 1. 交接结论

经过对现有代码和迭代8文档的复核，本轮已经具备进入研发实施的条件，但需要按**代码现实**而不是按早期设想推进。

当前可确认的事实如下：

1. 实盘主链路基础设施已经存在。
   - `backtrader/brokers/btapibroker.py` 已具备订单提交、账户同步、持仓同步、通知等基础能力。
   - `backtrader/stores/btapistore.py` 已具备 provider 接入、账户/持仓查询、broker/feed 工厂等能力。
   - `backtrader/feeds/btapifeed.py` 已存在，且具备历史 bar 回填、live bar 轮询、tick→bar 聚合、tick/bar 事件分发能力。
2. 实施前真正的阻塞项并不多，但非常集中。
   - 高频 live 侧缺少 `orderbook` 事件通道。
   - 缺少一组可稳定回归的 `<15s` 高频快速测试。
   - 缺少完整的 FakeClient → Feed → Strategy → Broker → Store 端到端验证。
   - 缺少一个轻量的运行装配入口来证明“同一策略不改代码、仅靠配置/装配切换 backtest/live”。
3. 因此，迭代8不应继续按“大而全 live 统一平台”展开，而应按“**补阻塞项 + 冻结边界 + 先打通高频最小闭环**”推进；本轮也已按这一顺序完成收口。

本交接说明的目的，是把迭代8从“文档已很丰富”进一步收敛为“研发知道先做什么、哪些不能扩、做到什么算交付”。

## 2. 当前实现判断

### 2.1 已具备的基础

- `BtApiBroker` 已能对策略提供基础 live broker 语义。
- `BtApiStore` 已能作为统一 provider 接入与归一化入口。
- `BtApiFeed` 已经不是“待创建组件”，而是“待增强组件”。
- 现有测试中，broker/store 基础测试已经具备，FakeBtApiClient 也已可复用。

### 2.2 已关闭的主要缺口

- `BtApiFeed` 的 `orderbook` 事件分发已完成。
- 高频 live 的小数据、快反馈回归资产已完成。
- `LiveProfile + build_cerebro()` 轻量装配层已完成。
- 中频 live 仍按 POC 冻结，不扩展为完整实现。

### 2.3 产品判断

本轮最容易失控的方式有两种：

1. 把迭代8做成“所有 provider、所有频率、所有 live 细节的一次性统一平台”。
2. 在 `BtApiBroker` / `BtApiStore` / `BtApiFeed` 之间继续模糊职责，一边补功能一边改边界。

因此，迭代8必须以**最小可验证主链路**为核心，而不是以“覆盖面”作为第一目标。

## 3. 优化后的范围冻结

### 3.1 P0 — 本轮必须完成

P0 只保留四个结果：

1. **高频 live 的 orderbook 通道补齐**
   - `BtApiStore.poll_orderbook(symbol)`
   - `BtApiFeed._drain_live_orderbooks()`
   - `FakeBtApiClient.live_orderbooks`
   - 策略 `notify_orderbook()` 在 live 下可收到回调
2. **高频快速测试基线**
   - 至少一组 `<15s` 高频快速测试
   - 覆盖 tick / orderbook / 下单 / 订单通知主链路
3. **端到端 live 最小闭环**
   - FakeClient → Feed → Strategy → Broker → Store 全链路可测
4. **轻量装配层证明“策略源码不改”**
   - 不引入重配置框架
   - 交付 `LiveProfile + build_cerebro()` 即可

### 3.2 P1 — 本轮建议完成，但不阻塞 P0 交付

1. `BtApiBroker.cancel()` 语义优化（保留兼容默认值）
2. `_drain_store_updates()` 的 error 处理补齐
3. 高频模式下 `getcash()/getvalue()` 缓存优化
4. 低频 live profile 验证样例补强

### 3.3 P2 — 明确延后

以下内容统一延后，不纳入本轮主链路交付：

1. 全 provider 的高级能力补齐
2. 完整的中频 live 能力落地
3. 断连重连、风控、监控、容灾平台
4. 回测/实盘逐笔一致性度量
5. 对实盘 broker 引入本地撮合语义

## 6. 测试冻结与兼容性规则

### 6.1 不能破坏的内容

以下稳定测试和行为口径应保持不变：

- 既有 `BtApiBroker` 基础订单/账户语义
- 既有 `BtApiStore` provider 切换与工厂行为
- 既有 `BtApiFeed` 的历史 bar / live bar / tick→bar 主能力
- 既有稳定测试文件的接口签名与断言语义

### 6.2 允许新增和扩展的内容

1. 新增 orderbook 通道相关测试
2. 新增高频快速测试
3. 新增 `LiveProfile` 装配测试
4. 扩展 `FakeBtApiClient` 的新属性和新方法

### 6.3 明确禁止

1. 不允许通过修改策略源码实现 live/backtest 切换
2. 不允许在实盘 broker 内重新引入本地撮合
3. 不允许为了中频 live 一次性引入完整复杂上下文体系
4. 不允许为了新增测试而破坏稳定测试基线

## 7. 研发容易踩坑的点

### 7.1 不要把 `LiveProfile` 做成重型配置框架

本轮只需要一个轻量装配层来证明“模式切换成立”，不是要上 YAML 框架、DI 容器或复杂配置系统。

### 7.2 不要在 `BtApiBroker` 上补 `process_tick()` / `process_orderbook()`

实盘 broker 不做本地撮合是正确设计。live 侧事件应继续由 feed → strategy 分发，broker 只负责订单和账户语义。

### 7.3 不要为了中频 live 过早承诺完整实现

中频 live 会牵涉上下文注入、高低频状态组合、事件顺序和实时一致性。本轮只需要做 POC，不能挤占高频主链路工时。

### 7.4 不要让 Fake 测试夹具失去兼容性

`FakeBtApiClient` 可以扩展，但不要修改现有接口签名，否则会误伤已有测试。

## 8. Definition of Done

研发完成本轮后，至少需要同时满足以下条件：

1. live 高频策略能收到 `notify_tick()` 和 `notify_orderbook()`。
2. `BtApiFeed` orderbook 通道有独立测试覆盖。
3. 至少一组高频快速测试满足 `<15s`。
4. 存在 FakeClient 驱动的端到端 live 集成测试。
5. `LiveProfile + build_cerebro()` 能证明同一策略可切换 backtest/live。
6. 中频 live 的范围被明确限制在 POC，不误导为已完整支持。
7. 文档、任务、设计、测试口径一致。
