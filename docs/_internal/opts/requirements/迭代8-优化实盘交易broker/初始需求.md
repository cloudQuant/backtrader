## 背景

当前 `backtrader` 在回测侧已经具备较完整的低频、高频和中频能力，实盘侧也已经具备 `BtApiBroker`、`BtApiStore`、`BtApiFeed` 等基础设施。但回测/实盘之间的统一运行契约、live 数据链路能力边界和高频快速回归资产仍不够清晰，需要在本轮迭代中统一收敛。

## 原始诉求

1. 低频、中频和高频策略应在**不修改策略源码**的前提下，同时支持回测与实盘切换；允许通过配置和装配差异区分 backtest/live。
2. 优先落地一批 `tickbroker` 相关的高频策略测试，复用 `examples` 中已有思路，并将单个测试用例执行时间控制在 15 秒以内。
3. 优化 `btapibroker.py`、`btapistore.py`，并评估/增强 `btapifeed.py`，使实盘链路能够逐步支撑低频、中频和高频策略。
4. 现有稳定测试需要保持兼容；除 `mixbroker`、`tickbroker` 相关新增测试外，其余既有测试不应调整断言语义。

## 需求梳理与迭代8实施计划

#### 1. 业务定义

1. 迭代8的核心诉求，不是单独“优化一个实盘 broker 文件”，而是让 `backtrader` 的策略运行模式形成统一的切换能力：
   - 同一份低频策略，可以在回测和实盘之间切换
   - 同一份中频策略，可以在回测和实盘之间切换
   - 同一份高频策略，可以在回测和实盘之间切换
2. 这里的“无需改策略代码”指的是：
   - 策略类本身不增加 `if live` / `if backtest` 之类的分支
   - 环境差异通过配置文件、broker/feed/store 装配和启动参数来表达
3. 实盘链路不仅包含 `BtApiBroker`，还包含：
   - `BtApiStore` 负责账户、持仓、订单、成交、行情等底层连接和归一化
   - `BtApiFeed` 负责把实盘行情映射成策略可消费的数据/事件
   - `Cerebro` 负责把 broker、feed、strategy 在运行时组装起来
4. 本轮业务目标不是让实盘与回测在“成交细节上完全一致”，而是让策略接口、数据访问方式、订单生命周期和配置切换方式尽可能一致。
5. 本轮还要求先用较小数据集建立高频策略的快速测试基线，确保新增能力可以被稳定回归验证。

#### 2. 对当前问题的分析结论

1. 当前回测侧已经形成三类主要能力：
   - `bbroker.py`：低频 bar 回测
   - `tickbroker.py`：高频 tick / orderbook 回测
   - `mixbroker.py`：中频混合模式回测
2. 当前实盘侧已经存在：
   - `backtrader/brokers/btapibroker.py`
   - `backtrader/stores/btapistore.py`
   - `backtrader/feeds/btapifeed.py`（已存在，本轮按增强方式处理）
   因此，迭代8更适合定义为“统一实盘运行契约并补齐 live 适配层”，而不是从零新建一套实盘框架。
3. 当前最大的缺口，不是“没有 live broker”，而是以下几个方面尚未系统统一：
   - 回测/实盘之间的策略运行契约是否一致
   - 低频/中频/高频三类策略在 live 模式下分别依赖哪些 feed 和 broker 能力
   - 配置层如何表达“同一策略切换到回测或实盘”
   - 实盘订单状态、账户同步、持仓同步、行情事件与策略回调之间是否形成清晰边界
4. 现有测试兼容性约束非常强：
   - 存量测试不能被破坏
   - 只有 `mixbroker`、`tickbroker` 相关的新增测试允许调整
   - 因此本轮必须优先采用“新增能力 + 最小侵入”的演进方式
5. 用户希望先做一批高频策略测试样例，这说明本轮除了架构整理，还必须尽快形成可快速执行的验证资产。

#### 3. 架构判断

1. “策略代码不改、靠配置切换回测/实盘”的前提，是先定义统一运行契约，而不是先堆更多 provider 细节。
2. 统一运行契约建议至少包括以下几类内容：
   - 策略如何读取数据
   - 策略如何下单、撤单、查询持仓/资金
   - broker 如何通知订单状态变化
   - feed 如何向策略推送低频、中频、高频事件
3. 回测与实盘应在“策略接口层”尽可能统一，但不强行要求在“内部实现层”完全统一：
   - 回测 broker 继续分别承担低频、中频、高频撮合职责
   - 实盘 broker 负责把策略订单路由到真实交易接口，并同步真实账户状态
   - 二者统一的是策略可见契约和配置切换方式
4. `BtApiFeed` 已经存在，因此本轮优先方向应是：
   - 评估并增强现有 `BtApiFeed`
   - 必要时补其高频/中频事件映射能力
   - 不优先新建平行 feed 框架
5. 对低频、中频、高频的 live 支持应理解为三种运行 profile：
   - 低频 live profile：bar 数据 + 实盘下单
   - 中频 live profile：bar + 高频状态/上下文 + 实盘下单
   - 高频 live profile：tick / orderbook + 实盘下单
6. 本轮不建议以修改策略类来适配实盘，而应通过配置、装配器和统一上下文能力来保证兼容。

#### 4. 迭代8目标

1. 明确“同一策略通过配置切换回测/实盘”的总体方案与边界。
2. 明确低频、中频、高频三类策略在 live 模式下分别需要哪些 broker/feed/store 能力。
3. 优化 `BtApiBroker`、`BtApiStore`、`BtApiFeed` 的职责协同，使其能支撑三类策略运行。
4. 建立一组轻量、快速、可回归的高频策略测试与样例，确保单个测试用例执行时间小于 15 秒。
5. 在不破坏现有稳定测试的前提下，形成迭代8的实施边界、阶段计划和验收标准。

#### 5. 本轮纳入范围

1. 回测/实盘统一运行契约的需求梳理
2. 配置驱动的运行 profile 设计
3. `BtApiBroker` 的订单提交流程、撤单流程、通知流程、账户同步流程优化
4. `BtApiStore` 的 provider 接入归一化、订单/成交/持仓/账户事件归一化优化
5. `BtApiFeed` 对低频、中频、高频策略所需数据事件的支持能力梳理与增强
6. 高频策略的小样例、小数据集、快速测试用例设计
7. 兼容性策略：现有测试保持稳定，新增测试按新契约补齐

#### 6. 本轮非目标范围

1. 不重写 `bbroker.py`、`tickbroker.py`、`mixbroker.py` 的核心回测语义
2. 不要求实盘成交结果与回测撮合结果逐笔完全一致
3. 不在本轮覆盖所有实盘 provider 的全部高级能力
4. 不把本轮目标定义为完成完整的风控、容灾、运维监控平台
5. 不改动既有稳定测试的断言口径，除非是 `mixbroker`、`tickbroker` 的新增测试资产

#### 7. 分阶段实施计划

##### Phase 0：需求澄清与现状基线（预计 1-2 天）

目标：把“回测/实盘统一运行到底要求什么、现有 live 链路缺什么”说清楚。

主要输出：

1. 低频、中频、高频三类策略的运行方式对比表
2. 回测链路与实盘链路的职责映射表
3. `BtApiBroker` / `BtApiStore` / `BtApiFeed` 当前能力清单与差距清单
4. 本轮范围、非范围、风险清单

阶段完成标准：

1. 明确“策略代码不改”的准确含义
2. 明确 live 模式下三类策略各自依赖的能力集合
3. 明确本轮优先从高频小样例和快速测试切入

##### Phase 1：统一运行契约与配置切换方案（预计 3-4 天）

目标：定义同一策略在回测/实盘之间通过配置切换的统一方案。

主要任务：

1. 定义回测与实盘的运行 profile 结构
2. 定义 broker / feed / store 的装配方式和配置项边界
3. 明确策略对外可依赖的统一能力：
   - 下单/撤单
   - 查询现金/持仓/账户价值
   - 接收 tick / orderbook / bar / 订单通知
4. 明确低频、中频、高频三类策略切换到 live 时的最小可运行配置

阶段完成标准：

1. 同一策略切换回测/实盘不需要修改策略源码
2. 环境差异由配置和装配承担，而不是由策略分支承担
3. 三类策略的 live profile 具备清晰定义

##### Phase 2：实盘数据链路与 Feed 能力梳理（预计 3-5 天）

目标：明确 `BtApiFeed` 如何支撑低频、中频、高频三类策略的数据输入。

主要任务：

1. 盘点现有 `BtApiFeed` 的能力与缺口
2. 明确 live 模式下：
   - 低频策略需要哪些 bar 数据与更新语义
   - 中频策略需要哪些高频状态与低频状态组合能力
   - 高频策略需要哪些 tick / orderbook 事件
3. 定义 live 数据事件到策略回调之间的映射规则
4. 明确时间戳、symbol、交易时段、数据完整性和事件顺序的约束

阶段完成标准：

1. 三类策略所需的数据事件来源明确
2. `BtApiFeed` 是否增强、扩展还是局部重构有明确结论
3. 实盘数据对策略回调的映射口径文档化

##### Phase 3：实盘订单链路与账户同步优化（预计 4-6 天）

目标：让 `BtApiBroker` 与 `BtApiStore` 稳定支撑三类策略的 live 下单与状态回写。

主要任务：

1. 梳理 `BtApiBroker` 的订单提交、接受、拒绝、取消、完成通知链路
2. 梳理 `BtApiStore` 的订单、成交、账户、持仓同步机制
3. 明确 live 模式下订单生命周期与策略通知契约
4. 明确账户资金、持仓、订单状态刷新节奏与一致性要求
5. 明确错误处理、断连恢复、交易暂停等基础行为边界

阶段完成标准：

1. 策略能够稳定收到订单状态变化
2. 账户/持仓同步链路具备清晰语义
3. `BtApiBroker`、`BtApiStore` 的职责边界清晰，不再混杂

##### Phase 4：高频样例与快速测试基线（预计 3-4 天）

目标：优先形成一批高频小样例和 15 秒以内的快速测试用例。

主要任务：

3. 补充 `tickbroker` 相关高频策略测试
4. 如有必要，为 `BtApiBroker` / `BtApiFeed` 增加 mock 或 fake 测试夹具
5. 明确哪些新增测试允许调整，哪些存量测试必须完全保持不变

阶段完成标准：

1. 至少形成一组高频策略快速测试
2. 单个测试用例执行时间控制在 15 秒以内

#### 8. 验收标准

1. 能明确回答：同一份低频、中频、高频策略，如何在不改策略代码的情况下切换回测/实盘。
2. `BtApiBroker`、`BtApiStore`、`BtApiFeed` 在 live 链路中的职责边界明确。
3. 至少形成一组高频策略小样例和快速测试用例，且单个测试执行时间小于 15 秒。
4. 现有稳定测试不被破坏，兼容性边界清晰。
5. 形成与迭代8配套的迭代计划、范围说明和后续文档输入基础。

#### 9. 主要风险

1. “策略代码不改”如果没有先定义清楚统一契约，后续很容易演变成大量隐式兼容逻辑。
2. `BtApiBroker`、`BtApiStore`、`BtApiFeed` 的职责若继续交叉，后续维护成本会持续上升。
3. live 数据事件与策略回调的映射如果定义不清，低频、中频、高频三类策略会表现不一致。
4. 如果高频测试数据集过大，回归成本会很快失控，难以满足 15 秒约束。
5. 如果一开始就试图覆盖过多 provider 或高级实盘能力，会影响本轮主链路交付。

## 技术经理评审补充

> 以下内容由技术经理基于完整代码审查后补充，供后续需求文档、设计文档和任务分解参考。

### A. 事实修正

早期需求讨论曾将 `backtrader/feeds/btapifeed.py` 视为“如果需要可以创建”的组件。**实际上该文件已存在**，共 320 行，具备以下能力：

- 历史 bar 回填（从 store 获取或初始化时注入）
- Live bar 轮询（通过 `store.poll_live()`）
- Tick→Bar 聚合（支持 Seconds / Minutes / Days 三种 timeframe）
- Tick 事件与 Bar 事件通过 `Cerebro.dispatch_channel_event()` 分发到策略的 `notify_tick()` / `notify_bar()` 回调
- `islive()` / `haslivedata()` 判断

因此迭代8的方向应从"是否需要创建 BtApiFeed"调整为"评估并增强现有 BtApiFeed"。

### B. 三组件当前能力清单与差距分析

#### B.1 BtApiBroker（707 行）

| 能力项 | 当前状态 | 评估 |
|--------|----------|------|
| 订单提交 → store 路由 | ✅ 已实现 | 稳定 |
| 本地下单前校验（tick size、max size、合约有效性） | ✅ 已实现 | 稳定 |
| 远程订单状态同步（accepted/canceled/rejected） | ✅ 已实现 | 稳定 |
| 远程成交回填（trade fill → position update） | ✅ 已实现 | 稳定 |
| 账户余额/持仓定时刷新 + 节流 | ✅ 已实现 | 稳定 |
| 交易控制（disable/enable/pause/resume/force_logout） | ✅ 已实现 | 稳定 |
| 批量撤单 | ✅ 已实现 | 稳定 |
| Runtime 事件审计日志 | ✅ 通过 store 代理 | 稳定 |
| `process_tick()` / `process_orderbook()` / `process_bar()` | ❌ 无 | **设计上正确**（实盘不做本地撮合），但需明确文档化 |
| `cancel()` 等待远程确认 | ❌ 立即标记本地 Canceled | ⚠️ 可能导致本地/远程状态短暂不一致 |
| `_drain_store_updates()` 对 "error" kind 的处理 | ❌ 忽略 | 应补充错误事件的转发 |
| `getcash()` / `getvalue()` 每次 force refresh | ⚠️ 高频调用时可能阻塞 | 考虑在高频 profile 下走缓存 |
| 断线重连逻辑 | ❌ 无 | 本轮不强求，但需标注为后续风险 |

**核心结论**：BtApiBroker 的订单路由链路已基本完整，缺口主要在**容错细节**和**高频场景下的性能优化**，而非功能缺失。

| Orderbook 数据流 | ❌ 无 store 级 API | BtApiFeed 的 tick 分发绕过了 store |
| 连接健康监控 / 心跳 | ❌ 无 | 后续风险项 |
| API 调用重试机制 | ❌ 无 | 后续风险项 |
| Gateway wrapper `poll_bar()` | ❌ 始终返回 None | Gateway 模式下 bar 流式推送不可用 |

**核心结论**：BtApiStore 是整个实盘链路的枢纽，功能覆盖面广。最大的**架构技术债**是 CTP 特定逻辑（~580 行 CtpClientWrapper 内部类）深度嵌入主模块，后续扩展新 provider 时维护成本会升高。但本轮不建议拆分——优先保持稳定，仅在接口层补齐缺口。

#### B.3 BtApiFeed（320 行）

| 能力项 | 当前状态 | 评估 |
|--------|----------|------|
| 历史 bar 回填 | ✅ 已实现 | 稳定 |
| Live bar 轮询 | ✅ 已实现 | 稳定 |
| Tick→Bar 聚合 | ✅ 支持 Seconds/Minutes/Days | 缺少 session 边界处理 |

### C. 回测/实盘策略接口一致性分析

#### C.1 策略已有的统一接口

| 策略接口 | 回测侧支持 | 实盘侧支持 | 一致性 |
|----------|-----------|-----------|--------|
| `next()` | ✅ bbroker bar 驱动 | ✅ BtApiFeed bar 驱动 | ✅ |
| `notify_tick(tick)` | ✅ TickBroker/MixBroker channel 驱动 | ✅ BtApiFeed `_dispatch_event()` | ✅ |
| `notify_orderbook(ob)` | ✅ TickBroker/MixBroker channel 驱动 | ❌ BtApiFeed 无 orderbook 分发 | **❌ 缺口** |
| `notify_bar(bar)` | ✅ MixBroker channel 驱动 | ✅ BtApiFeed `_enqueue_bar_event()` | ✅ |
| `notify_order(order)` | ✅ broker → strategy | ✅ BtApiBroker → strategy | ✅ |
| `self.buy()` / `self.sell()` | ✅ broker.buy/sell | ✅ BtApiBroker.buy/sell | ✅ |
| `self.cancel()` | ✅ broker.cancel | ✅ BtApiBroker.cancel | ✅ |
| `self.broker.getcash()` | ✅ 本地计算 | ✅ 远程查询 | ✅（语义一致，来源不同） |
| `self.broker.getposition()` | ✅ 本地计算 | ✅ 远程同步 | ✅（语义一致，来源不同） |

#### C.2 结论

**低频策略**（`next()` + bar data + `buy/sell`）：回测/实盘切换**已基本可行**，只需不同的 Cerebro 装配。

**高频策略**（`notify_tick()` + `notify_orderbook()` + `buy/sell`）：回测侧通过 TickBroker 的 channel 完整支持，实盘侧 **tick 可用但 orderbook 缺失**。这是迭代8必须解决的关键缺口。

**中频策略**（`notify_tick()` + `next()` + MidFreqContext + `buy/sell`）：回测侧通过 MixBroker 支持。实盘侧需要 BtApiFeed 同时提供 tick 事件和聚合 bar，且 BtApiBroker 需要能接收 MidFreqContext 或等价的上下文注入。这是最复杂的场景，建议本轮**仅做可行性验证**，不做完整实现。

### D. Cerebro channel 事件循环与 BtApiBroker 的关系

Cerebro 的 channel 事件循环（`cerebro.py:1006-1033`）会调用：
```python
if ch == "tick" and hasattr(self._broker, "process_tick"):
    self._broker.process_tick(evdata)
elif ch == "orderbook" and hasattr(self._broker, "process_orderbook"):
    self._broker.process_orderbook(evdata)
elif ch == "bar" and hasattr(self._broker, "process_bar"):
    self._broker.process_bar(evdata)
```

BtApiBroker **不实现**这三个方法，因此 channel 事件对 broker 层是透传的——broker 不做本地撮合，策略仍能通过 `notify_tick()` / `notify_orderbook()` / `notify_bar()` 收到事件。

**这是正确的设计选择**：实盘 broker 不应本地撮合，订单执行由交易所完成。但需要注意：
1. 实盘模式下策略通过 `notify_tick()` 获取的 tick 数据和通过 `self.data.close[0]` 获取的 bar 数据来源不同（前者是 feed 推送的事件，后者是 feed `_load()` 产生的 line 数据）
2. 需在文档中明确：实盘高频策略应依赖 `notify_tick()` 回调而非 line 数据来做决策

### F. 对初始需求"需要解决的问题"的技术回答

#### F.1 如何让同一份策略不改代码切换回测/实盘？

**答案**：通过不同的 Cerebro 装配脚本（或配置文件）控制 broker/feed/store 实例化。

```python
# 回测模式
cerebro.setbroker(bt.brokers.BackBroker(cash=100000))
cerebro.adddata(bt.feeds.GenericCSVData(dataname="data.csv"))

# 实盘模式
store = BtApiStore(provider="ctp", **ctp_config)
cerebro.setbroker(store.getbroker())
cerebro.adddata(store.getdata(dataname="rb2510"))
```

策略类本身只使用 `next()` / `notify_tick()` / `notify_orderbook()` / `buy()` / `sell()` 等统一接口，不感知运行模式。

**本轮建议**：定义一个轻量的 `LiveProfile` 配置结构（可以是 dict 或 dataclass），封装 provider/symbol/timeframe/broker_params 等信息，配合一个 `build_cerebro(profile)` 工厂函数，实现一行切换。不建议引入重量级配置框架。

#### F.2 三组件职责边界如何收敛？

| 组件 | 职责 | 不应承担 |
|------|------|---------|
| **BtApiStore** | 连接管理、API 生命周期、原始数据获取（bar/tick/orderbook）、订单提交/撤单路由、账户/持仓查询代理、runtime 事件审计 | 不做订单校验、不维护本地持仓状态、不聚合数据 |
| **BtApiBroker** | 订单校验与路由、本地订单状态管理、远程状态同步、持仓缓存、交易控制、策略通知 | 不做数据获取、不做本地撮合、不直接调用 API |
| **BtApiFeed** | 数据格式化、历史回填、live 数据轮询/推送、tick→bar 聚合、事件分发到 Cerebro channel | 不做订单操作、不维护账户状态 |

当前实现基本符合上述分工，无重大职责交叉。需要补齐的是 BtApiFeed 的 **orderbook 事件通道**。

#### F.3 三类策略 live 模式依赖什么？

| 策略类型 | Feed 依赖 | Broker 依赖 | Store 依赖 |
|---------|----------|------------|-----------|
| 低频 | bar 回填 + live bar 推送 | 订单路由 + 账户同步 | 连接 + bar 获取 + 订单/账户 API |
| 中频 | bar + tick 混合推送 | 订单路由 + 账户同步 + MidFreqContext 注入 | 连接 + bar+tick 获取 + 订单/账户 API |
| 高频 | tick + orderbook 推送 | 订单路由 + 账户同步 | 连接 + tick+orderbook 获取 + 订单/账户 API |

#### F.4 如何构造 <15s 的高频测试？

建议方案：
1. 从现有 `tests/datas/tick_data/tick_BTC_USDT.csv`（已存在）截取前 500-1000 条 tick
2. 使用 `FakeBtApiClient` 注入预加载 tick 队列，模拟 live tick 推送
3. 策略在收到 N 条 tick 后自动停止（`cerebro.runstop()`）
4. 验证策略收到了正确的 `notify_tick()` 回调、bar 聚合正确、订单路由正确

这种方式无需网络连接，数据量可控，执行时间可稳定在 1-5 秒。

#### F.5 如何不破坏既有测试？

1. **不修改**任何现有 `test_btapibroker.py`、`test_btapistore.py`、`test_btapifeed.py` 的断言
2. **新增**测试文件：`test_btapifeed_tick_aggregation.py`、`test_live_profile.py`、`test_hft_quick_baseline.py` 等
3. 对 `fake_btapi.py` 可以**扩展**（新增方法/属性），但不能**修改**现有接口签名
4. 所有新增代码通过 `pytest --co` 确认不影响现有测试发现

### H. 关键技术决策建议

1. **BtApiBroker 不需要 `process_tick()` 等方法**——实盘不做本地撮合是正确设计，保持当前架构
2. **BtApiFeed 是实盘数据链路的唯一入口**——不需要创建新的 feed 类，增强现有实现即可
3. **BtApiStore 的 CTP 内嵌代码本轮不拆分**——保持稳定，仅在接口层补齐 orderbook polling
4. **配置切换通过轻量 LiveProfile + 工厂函数实现**——不引入 YAML 框架或 DI 容器
5. **高频测试使用 FakeBtApiClient 注入数据**——不依赖网络，确保 CI 稳定和 <15s 执行
6. **cancel() 语义调整为"请求撤单"而非"立即撤单"**——本地状态等 `_drain_store_updates()` 收到远程确认后再更新

### I. 风险补充

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| BtApiFeed orderbook 通道需要 store 侧也支持 orderbook polling | 如果底层 API 不支持，则高频 live 无法完整工作 | FakeBtApiClient 先实现 mock，真实 provider 后续迭代补齐 |
| cancel() 语义变更可能影响现有 live 测试 | `test_simnow_ctp.py` 中的 order placement 测试依赖当前 cancel 行为 | 增加 `immediate_cancel` 参数保持兼容 |

| 中频 live 支持复杂度远超预期 | 可能占用过多工时影响 P0 交付 | 严格控制为 POC 级别，不做完整实现 |
