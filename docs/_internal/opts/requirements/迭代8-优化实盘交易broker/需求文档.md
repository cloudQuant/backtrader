## 1. 背景与动机

### 1.1 项目现状

backtrader 当前已经逐步形成“多 broker + 多运行模式”的基础能力：

| 模块 | 文件 | 主要用途 | 当前问题 |
|------|------|------|------|
| BackBroker | `backtrader/brokers/bbroker.py` | 低频 bar / kline 回测 | 更偏低频，不直接解决 live 接入问题 |
| TickBroker | `backtrader/brokers/tickbroker.py` | 高频 tick / orderbook 回测 | 适合高频回测，但和 live 运行契约未系统统一 |
| MixBroker | `backtrader/brokers/mixbroker.py` | 中频混合模式回测 | 中频回测能力刚建立，尚未与 live 模式统一 |
| BtApiBroker | `backtrader/brokers/btapibroker.py` | 实盘订单路由与账户状态回写 | 可用但职责边界、策略兼容口径和配置切换方式仍不清晰 |
| BtApiStore | `backtrader/stores/btapistore.py` | 底层 provider 对接、订单/成交/账户归一化 | provider 归一化能力较重，但对上层运行契约表达不足 |
| BtApiFeed | `backtrader/feeds/btapifeed.py` | live 行情输入映射 | 当前按业务口径视为“已存在的 live 数据桥接层”，本轮按增强方式处理 |

当前业务提出的核心诉求，不是“给 live broker 再补几个接口”，而是希望把以下能力统一起来：

1. 同一份低频策略，在不改策略代码的情况下，能切换回测和实盘。
2. 同一份中频策略，在不改策略代码的情况下，能切换回测和实盘。
3. 同一份高频策略，在不改策略代码的情况下，能切换回测和实盘。
4. 回测/实盘的差异通过配置和装配完成，而不是通过策略类内部写条件分支完成。
5. 优先从高频策略小样例和快速测试切入，建立 live 侧的可验证主链路。

### 1.2 本轮业务定义

本轮定义的“优化实盘交易 broker”，不是单独优化 `BtApiBroker` 这个类，而是建立以下统一能力：

- 策略接口层统一：策略在回测和实盘中看到的调用方式尽量一致。
- 运行装配层统一：通过配置文件决定当前运行在 backtest 还是 live。
- 数据输入层统一：低频、中频、高频策略在 live 模式下分别拿到自己需要的数据事件。
- 订单状态层统一：订单提交、拒绝、取消、部分成交、完成等状态对策略的可见语义稳定。
- 账户回写层统一：现金、持仓、账户价值、成交通知和持仓同步形成一致口径。

### 1.3 典型业务期望

业务对本轮至少有以下明确期望：

1. 策略源码不改。
   - 不希望在策略类里出现 `if live:`、`if backtest:` 的分支。
2. 配置文件可改。
   - 可以通过配置切换 broker、feed、store、provider 和运行模式。
3. 先从高频切入。
   - 参考 `examples` 中已有高频策略，裁剪成小数据集、小场景、快速测试。
4. 兼容存量测试。
   - 现有稳定测试用例不能被破坏；`tickbroker`、`mixbroker` 新增测试允许按新契约调整。

### 1.4 当前核心问题

结合现有实现和业务目标，本轮需要解决以下核心问题：

1. 回测/实盘运行契约不清。
   - 当前没有明确文档定义：策略到底依赖哪些最小能力，才可以“无代码修改”切换模式。
2. live 侧 broker / store / feed 边界不清。
   - `BtApiBroker`、`BtApiStore`、可选的 `BtApiFeed` 之间的职责界面还没有系统收敛。
3. 三类策略的 live 支撑能力不清。
   - 低频、中频、高频策略各自需要什么事件、什么账户能力、什么通知节奏，还缺少统一定义。
4. 快速验证资产不足。
   - 如果没有一套小而快的高频测试，live 改造很难持续回归。
5. 兼容性风险较高。
   - 本轮必须在不破坏既有稳定测试的前提下增加 live 能力。

## 2. 重分析结论

### 2.1 本轮不是单点补丁优化

本轮不应理解为：

- 给 `BtApiBroker` 增加几个方法。
- 给 `BtApiStore` 增加几个 provider 分支。
- 临时新建一个 feed 文件就算完成 live 支持。

更合理的目标是：

1. 明确统一运行契约。
2. 明确三类策略在 live 模式下的能力矩阵。
3. 明确 broker / store / feed 的职责边界。
4. 明确配置切换方式和最小可运行 profile。
5. 建立可快速执行的验证资产。

### 2.2 架构方向结论

1. 回测与实盘应在“策略接口层”尽可能统一。
   - 策略如何下单、撤单、收通知、查持仓，应尽量保持一致。
2. 回测与实盘不要求在“内部执行实现”完全统一。
   - 回测 broker 继续负责模拟撮合。
   - live broker 负责真实订单路由和账户同步。
3. 配置是本轮的关键边界层。
   - broker、feed、store、provider、运行模式必须可通过配置表达。
4. `BtApiFeed` 在本轮应视为已存在的 live 数据桥接层。
   - 本轮按增强方式处理，优先补齐 `orderbook` 事件通道。
5. 高频小样例和快速测试是本轮的第一验证入口。
   - 不先建立测试基线，就很难稳步推进 live 改造。

### 2.3 本轮目标修正

本轮优先目标不是：

- 覆盖所有 live provider 的全部能力。
- 实现逐笔级别的回测/实盘完全一致。
- 一次性补齐风控、容灾、监控平台。
- 重写低频、中频、高频三类回测 broker。

本轮优先目标是：

- 明确统一运行契约。
- 明确 live 装配和配置切换方式。
- 打通高频 live 的最小可用验证链路。
- 为低频/中频/高频三类 live 运行建立统一框架。

## 3. 需求定义

### 3.1 P0 — 必须实现

#### R-P0-01：统一运行契约

**描述**：定义同一份策略在回测/实盘之间切换时所依赖的最小统一能力。

**功能要求**：

1. 策略不需要修改源码即可切换回测/实盘。
2. 策略侧的下单、撤单、通知、查询持仓/现金/账户价值的语义保持稳定。
3. broker/feed/store 的装配差异由配置承担，而不是由策略承担。
4. 统一运行契约需要对低频、中频、高频三类策略都可解释。

**验收标准**：

- 可以清楚说明“什么样的策略算满足回测/实盘无代码切换前提”。
- 三类策略都有对应的运行契约说明。
- 文档可直接指导后续设计与实现。

#### R-P0-02：配置驱动的 backtest/live 切换能力

**描述**：使用配置文件表达运行模式差异，而不是改策略代码。

**功能要求**：

1. 配置文件中能区分 backtest 与 live 模式。
2. 配置文件中能指定 broker、feed、store、provider 以及必要参数。
3. 配置文件中能表达低频、中频、高频三类运行 profile。
4. 配置层不能把复杂环境差异泄漏到策略源码中。

**验收标准**：

- 同一策略通过不同配置可以指向回测和实盘。
- profile 结构清晰、可文档化、可扩展。
- 不需要为了切换模式去修改策略实现。

#### R-P0-03：live 侧 broker / store 职责收敛

**描述**：明确 `BtApiBroker` 与 `BtApiStore` 在 live 链路中的职责边界。

**功能要求**：

1. `BtApiBroker` 主要负责订单生命周期管理、策略通知、账户/持仓对外语义。
2. `BtApiStore` 主要负责底层 provider 接入、数据归一化、订单/成交/账户事件桥接。
3. broker 与 store 的职责不能继续混杂。
4. 需要形成对外行为契约说明。

**验收标准**：

- 能清楚解释 broker 与 store 的边界。
- 订单提交流程、取消流程、状态通知链路清晰。
- 账户与持仓同步责任划分明确。

#### R-P0-04：三类策略的 live 能力矩阵

**描述**：明确低频、中频、高频三类策略在实盘模式下分别需要哪些输入与能力。

**功能要求**：

1. 低频策略的 live 输入与更新语义明确。
2. 中频策略需要的高频状态 + 低频状态组合能力明确。
3. 高频策略需要的 tick / orderbook 输入与通知能力明确。
4. 三类策略所依赖的账户查询能力和订单通知语义明确。

**验收标准**：

- 三类策略都有清晰的 live 支持边界。
- 不同 profile 的差异可以通过能力矩阵表达。
- 能指导后续 feed/broker 设计和测试设计。

#### R-P0-05：高频快速测试基线

**描述**：优先建立高频策略的小样例、小数据集和快速测试用例。

**功能要求**：

1. 参考 `examples` 中现有高频策略形成最小样例。
2. 使用较小数据集构造快速测试。
3. 单个测试执行时间控制在 15 秒以内。
4. 测试能覆盖 live 改造的关键链路或关键契约。

**验收标准**：

- 至少形成一组高频快速测试场景。
- 单个用例满足 15 秒时间约束。
- 测试输出可用于持续回归。

### 3.2 P1 — 应当实现

#### R-P1-01：BtApiFeed 增强方案与落位

**描述**：明确 `BtApiFeed` 在现有 live 数据链路中的增强方向和职责边界。

**功能要求**：

1. 明确 `BtApiFeed` 已存在，本轮优先增强现有实现。
2. 需要明确：是直接增强现有实现，还是在特殊场景下补充额外桥接层。
3. `BtApiFeed` 需要定义其对低频、中频、高频策略的适配边界。
4. `BtApiFeed` 与 `BtApiStore` 之间的数据职责边界要清楚。

**验收标准**：

- 对 `BtApiFeed` 的结论明确。
- 能指导后续实现与目录结构落位。
- 不与 store 职责重叠混乱。

#### R-P1-02：live 数据事件映射规则

**描述**：定义 live 侧 tick / orderbook / bar / 订单通知等事件如何映射到策略可见层。

**功能要求**：

1. 时间戳语义明确。
2. symbol 映射规则明确。
3. 同类事件的推送节奏和可见性语义明确。
4. 策略通知顺序与 broker 状态更新顺序应可解释。

**验收标准**：

- 可以清晰描述 live 数据从底层 provider 到策略回调的传递路径。
- 不同类型事件的可见性口径明确。
- 能支撑测试设计。

#### R-P1-03：兼容性冻结规则

**描述**：在增加 live 能力的同时，明确哪些测试和行为必须保持稳定。

**功能要求**：

1. 现有稳定测试不能被破坏。
2. `tickbroker`、`mixbroker` 新增测试允许按新契约调整。
3. 兼容性边界需文档化，而不是隐式理解。
4. 需要形成回归重点清单。

**验收标准**：

- 能清楚说明哪些测试不能改，哪些新增测试允许调整。
- live 改造的回归边界明确。
- 测试风险可管理。

### 3.3 P2 — 可选实现

#### R-P2-01：更多 provider 扩展策略

- 对更多 provider 做统一归一化策略设计。
- 形成通用 provider 接口约束。

#### R-P2-02：更强的 live 运维支撑

- 断连重连、运行观测、风控限流、异常告警等可以作为后续增强项。

#### R-P2-03：回测/实盘一致性度量

- 后续可引入“回测策略行为 vs live 策略行为”的一致性观测指标，而非逐笔严格相等。

## 5. 非功能需求

### 5.1 可配置性

- 模式切换应通过配置完成。
- 低频、中频、高频三类 profile 应可扩展。
- provider 和 broker/feed/store 装配方式应可表达。

### 5.2 可解释性

- 策略在 live 模式下为什么收到某种事件、某种订单状态，必须可解释。
- broker / store / feed 的职责边界必须可解释。

### 5.3 可测试性

- 本轮必须优先建立快而稳定的高频测试样例。
- 核心行为要可以通过 mock、fake 或小数据集进行验证。

### 5.4 可扩展性

- 后续可以增加更多 provider。
- 后续可以增加更复杂的 live feed 形态。
- 后续可以扩展更强的运维、风控、监控能力。

### 5.5 性能目标

- 高频快速测试场景下，单个测试执行时间控制在 15 秒以内。
- 本轮关注的是“可持续回归成本可控”，而不是极限吞吐。

## 6. 风险评估

| 风险 | 等级 | 影响 | 缓解措施 |
|------|------|------|----------|
| 统一运行契约定义不清 | 高 | 策略层会出现大量隐式兼容逻辑 | 先冻结最小统一契约，再进入详细设计 |
| broker/store/feed 职责继续重叠 | 高 | live 链路后续维护成本高 | 在设计阶段先收敛职责边界 |
| 高频快速测试数据集过大 | 高 | 回归成本失控，难以持续执行 | 用小数据集和 mock/fake 夹具优先建立基线 |
| 低频/中频/高频 live 支持边界不清 | 中 | 文档、实现、测试口径不一致 | 建立三类策略能力矩阵 |
| 试图一次覆盖过多 provider 能力 | 中 | 拖慢本轮主链路交付 | 把 provider 全覆盖放到后续迭代 |

## 8. 术语表

| 术语 | 定义 |
|------|------|
| 统一运行契约 | 策略在回测/实盘之间切换时依赖的稳定接口与行为口径 |
| 运行 profile | 用配置描述的一组运行模式装配方案，如 lowfreq-live、midfreq-backtest、hft-live |
| live broker | 面向真实交易接口的 broker 实现，负责订单路由与账户状态对外语义 |
| live store | 面向底层 provider 的接入与归一化层 |
| live feed | 面向策略输入的数据桥接层；本轮对应已存在的 `BtApiFeed`，按增强方式处理 |
| 配置驱动切换 | 通过配置而非修改策略源码来切换 backtest/live |
| 高频快速测试 | 使用小数据集、小场景、低执行时间建立的高频回归测试 |
| 能力矩阵 | 用于说明低频、中频、高频三类策略在 live 模式下所需输入和能力的矩阵 |

### 9.2 需求细化与修正

#### R-P0-01 修正：统一运行契约

原始描述过于抽象。基于代码审查，建议将"统一运行契约"具体化为以下**最小统一接口集**：

```python
# 策略可依赖的最小统一接口 —— 回测/实盘均需实现
class UnifiedStrategyContract:
    # 数据访问
    self.data.close[0]              # bar 数据
    self.data.volume[0]             # bar 数据
    
    # 事件回调（按策略类型选用）
    def next(self): ...             # 低频/中频
    def notify_tick(self, tick): ...       # 高频/中频
    def notify_orderbook(self, ob): ...    # 高频/中频
    def notify_bar(self, bar): ...         # 中频
    def notify_order(self, order): ...     # 所有
    def notify_trade(self, trade): ...     # 所有
    
    # 订单操作
    self.buy(size=, price=, exectype=)
    self.sell(size=, price=, exectype=)
    self.cancel(order)
    
    # 账户查询
    self.broker.getcash()
    self.broker.getvalue()
    self.broker.getposition(data)
```
