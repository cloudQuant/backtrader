## 1. 测试策略

### 1.1 总体原则

| 原则 | 说明 |
|------|------|
| 契约优先 | 本轮测试首先验证回测/实盘统一运行契约，而不是验证单一 provider 的全部细节 |
| 高频优先 | 先建立高频 live 的快速测试基线，再扩展中频和低频 |
| 配置优先 | 重点验证 profile 配置与装配是否成立，而不是要求策略源码适配 |
| 分层验证 | 采用规则测试、场景测试、集成测试、样例测试四层结构 |
| 兼容优先 | 既有稳定测试不能被破坏，新增 live 测试应最小侵入 |
| 快速回归 | 高频新增测试单个用例执行时间必须控制在 15 秒以内 |

### 1.2 测试分层

```text
Layer 4: 端到端 / 样例测试
  - 高频 live/backtest 统一样例
  - 中频 live profile 可运行样例
  - 低频 live profile 可运行样例

Layer 3: 集成测试
  - BtApiBroker + BtApiStore 链路
  - 可选 BtApiFeed 数据桥接
  - 配置驱动的 runtime profile 装配

Layer 2: 场景测试
  - 订单生命周期
  - 账户/持仓同步
  - live 数据事件映射
  - 三类策略能力矩阵验证

Layer 1: 规则测试
  - 配置切换规则
  - 兼容性冻结规则
  - 事件字段归一化规则
  - 高频快速测试时间约束
```

### 1.3 测试口径

1. 本轮测试主线是“**统一运行契约 + live 主链路可验证**”。
2. 测试不要求证明回测和实盘逐笔成交完全一致。
3. 测试重点不在收益结果，而在：
   - 同一策略是否可通过配置切换 backtest/live
   - broker / store / feed 的职责是否清晰
   - 订单生命周期是否可解释
   - 账户和持仓同步是否稳定
   - 高频快速测试是否满足 15 秒约束
4. `BtApiFeed` 本轮按“**已存在并增强现有实现**”处理。
   - 测试重点是验证其既有 tick/bar 能力不被破坏
   - 并补齐 `orderbook` 事件通道覆盖

## 3. 推荐测试场景

### 3.1 统一契约场景

#### 场景 S-01：同一高频策略通过 profile 切换 backtest/live

**输入条件**：

- 使用同一个策略类。
- 提供两套配置：`hft-backtest` 和 `hft-live`。

**预期结果**：

- 不修改策略源码。
- 两种模式都能完成启动装配。
- 策略可以正常下单和接收通知。

#### 场景 S-02：同一低频策略通过 profile 切换 backtest/live

**预期结果**：

- 低频策略保持 bar 驱动语义。
- 切换模式不要求在策略类中增加条件分支。

### 3.2 broker / store / feed 边界场景

#### 场景 S-03：broker 只负责对策略暴露订单语义

**预期结果**：

- `BtApiBroker` 承担订单提交流程与策略通知。
- provider 细节不直接泄漏到策略层。

#### 场景 S-04：store 只负责底层归一化

**预期结果**：

- `BtApiStore` 负责归一化账户、持仓、订单、成交或行情事件。
- 不直接承担策略级订单语义定义。

#### 场景 S-05：BtApiFeed 增强边界明确

**预期结果**：

- `BtApiFeed` 作为现有 live 数据桥接层继续使用。
- 本轮增强仅补齐 `orderbook` 通道，不改变其 broker/store 职责边界。

### 3.3 订单生命周期场景

#### 场景 S-06：订单提交后被接受

**预期结果**：

- 策略收到可解释的接受通知。
- broker 内部订单状态与 store 返回结果一致。

#### 场景 S-07：订单被拒绝

**预期结果**：

- 策略能读到拒绝状态和错误原因。
- 不出现伪造持仓变化或错误账户回写。

#### 场景 S-08：订单取消

**预期结果**：

- 撤单请求后状态演进清晰。
- 已取消订单不会继续被当作活动订单处理。

#### 场景 S-09：部分成交与完成

**预期结果**：

- 成交通知顺序可解释。
- 账户/持仓更新与成交结果一致。

### 3.4 高频快速测试场景

#### 场景 S-10：高频最小样例快速运行

**预期结果**：

- 单个用例在 15 秒以内完成。
- 能验证最小 live 主链路。

#### 场景 S-11：小数据集重复回放稳定

**预期结果**：

- 相同测试数据重复运行行为稳定。
- 结果差异不来自不受控外部依赖。

#### 场景 S-12：mock / fake provider 注入

**预期结果**：

- 可以在不依赖真实 provider 的情况下验证 broker/store/feed 主链路。
- 错误、断连、延迟等基础场景可被模拟。

## 8. 验收结论口径

本轮测试最终要回答的不是“策略能不能赚钱”，而是以下问题：

1. 同一策略是否能通过配置切换 backtest/live，而不改源码。
2. `BtApiBroker`、`BtApiStore`、`BtApiFeed` 的职责边界是否清晰。
3. 低频、中频、高频三类策略在 live 模式下是否具备清晰能力边界。
4. 高频 live 的最小验证链路是否已经建立，并且单测成本可控。
5. 现有稳定测试是否保持兼容，新增 live 测试是否形成持续回归基础。
