## 2. 本轮问题定义

经过现状分析与 2026-03-17 代码审计，本轮要解决的不是单点 bug，而是五类系统性问题：

1. **仓库卫生问题**（已治理完毕，仅需确认冻结）
   - `.gitignore` 与 CI 拦截规则已就绪，仅需在本轮文档中冻结为基线。

2. **策略实例化脆弱问题**
   - `Strategy` 当前通过手工干预 `__new__` / `_create_strategy_safely` 完成参数与实例拼接。
   - `__init__` 中硬编码 `self.__class__.__name__ == "TestStrategy"` 特殊分支（`strategy.py:281`），导致任何用户类取名 `TestStrategy` 都会得到非预期处理。
   - `_create_strategy_safely()` 中 `filtered_kwargs = {}` 带有 `# TestStrategy.__init__ takes no kwargs` 注释，属于测试假设泄露。
   - `__new__` 完成完整参数设置后，`__init__` 又对 kwargs 做二次过滤处理，双重处理路径易出错。
   - 子类初始化行为、参数透传、错误定位的心智成本偏高。

3. **异常被吞与调试不可见问题**
   - `Cerebro` channel 模式 `run()` 中策略初始化异常后静默 `except Exception: continue`（`cerebro.py:990`）。
   - `Strategy.__init__` 中用户子类 init 失败后静默 fallback 到 `filtered_kwargs` 重试（`strategy.py:337`），真实错误被掩盖。
   - `_runnext()` 将整个主循环包裹在 `try/except Exception` 中，异常写入 `~/cerebro_runnext_error.log` 硬编码路径后继续执行（`cerebro.py:2147-2152`），属于调试产物。
   - 这类行为会导致"运行结果不正确但又不明显失败"。

4. **核心主循环与死代码问题**
   - `Cerebro` 主循环中有较多历史遗留判断、注释掉的分支与待清理变量。
   - 具体包括：`my_num` 调试计数器（第 1983 行）、`_rs`/`_rp` 计算后未使用（第 1963-1965 行）、`lastqcheck` 已注释（第 1978 行）、`dt0 < 1` 检查被标记为错误但未修复（第 2043 行）、多处注释掉的 debug print。
   - `btapistore.py:231-234` 存在 `return` 之后的不可达死代码片段。

5. **Store 抽象层级竞争问题**
   - 当前存在 **两套 Store 继承体系**：
     - 旧版：`Store`（`backtrader/store.py`）基于 `SingletonMixin` + `StoreParams`，`getbroker()` 为 `@classmethod`，`broker._store = cls`（绑定类而非实例）。
     - 新版：`LiveStoreBase`（`backtrader/stores/livestore.py`）基于 ABC，`getbroker()` 为实例方法。
   - `BtApiStore` 继承自 `LiveStoreBase`，**不继承 `Store`**。原文档将两者混为一个层级讨论。
   - `LiveStoreBase.start()` 签名为 `start(self)` 无参数，但 `BtApiStore.start(self, data=None, broker=None)` 接受参数，违反 ABC 契约。
   - `BtApiStore` 自定义了 `emit_runtime_event()` 方法，`LiveStoreBase` 未定义；旧版 `Store` 使用 `put_notification()` / `get_notifications()`，两套通知机制并存。

### 6.2 当前核心风险点（2026-03-17 代码审计更新）

#### 风险点 A：`Strategy` 实例化链路脆弱

主要涉及：

1. `backtrader/strategy.py`
2. `backtrader/cerebro.py`

已观察到的问题（附精确行号）：

1. `_create_strategy_safely()`（`strategy.py:120-133`）调用 `cls.__new__()` 后手工调用 `Strategy.__init__()`，且 `filtered_kwargs = {}` 写死为空字典，注释 `# TestStrategy.__init__ takes no kwargs` 暴露了测试假设泄露。

   - **channel 模式**（`cerebro.py:984-991`）：`except Exception: continue`，静默跳过失败策略。
   - **标准模式**（`cerebro.py:1613-1623`）：仅捕获 `StrategySkipError`，行为正确。

#### 风险点 B：`Cerebro` 异常可见性缺陷

主要位置与精确行号：

1. **channel 模式 `run()` 静默跳过**（`cerebro.py:990-991`）：`except Exception: continue`，策略构造失败后调用者无感知。

#### 风险点 D：Store 两套继承体系竞争

> **重要纠正**：原文档将 `Store` 与 `BtApiStore` 描述为同一继承链上的基类与子类关系，这与代码事实不符。

当前存在两套独立的 Store 继承体系：

| 维度 | 旧版 `Store` | 新版 `LiveStoreBase` |
| --- | --- | --- |
| 文件 | `backtrader/store.py` | `backtrader/stores/livestore.py` |
| 基类 | `SingletonMixin` + `StoreParams` | `ABC` |
| `getbroker()` | `@classmethod`，`broker._store = cls` | 实例方法（`@abstractmethod`） |
| `start()` | `start(self, data=None, broker=None)` | `start(self, data=None, broker=None)`（已修复） |
| 通知机制 | `put_notification()` / `get_notifications()` | 无（`BtApiStore` 自定义 `emit_runtime_event()`） |
| 单例 | 是 | 否 |
| 实际使用者 | 旧版 stores（如遗留的 IB/Oanda/VC stores） | `BtApiStore` |

已观察到的问题：

1. `BtApiStore.start(self, data=None, broker=None)` 签名与 `LiveStoreBase.start(self)` ABC 契约不匹配。由于 Python 不严格校验签名，运行时不报错，但违反接口约定。
2. `BtApiStore` 没有继承旧版 `Store` 的 `put_notification()` / `get_notifications()`，而是自己定义了 `notifs` deque 和 `emit_runtime_event()`，两套通知机制并存。
3. 旧版 `Store.getbroker()` 将 `broker._store = cls`（绑定类），`BtApiStore.getbroker()` 将 `self._broker = broker`（绑定实例），语义完全不同。

### Phase 4：`Store` 契约澄清与最小统一 ✅ 已完成

- 优先级：P1
- 状态：**已完成**
- 目标：减少基类语义与 live store 现实用法的冲突

#### 问题摘要（2026-03-17 审计更新）

> **重要纠正**：原文档将 `Store` 与 `BtApiStore` 描述为同一继承链，这与代码事实不符。

当前存在 **两套独立的 Store 继承体系**：

- **旧版 `Store`**（`backtrader/store.py`）：基于 `SingletonMixin` + `StoreParams`，`getbroker()` 为 `@classmethod`（`broker._store = cls` 绑定类），`start(self, data=None, broker=None)`。用于遗留 stores。
- **新版 `LiveStoreBase`**（`backtrader/stores/livestore.py`）：基于 ABC，`getbroker()` 为实例方法（`@abstractmethod`），`start(self, data=None, broker=None)`（已修复）。`BtApiStore` 继承此类。

审计发现的具体问题：

1. `BtApiStore.start(self, data=None, broker=None)` 签名与 `LiveStoreBase.start(self)` ABC 不匹配。
2. `BtApiStore` 自定义 `emit_runtime_event()`，`LiveStoreBase` 未定义；旧版 `Store` 使用 `put_notification()`/`get_notifications()`，两套通知机制并存。
3. 旧版 `Store.getbroker()` 绑定类（`broker._store = cls`），`BtApiStore.getbroker()` 绑定实例（`self._broker = broker`），语义完全不同。

#### 建议方向

1. 先做"最小统一"，而不是全面推翻。**明确旧版 `Store` 为遗留接口，新版 `LiveStoreBase` 为活跃接口。**
2. 修复 `LiveStoreBase` ABC 契约：
   - 将 `start()` 签名调整为 `start(self, data=None, broker=None)` 以匹配 `BtApiStore` 的实际用法，或在 `BtApiStore` 中使用 `**kwargs` 兼容。
   - 考虑在 `LiveStoreBase` 中增加可选的 `emit_runtime_event()` 默认实现或 `put_notification()` 抽象。
3. 在 `store.py` 中增加文档注释，说明旧版 `Store` 与新版 `LiveStoreBase` 的关系和迁移方向。
4. 用测试锁定 `BtApiStore` 的现实行为。

#### 任务拆分

1. 修复 `LiveStoreBase.start()` 签名以匹配 `BtApiStore` 的实际用法。
2. 在 `LiveStoreBase` 中增加可选的通知机制接口（`emit_runtime_event` 或 `put_notification`）。
3. 在 `store.py` 中增加文档注释，说明两套继承体系的关系。
